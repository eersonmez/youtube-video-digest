No, You Really Don't Need MediatR in .NET

Nick Chapsas · 2026-05-20T12:30:34+00:00
.NET Architecture and the overuse of the MediatR library
Nick Chapsas argues that developers often default to using MediatR in their .NET projects without a real need, resulting in unnecessary complexity and poor debuggability. He explains that direct service invocation is usually a simpler and more explicit design. He also demonstrates how to implement cross-cutting concerns like validation and logging using native ASP.NET Core features (such as endpoint filters or decorators) instead of relying on MediatR pipelines.
  • MediatR is frequently overused in .NET web APIs to implement patterns like CQRS and Clean Architecture, leading to unnecessary layers of abstraction.
  • Using MediatR introduces 'magic' indirection via reflection-based routing, which makes code navigation, static analysis, and debugging significantly harder.
  • For simple to moderately sized applications, calling services or logic directly is more explicit, performant, and maintainable.
  • Cross-cutting concerns like validation, logging, and caching (historically solved using MediatR's pipeline behaviors) can be cleaner and more naturally implemented using ASP.NET Core endpoint filters, decorators, or middleware.
  • The critique is not a condemnation of the MediatR library itself, but rather a warning against cargo-cult programming and over-engineering by default.
