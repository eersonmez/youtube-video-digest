Scott Hanselman interviews Zed founder Nathan Sobo about DeltaDB and rethinking version control for collaborative, agent-assisted programming.

Scott Hanselman · 2026-06-21T23:17:37+00:00

The space between the Commits with Zed and DeltaDB's Nathan Sobo

Scott Hanselman talks with Nathan Sobo, founder of Zed, about editor monocultures, the history from Atom and Electron to Visual Studio Code, and why Zed was built by reinventing the stack for a faster, more intentional developer experience. The discussion then focuses on DeltaDB, Zed’s new version-control layer that builds on Git rather than replacing its core snapshot model. Sobo explains that Git captures commits, while DeltaDB records and indexes the work that happens between commits: edits, agent conversations, work trees, references, and collaboration context. The goal is to make work trees shareable, trace code back to the conversations that produced it, jump to exact historical code moments through “portals,” and follow code as it moves across files. They also discuss privacy concerns, possible summarization or selective sharing, and early access at zed.dev/deltadb.

  • Nathan Sobo created Atom at GitHub and later founded Zed to build a faster editor stack.
  • Zed was motivated by dissatisfaction with Electron-based editor performance and VS Code dominance.
  • DeltaDB is presented as a layer on top of Git that records fine-grained deltas between commits.
  • It can preserve agent conversations, edits, and work-tree state as collaborative artifacts.
  • DeltaDB aims to let developers trace code to its originating conversation and follow it through history.
  • Sobo emphasizes privacy controls and selective sharing to avoid workplace-surveillance dynamics.
