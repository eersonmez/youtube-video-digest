A case study of Gili Kamma stabilizing and scaling a legacy embedded smart-metering system.

NDC Conferences · 2026-07-20T13:34:47+00:00

Boosting Software Efficiency : A Case Study of 100% Performance Improvement in ... - Gili Kamma

In 2017, Gili Kamma inherited a legacy embedded smart-metering concentrator that collected wireless water-meter readings and forwarded them to READy Manager. Running on a 32 MB, older Linux/C++/Qt platform, it suffered unexplained resets, memory leaks, data loss, and poor scalability. Kamma instrumented allocations by overriding new/delete, built a small internal simulator that multiplied real messages, and reproduced failures at high load. A reproducible breakpoint exposed two threads allocating roughly 5 MB messages simultaneously; serializing the work and using fixed/static storage reduced memory pressure and doubled capacity from about 5,000 to 10,000 meters. He persisted outgoing data, retried until acknowledgment, sent recent logs after resets, reinitialized the UART after receiver-induced silence, and added proactive fleet monitoring. Continuous load tests and automation followed. After eight months the product had no resets, and release cycles improved from six months with three engineers to every two or three months, while enabling new features.

  • Built an internal simulator that multiplied real meter messages to reproduce failures at high load.
  • Measured allocation counts, sizes, and peaks to investigate memory leaks, fragmentation, and exhaustion.
  • Serialized large-message processing and used fixed or static storage, increasing capacity from about 5,000 to 10,000 meters.
  • Stored outgoing data in nonvolatile memory and retried transmission until acknowledgment to prevent significant data loss.
  • Added reset diagnostics, UART recovery, fleet summaries, proactive monitoring, and continuous load testing.

Algorithms & Combinators - Conor Hoekstra - NDC Toronto 2026

In this talk, Conor Hoekstra builds an algorithmic mental model around semantics rather than names, comparing array-language operations with Python and C++. He first examines the copy family: integer arguments produce repeat or replicate behavior, Boolean masks support compress, unary predicates implement filter or copy_if, and binary predicates enable adjacent filtering, including unique-like behavior. He then discusses the map family, including binary map and adjacent map, with names such as zip_transform and adjacent_transform. The final section covers cuts, especially chunk, chunk_by, and chunk_by_key, which split sequences into groups using sizes, adjacent boundaries, or key values. Combinators such as C, W, S, sigma, and side are presented as function-manipulating composition tools; C++ range projections are related to the side pattern. The central message is to recognize algorithm variants by their argument shapes and semantics.

  • Algorithm names vary, so argument shape and semantics are more useful than labels.
  • The copy family spans repeat, compress, replicate, filter or copy_if, and adjacent filtering.
  • The map family includes binary map and adjacent map, with names such as zip_transform and adjacent_transform.
  • The chunk family includes fixed-size chunks, chunk_by, and chunk_by_key.

Carbon: graduating from the experiment - Chandler Carruth - NDC Toronto 2026

Chandler Carruth presents the current status of Carbon, a language project intended to address C++'s technical debt, slow evolution, compile times, macros, weak generic and package systems, and lack of memory safety. He demonstrates Carbon syntax with a vending-machine example, including methods, references, access control, packages, and interoperability with C++ through imported headers and inline C++ blocks. Carbon delegates C++ API selection and overload resolution to Clang while applying Carbon's conversion rules. The talk explains proposed temporal and thread memory-safety mechanisms: APIs can declare invalidation effects and protected places so the compiler detects dangling pointers and mismatched mutex protection. Carruth describes incremental migration from permissive Carbon, mechanically ported from C++, toward strict checking without increasing undefined behavior. He says core feasibility questions are nearing positive answers, with more implementation, evaluation feedback, build-system integration, and limited production use still ahead.

  • Carbon targets C++ technical debt, slow evolution, compile times, and missing memory safety.
