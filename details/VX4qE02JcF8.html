Using Artificial Intelligence to Assist in Test-Driven Development for Code

NDC Conferences · 2026-06-24T13:43:07+00:00 · Test Driving Code with the Help of AI - Venkat Subramaniam - NDC Copenhagen 2026

What This Video Is About

This video explores the concept of "test driving code" by integrating Artificial Intelligence (AI) tools into the Test-Driven Development (TDD) workflow. The central argument is that automated testing serves two critical purposes in software development: providing regression benefits and acting as an act of design rather than mere verification [0:00]. While AI can generate test cases, the speaker strongly cautions that its output quality is entirely dependent on the skill and guidance provided by the human developer.

The presentation uses Conway's Game of Life as a practical case study to demonstrate how an experienced developer acts as an "active navigator" when using AI. The core message is that AI functions as an amplifier of existing expertise; it does not possess inherent understanding or design intuition. Therefore, developers must remain highly vigilant, directing the AI's efforts precisely rather than passively accepting its suggestions.

Detailed Walkthrough

The speaker begins by establishing the value proposition of automated testing, noting that while regression benefits are important, the design benefit—writing tests first to drive code structure—is paramount [0:00]. The speaker expresses excitement about using AI because it can handle repetitive tasks like generating variations of tests, freeing up human time for higher-level concerns.

The demonstration begins with Conway's Game of Life, specifically aiming to model an infinite universe rather than a finite grid [5:00]. The initial step involves creating a "canary test," which is described as the simplest possible test (e.g., asserting true equals true) used solely to confirm that the testing environment is set up correctly and functioning [7:34].

The speaker then moves into the core TDD loop, emphasizing an incremental design process where one first examines and refines tests before writing code. When prompting AI for a test (e.g., "test two"), the initial output might be useless or overly generic, such as checking if something is not null [10:00]. The speaker notes that AI often learns from common but poor practices in existing codebases, leading it to generate similarly weak tests.

To guide the AI effectively, the developer must provide highly specific instructions. For instance, when testing a dead cell's behavior, the prompt moves beyond simple assertions to define inputs (current state is dead, live neighbors is zero) and expected outputs (next state equals dead) [15:00]. The speaker highlights that AI tends to "jump ahead" by inferring complex structures like Arrange-Act-Assert or generating subsequent related tests prematurely.

The process continues with refinement. When the AI generates multiple independent assertions within a single test, the developer corrects this, adhering to the principle of never putting multiple independent asserts in one test so that failure visibility is complete [19:00]. The AI quickly adapts after these corrections, demonstrating "instantaneous learning" by generating related tests for alive cells based on the initial dead cell logic [23:48].

The discussion then addresses limitations. The speaker points out the AI's "stopping problem"—it may not know when enough test cases have been written and will continue to generate them unnecessarily—and its tendency toward an "edge case" obsession, such as testing for negative cell counts which are nonsensical in this context [24:56].

A significant portion of the demonstration focuses on evolving design patterns. The speaker moves away from state-based testing (using setters and getters) towards behavior testing. When prompting AI to generate signals for a single cell with specific neighbors (e.g., two and one), it can correctly infer the required structure, including visualizing positions [29:00]. However, the speaker warns that this apparent intelligence is often just pattern matching or copying from existing code, which they term "plagiarism" in an AI context [31:48].

Finally, when testing aggregation functions like `count signals`, the AI initially defaults to imperative programming styles because it has seen more examples of that style. By explicitly directing the AI with directives such as "write this in functional style," the speaker successfully guides it toward using appropriate functional operations like flat map/select many [45:00]. The ultimate philosophy presented is that AI is an "amplified intelligence"—it enhances the capabilities of a skilled user but cannot compensate for fundamental lack of knowledge.

Key Takeaways

  • Automated testing provides both regression benefits and acts as a crucial act of design in software development.
  • AI excels at automating repetitive tasks in test generation, minimizing manual effort like copy-pasting variations.
  • The quality of AI-generated tests is directly proportional to the skill and specificity of the human developer guiding it.
  • Developers must be "active navigators," constantly directing the AI's focus rather than passively accepting its output.
  • Initial AI outputs are often useless or based on common but poor coding patterns, requiring immediate expert correction.
  • A canary test is useful for quickly verifying that the testing environment itself is correctly configured before diving into application logic.
  • Developers should strive to design code as stateless as possible when writing tests to encourage behavior-focused testing over state manipulation.
  • AI exhibits a "stopping problem," meaning it may continue generating unnecessary tests if not explicitly directed to stop or pivot.
  • To elicit desired coding styles (e.g., functional vs. imperative), developers must provide explicit stylistic directives in prompts or skill files.
  • The speaker advises treating the AI as a tool that amplifies intelligence, not a substitute for foundational knowledge.

Practical Notes

  • Workflow: Start with a canary test to validate the environment [7:34]. Follow this with highly specific TDD steps (Test -> Refine Test/Design Interface -> Implement Code).
  • AI Prompting Strategy: Begin with minimal, high-level descriptions of desired behavior rather than complex implementation details. For example, instead of asking for a full class structure, ask for the test asserting the expected outcome for a specific input pattern [15:00].
  • Code Style Directives: To enforce preferred coding styles (e.g., functional), use explicit directives in prompts or skill files, such as "Where possible, write functional style code" [45:00].
  • Risk Mitigation: Always manually verify that the AI-generated tests are logically correct and meaningful; do not assume correctness based on successful generation [31:48, 55:00]. Be mindful of potential source derivation/plagiarism warnings from tools like Copilot [32:06].
  • Design Philosophy: Focus initial tests on defining the "skin" or interface of a method before writing tests that probe its internal implementation details ("guts") [40:15].
