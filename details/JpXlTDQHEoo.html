Ten tips to enhance productivity when using AI-assisted coding tools.

NDC Conferences · 2026-06-22T09:35:08+00:00 · 10 tips to level up your ai-assisted coding - Aleksander Stensby - NDC Copenhagen 2026

What This Video Is About

This video provides ten actionable tips for developers who use AI coding assistants, such as Claude and GitHub Copilot, aiming to move beyond simply delegating tasks to the AI. The central theme is shifting from a passive user role to an active collaborator, treating the AI more like a pair programmer that needs guidance and continuous improvement. The speaker emphasizes that while AI models are rapidly advancing—to the point where they can write working code—the developer's role evolves into one of defining intent, managing context, and overseeing the process.

The presentation argues against using AI in a "lazy way," which involves accepting output without critical review, warning that this approach could lead to skill degradation for developers. Instead, it advocates for an active partnership where the developer guides the AI through processes like "compound engineering"—instructively teaching the AI from its mistakes so it improves over time. The advice is applicable across various coding harnesses but includes specific examples related to Claude's features.

Detailed Walkthrough

The speaker structures the guidance around ten categories, though he notes there are more tips than categories, leading to a high-paced delivery.

Mindset Shift (Tip Zero): Before diving into technical tips, the speaker stresses adopting a pair programming mindset [0:00]. Developers must be comfortable with the quality of AI output and actively engage by asking the AI to explain its work or walk through processes, even in languages they are unfamiliar with, such as Rust when using Claude [5:00]. A key concept introduced is "compound engineering," which involves explicitly telling the AI never to repeat past errors, forcing it to learn from mistakes.

Tip 1: Context is King: The most crucial element when working with AI is providing sufficient context, as the model cannot read minds [10:00]. While models are smart, guiding them by pointing to relevant files or using visual inputs like annotated screenshots instead of purely textual descriptions is highly beneficial. Regarding context windows—which have expanded significantly (e.g., up to one million tokens with Opus)—the speaker advises against relying solely on built-in auto-compaction. He recommends keeping an active view of context usage, aiming not to approach 70% or 80% capacity, as quality deteriorates near these limits [10:00]. For complex tasks, utilizing sub-agents allows for effectively endless context by leveraging each agent's own window. If the window fills up, one must take control of compaction, either by asking the AI to summarize progress into a file or starting a new chat and instructing it to continue from where it left off [10:00]. Context includes not just prompts but also all responses, tool definitions, rules (like Claude MD), and loaded skills.

Tip 2: Actively Manage Rules: Relying solely on the AI's built-in memory is insufficient; active ownership of rules is necessary [15:00]. The developer should maintain trimmed and updated `Claude MD` or `agents MD` files, which act as scaffolding to compensate for model limitations. A critical recommendation is to periodically strip away existing scaffolding—the rules from previous models—to see if the newer, more capable models can perform tasks in novel ways without those constraints [15:00].

Tip 3: Smart Prompting: Instead of overengineering prompts, start with a simple prompt and then ask the AI to interview you. The instruction should be: "What more information do you need from me? What context is missing for you to give me a great answer or produce the best possible outcome?" This forces the AI to extract necessary details rather than relying on pre-packaged instructions [15:00].

Tip 4: Invest in Skills: Developers should invest time in creating "skills," which are essentially markdown files containing instructions that can be bundled with templates and helper scripts. The primary benefit of skills is portability; they are not locked to a single provider, unlike proprietary features [20:00]. These skills function similarly to slash commands and can achieve deterministic behavior from an LLM by bundling code or scripts. Skills should be broken down into smaller, expert pieces that can then be chained together. The best way to write a skill is to first perform the desired work with the coding assistant, get feedback, and then instruct the agent to turn that successful process into a repeatable skill [20:00].

Tip 5: Invocation Control: When using skills, good descriptions are vital because this is what the AI reads when selecting tools. To ensure a skill always runs manually, one can use "disable model invocation" in the front matter of the skill. Conversely, to hide a skill from user view while ensuring it's used by the model, set `user invocable false` [20:00]. For guaranteed execution, especially when using plan mode (a read-only exploration mode), adding the skill to the `Claude MD` or `agents MD` file is recommended so the agent knows which skills are essential for safe codebase exploration [25:00].

Tip 6: Plan First: Always begin complex tasks by asking the AI to generate a detailed plan first. Even if you don't use an advanced coding agent, using general chat to describe what you want and then asking the AI to create a plan is powerful; this thought process can then be fed into a specialized coding agent [25:00]. The goal is to invest heavily in describing the steps, architecture, and implementation plan, as this remains where human judgment is most valuable.

Tip 7: Rethink Bottlenecks: As AI solves more of the coding bottleneck, new bottlenecks emerge. These include the human ability to feed the AI enough work (describing intent) and the subsequent need for evaluation, validation, and testing [30:00]. Developers should start rethinking processes designed for a world where writing code was the scarce resource.

Tip 8: Use HTML for Interaction: To stay engaged during long planning phases, developers can ask AI to present plans in an interactive format instead of a standard markdown file. Asking Claude to present output as an HTML file allows users to interact with architecture and design proposals visually [30:00]. Plugins like "Superpowers" offer this capability by spinning up web servers for design review [30:00].

Tip 9: Leverage Tools and Agent Capabilities: To make the AI agent perform tasks similar to a human developer, it needs access to the same tools. This includes providing access to browsers (like Chrome plugins or DevTools MCP) so agents can verify their own work by interacting with the running application [35:00]. Advanced capabilities include "agentation," which inject an overlay menu bar into applications (e.g., for Next.js apps), allowing users to interact directly with DOM elements and provide feedback that is then fed back as a prompt [40:00]. Furthermore, agents can be used for long-running loops using the `/goal` command, where a separate agent evaluates whether the main goal has been met, preventing lazy completion [40:00].

Tip 10: Release and Parallelize: Since AI tasks take time, developers should embrace parallelization rather than waiting idly. This involves starting new sessions or pieces of work while previous ones run. Tools like native support for "work trees" allow running many concurrent tasks within a single repository [55:00]. Furthermore, agents can be used for automation via scheduling features (like `/schedule` or routines), which can even be moved to cloud environments for execution outside the local machine [1:00:00].

Key Takeaways

  • Adopt a pair programming mindset when using AI; do not accept output passively.
  • Implement "compound engineering" by explicitly instructing your AI to learn from its mistakes and never repeat errors.
  • Treat context as a precious resource, actively monitoring usage and taking control of context compaction before quality degrades (around 70-80% capacity).
  • Actively manage rules via `Claude MD` or `agents MD` files rather than relying on the AI's inherent memory.
  • Periodically review and strip away existing scaffolding/rules to allow newer models to explore novel solutions.
  • Start complex tasks by prompting the AI to "interview" you to extract necessary context instead of over-prompting.
  • Invest in creating portable "skills," which are instruction sets that can be chained together across different providers.
  • Use visual aids, such as annotated screenshots or interactive HTML outputs, to convey complex design intent.
  • Utilize sub-agents and the `context: fork` argument within skills to engineer effectively endless context windows for parallel tasks.
  • Employ hooks (e.g., before tool use) to enforce deterministic guardrails outside of standard inference logic.
  • Use `/insights` in coding harnesses to analyze past sessions, identify repetitive patterns, and generate suggested improvements or rules.

Practical Notes

  • Context Management: Use commands like `/context` or `/status line` (in Claude Code) for visual context window monitoring. For complex tasks, orchestrate sub-agents.
  • Rule Maintenance: When adopting a new model, run the agent to review and propose trims/optimizations for existing `Claude MD` files.
  • Prompting Strategy: Begin with a simple prompt like "What more information do you need?" rather than an exhaustive instruction set.
  • Skill Creation Workflow: Perform the desired work manually or via the agent until satisfied, then instruct the agent: "Now, let's turn this into a repeatable skill." If something is done more than once, it should become a skill.
  • Tool Usage Caution: Be conservative when adding tools/MCPs; only provide access to what is strictly necessary (e.g., rely on the AI using CLI tools rather than loading 70 tool definitions from an MCP).
  • Agent Control: To ensure manual execution of a skill, use `disable model invocation` in its front matter. To force model-only usage, set `user invocable false`.
  • Advanced Automation: For long-running processes, utilize the `/goal` command with a separate evaluating agent to create a "Ralph Wigum loop."
