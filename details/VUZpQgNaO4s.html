Strategies for improving the defensive coding capabilities of AI agents.

NDC Conferences · 2026-07-01T14:20:48+00:00 · Teaching your coding agent defensive coding - Erlend Oftedal - NDC AI 2026

What This Video Is About

This video addresses the inherent security risks associated with using current AI coding agents to generate software. The central argument is that because these agents are trained on vast amounts of internet code, much of which contains vulnerabilities or insecure patterns, they can inadvertently introduce and amplify security flaws into new code. If an agent starts with a flawed codebase, it tends to perpetuate those weaknesses rather than correcting them.

The discussion pivots from simply scanning generated code for existing issues after creation to implementing proactive defensive coding strategies during the planning phase. The speaker proposes using structured guidance skills—specifically leveraging standards like OWASP Application Security Verification Standard (ASVS)—to instruct agents on secure development practices before they write any functional code, thereby shifting the focus toward building robust security in from the start.

Detailed Walkthrough

The presentation begins by demonstrating a classic SQL injection vulnerability in a piece of Python code [0:36]. When this vulnerable code is presented to an AI agent like Claude, it correctly identifies the issue as SQL injection [1:09], but the speaker immediately cautions that this simple success does not represent the general case.

The core problem with current AI agents stems from their probabilistic nature and training data composition [1:21]. They are trained on all available internet code, which includes insecure examples found in public repositories like GitHub [2:12-2:35]. The models lack clear labeling distinguishing secure code from insecure code. Furthermore, the reinforcement learning process has historically prioritized functional correctness (code that compiles and runs) over security [3:02-3:19].

A key finding cited is that "AI is an amplifier" [4:01], meaning if a developer writes bad code, the AI will likely perpetuate or worsen those poor practices. This was demonstrated by showing how an agent would repeat an existing vulnerability in new functions when prompted to complete them [4:58-5:12].

The speaker then explores token prediction mechanisms using a small model example for password storage algorithms [5:19-6:09]. By manipulating the context (e.g., adding unrelated system paths), the agent's suggestions shift, showing how probability can lead to insecure choices like SHA over bcrypt [6:32-6:47]. This probabilistic nature means that even simple prompts like "make it secure" are insufficient because they lack specific, contextual knowledge of what constitutes modern best practices [8:14-8:30].

The proposed solution is a paradigm shift from post-generation scanning to proactive guidance during the planning stage [9:16-9:34]. Instead of asking an agent to fix vulnerabilities (a reactive cycle), the goal is to instruct it *before* coding. This is achieved by creating "security guidance skills" based on comprehensive standards like OWASP ASVS, which details hundreds of verification points across various domains such as cryptography and file uploads [11:13-12:03].

These skills are designed not to dump all 345 verifications into the context, but rather to allow the agent to look up relevant guidance based on its current plan—for instance, pulling database interaction best practices when planning a SQL operation [12:42-12:59]. This provides just-in-time security context.

The speaker also discusses using "hooks" within the agent's coding loop. These hooks inject reminders into the agent’s self-reflection process—for example, reminding it to apply security guidance whenever it edits a file [17:02-18:05]. Testing showed that using such skills leads to consistently better results compared to running agents without them [18:30-18:37].

Finally, the talk contrasts this structured approach with existing tools. While dedicated security scanners are fast at finding known issues (e.g., Open Grip), relying on agents to run these tools can be token-inefficient. For tasks like dependency scanning, using specialized, fast audit functions (like NPM audit) is more efficient than having an agent parse files and then query external vulnerability databases [19:47-20:18].

Key Takeaways

  • AI coding agents are susceptible to introducing security vulnerabilities because they are trained on large datasets that include insecure code from the internet.
  • The "amplifier" effect means AI will reinforce bad coding habits if not explicitly guided toward secure practices.
  • Effective mitigation requires shifting focus from post-generation vulnerability scanning to proactive defensive planning during the agent's initial stages of work.
  • Implementing security guidance skills based on standards like OWASP ASVS allows agents to look up relevant best practices contextually before writing code.
  • These skills provide structured, verifiable advice (e.g., "Verify file sizes") rather than relying on vague prompts like "be secure."
  • Using agent hooks can inject continuous reminders into the coding loop, prompting self-review for security compliance during file edits.
  • For specific tasks, such as dependency checking, using dedicated, fast auditing tools is more token and time efficient than tasking an AI agent to perform the analysis itself.

Practical Notes

  • Core Concept: Implement a "security guidance skill" that interfaces with structured standards like OWASP ASVS [11:13].
  • Workflow Integration: The ideal workflow involves the agent entering a planning mode, triggering the security guidance skill based on the planned action (e.g., database interaction), incorporating that advice into the plan, and then writing secure code.
  • Skill Design: Skills should be designed to provide relevant context only when needed, avoiding massive context stuffing from all 345 ASVS verifications [12:18-12:20].
  • Agent Customization: Agents can be customized with internal security guidelines by creating custom skills or using configuration files like `cloud_security_guidance.md` and `securitypatterns.yaml` (as seen in Anthropic's implementation) to define specific rules [21:48-22:03].
  • Tool Usage: For known, high-volume tasks like dependency scanning, prioritize established tools (e.g., NPM audit) over agentic attempts to replicate that functionality for efficiency [20:06-20:19].
