Guidelines for the responsible abandonment of open source projects.

NDC Conferences · 2026-07-07T09:57:47+00:00 · Responsibly Abandoning Open Source Projects - Jason Turner - NDC Copenhagen 2026

What This Video Is About

This video provides guidelines and best practices for maintaining open source projects to a point where they can be responsibly concluded or "finished," rather than simply abandoned in a way that makes them unusable. The central theme is ensuring that any project, whether it's an internal tool or public software, meets rigorous standards of quality so that future developers can pick it up and continue development without excessive friction or risk.

The speaker argues against the common perception that if a project hasn't seen updates in a year, it is automatically abandoned and unworthy of use. Instead, he advocates for defining "completion" through demonstrable technical excellence—making the code proven, maintainable, resumable, and provable. This approach shifts the focus from mere maintenance to achieving a state where the project is robust enough to stand on its own or be easily continued by others.

Detailed Walkthrough

The discussion begins with the premise of responsibly concluding a long-term open source project after all initial goals have been met and no critical bugs remain [0:00]. The speaker addresses the contradictory sentiment that users wish software would "just leave things alone that work," while simultaneously dismissing projects lacking recent updates. This contradiction stems from placing blame; the goal is to remove any reason for a future developer to feel they must take on unnecessary risk or overhead.

The talk then outlines four core requirements for a project to be considered complete: proven, maintainable, resumable, and provable [15:00].

Proven Code: This involves rigorous analysis of the code without execution. Static analysis tools examine code structure (e.g., checking if JSON or XML schemas are valid) [20:00]. Examples of such tools mentioned include Ruff for Python, Clippy for Rust, and compiler warning levels in C++ and Rust. It is crucial that these tools run automatically, and warnings should be treated as errors to prevent them from being ignored [18:30]. Furthermore, strong typing (e.g., using a structured type like "point" and "color" instead of raw integers) makes code less prone to logical errors by enforcing data integrity at compile time [25:00].

Maintainable: This relates to the ease with which the project can be understood and modified. Key practices include utilizing automatic code formatters (like Rust format or Black for Python) to eliminate style arguments during pull requests, thereby lowering the barrier to entry for contributors [40:00].

Resumable: A project is resumable if a new developer can quickly get it running. The speaker suggests that building and running all tests, static analysis, and runtime checks should take at most five lines of commands for a newcomer [38:50]. This requires simplifying the setup process to avoid lengthy documentation or complex environment configurations. Clean history in source control is also vital so future contributors understand the project's evolution [40:00].

Provable: This demands comprehensive testing strategies. Testing should move beyond simple unit tests into behavioral and integration testing, treating the software like a black box to verify expected outputs [27:30]. Coverage metrics are essential, specifically moving from line coverage to branch coverage, and ideally to path coverage (though path coverage is noted as being extremely expensive and usually applied per function) [30:00]. Beyond standard tests, runtime analysis tools such as sanitizers (like those available in Rust via LLVM for undefined behavior or thread race conditions) should be employed [35:00]. Finally, mutation testing—which checks if the test suite is actually effective by seeing if small code changes cause existing tests to fail—is recommended to ensure test validity [36:40].

The speaker advises a strategic approach to tooling: use every possible tool during active development, but when ramping down, pare the list back to only industry-standard tools to minimize friction for future maintainers who may be using newer compilers or dependencies [45:00].

Key Takeaways

  • A project is considered "finished" not just because it has no open bugs, but because it meets criteria of being proven, maintainable, resumable, and provable.
  • Static analysis tools (like Ruff or Clippy) should be integrated into the workflow to analyze code without execution, treating warnings as errors to enforce quality.
  • Code formatting must be automated using tools like Rust format to prevent style-related friction in pull requests.
  • The build and test process for a new contributor should ideally execute within five lines of commands.
  • Testing strategies must progress beyond line coverage to include branch coverage, and ideally path coverage on individual functions.
  • Runtime analysis is critical; developers should investigate sanitizers (for memory/thread issues) and fuzz testing (to find unexpected input errors).
  • Mutation testing verifies the quality of the test suite itself by ensuring tests fail when small code changes are made.
  • When concluding a project, triage all existing issues to ensure no open problems remain that undermine the claim of completion.
  • Do not use GitHub's "archived" feature unless there is a proven successor fork; maintain ownership until the decision to cease active care is final.
  • For long-term viability, limit tooling during the winding-down phase to only industry standards to prevent new warnings from breaking older codebases due to updated compilers or dependencies.

Practical Notes

  • Static Analysis Tools: Examples include Ruff (Python), Clippy (Rust), and compiler warning levels in C++/Rust. Clang tidy is mentioned as a free, open-source static analysis tool maintained by Google developers for C++.
  • Code Formatting: Automatic formatters should be used to eliminate style debates during code reviews. Specific tools mentioned are Rust format and Black (Python).
  • Build/Test Workflow: The goal is to consolidate the build command so that it automatically runs all tests, static analysis, and runtime checks in a single step, ideally within five lines of setup for new users.
  • Testing Coverage Hierarchy: Line coverage -> Branch coverage -> Path coverage. If branch coverage isn't measured, path coverage almost certainly won't be either.
  • Runtime Analysis: Developers should investigate sanitizers (e.g., undefined behavior sanitizer, address sanitizer) and fuzz testing to catch issues like memory errors from unexpected inputs (e.g., parsing an empty string).
  • AI Assistance: An AI agent can be used to monitor open source tools for new bug reports or help implement requested features by automatically creating forks and pull requests.
