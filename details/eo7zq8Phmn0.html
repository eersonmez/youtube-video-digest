The talk explores postmodern programming as a way to rethink C++ design, APIs, code organization, and developer communication.

NDC Conferences · 2026-07-22T09:30:09+00:00 · Postmodern Programming - Tony Van Eerd - NDC Toronto 2026

What This Video Is About

Tony Van Eerd presents “postmodern programming” as a way to rethink C++ design rather than as a language feature or a new version of “modern C++.” His central argument is that code gets its meaning from context, contrast, conventions, and the way other people use it. The talk uses rhymes, tweets, graphs, references, self-referential slides, and deliberate reversals of familiar patterns to demonstrate that idea while discussing practical C++ choices.

The technical examples focus on reducing misunderstanding. Van Eerd recommends separating state maintenance from decisions about behavior, choosing `vector` as a default container and documenting deviations, designing APIs from their call sites, and choosing names that do not create false confidence. A name such as `has_value` is useful when its meaning is conventional, but a deliberately unusual name such as `valueless_by_exception` can be better when the familiar name would cause callers to assume the wrong behavior.

The later sections apply the same reasoning to formatting, abstractions, and function boundaries. Context-sensitive formatting can communicate emphasis or make debugging easier, while rigid automatic formatting can sometimes obscure intent. Small functions are presented primarily as tools for cognitive isolation, not merely for reuse. The final message is that code communicates simultaneously with the compiler, the computer, end users, other developers, and possibly AI systems. Better code therefore begins with changing how the programmer thinks about context, ambiguity, and responsibility.

Detailed Walkthrough

A deliberately constrained form for a talk about communication

At [02:29], Van Eerd explains that the presentation will mix short rhymes, tweets, jokes, graphs, and longer sections meant to leave the audience with ideas to consider. He says that almost half the slides are graphs, although many exist mainly for humor. The unusual structure is relevant to the topic: the talk repeatedly examines how a message changes when its form, audience, or surrounding context changes.

He jokes about the phrase “modern C++,” pointing out that C++ is an old language and that discussions of “modern” C++ have existed for many years. He then proposes “postmodern C++” as the next framing device. The term is used rhetorically to mean questioning established patterns and examining how programming conventions communicate, rather than adopting a particular library or language feature.

Deference: meaning comes from difference and context

At [05:45], the talk introduces “deference,” described as a blend of difference and deference. The idea is that a word or construct is not understood in isolation. Its meaning is influenced by the alternatives that were not chosen and by the surrounding words.

Van Eerd illustrates this with ordinary language. A house might instead have been called a mansion, a hovel, or a nest. The selected word becomes meaningful partly because it differs from those alternatives. The same reasoning applies to programming names and constructs: a name communicates not only what it says directly, but also what a reader expects from nearby or similar names.

The talk then applies the idea to a user-interface interaction from Van Eerd’s time at BlackBerry. On a phone, tapping and holding can produce a context menu. If the user scrolls and then pauses, however, the same menu should not appear, because the pause is probably part of reading rather than an invitation to open a menu [06:45]. The code tracks whether movement occurred and uses that state when deciding whether the menu may appear.

Van Eerd criticizes a double-negative condition in the example. Renaming or restructuring the condition can leave the compiler’s logic exactly unchanged while making the programmer’s interpretation easier [07:40]. The important distinction is between machine semantics and human communication: two forms can execute identically but impose different mental effort on the reader.

The deeper problem is ownership of the state. If another part of the program also wants to disable the menu, it becomes unclear whether the variable represents a menu setting, scrolling state, or something else. Resetting the variable in mouse-handling code could also overwrite a decision made for another reason [08:24]. Van Eerd’s refactoring is to give the responsibilities clearer boundaries: the mouse code maintains the mouse-related state, and separate code reads that state and decides the menu’s fate [09:07].

He summarizes the principle as separating state from deciding. One portion of the code records what has happened; another portion determines what behavior follows from the current state. A short comment can then document this division of responsibility rather than trying to explain a tangled block of code [10:00].

The same idea applies to names more generally. At [10:48], Van Eerd says that the meaning of a function, variable, or thrown exception may be “deferred” to its context. A name such as `fu` has no useful meaning by itself; its role is established by where and how it is used.

Defaults, deviations, and deconstruction

At [11:18], the container discussion turns into a strong default recommendation: use `vector`. Van Eerd emphasizes that a vector is contiguous in memory and contrasts it with `list`, whose useful properties are different. Each container has situations where its traits matter, but his practical guideline is to start with `vector` rather than reaching automatically for another container.

He presents the guideline in rhyme: when choosing a container, remember that vector is best, and leave a comment when choosing something else [11:47]. He says that wording was once included in the standard and was later removed. Regardless of the status of that exact wording, the design rule is clear: use the habitual default unless there is a reason not to, and make the reason visible.

At [12:15], he generalizes the container advice into a broader pattern. Whenever the typical pattern does not fit a particular problem, a comment should explain why the code departs from the expected approach. He jokingly names this a pattern for resolving muddled patterns and then applies the idea recursively: if a programmer fails to use the expected pattern and fails to explain that departure, they could comment on why they did not write the explanatory comment. The recursion is a joke, but the practical rule is straightforward: exceptions to convention should be legible to the next person who encounters them.

The talk next introduces deconstruction at [13:38]. In this context, deconstruction means taking a familiar pattern apart, reversing its assumptions, and examining what it reveals. Van Eerd uses this approach to challenge the common idea that an API is primarily defined by the code that implements it.

His alternative formulation is that an API is defined by the code that calls it. The important contract is formed at the junction where client code meets the API, so examples and tests should be written early to expose how the interface will actually be used [14:22]. Once an API’s usage has spread through a codebase, changing it becomes disruptive. The call sites therefore deserve attention before implementation details become entrenched.

API names should prevent misunderstanding

At [15:26], Van Eerd compares several value-checking APIs. An optional integer can either contain an integer or be empty, so `has_value` communicates its purpose clearly. An `any` can hold many kinds of values and also uses `has_value`. An `expected` contains either a value or an explanation of failure, and `has_value` again fits the question being asked.

The example changes with `variant`. In the talk’s simplified example, a variant can hold a red or a blue. The relevant unusual state is represented by the function name `valueless_by_exception`, rather than another generic `has_value` function [16:47]. Van Eerd says he contributed that name to the standard.

The deliberately specific name is important because a familiar name could be misleading. If the function were called `has_value`, a programmer might assume it behaves like the corresponding functions on optional, any, or expected. The programmer would feel that they understood the API while actually applying the wrong mental model. `valueless_by_exception` is less immediately understandable, but that forces the reader to investigate instead of encouraging a false assumption [18:22].

This leads to one of the talk’s clearest API principles: not understanding is safer than misunderstanding. Something that is unfamiliar can be looked up. Something that appears familiar but has different semantics can silently produce incorrect code.

Van Eerd also gives a broader consistency rule at [19:08]. Consistent naming makes deviations visible. He recalls that related standard functions initially used names such as `is_empty`, `has_value`, and `is_valid`. Once a family of APIs adopts a consistent vocabulary, an intentionally different name stands out and signals that the semantics are different. Consistency is therefore not only about aesthetics; it helps readers detect meaningful exceptions.

Formatting is part of the message

At [19:56], the talk turns to formatting and bracket placement. Van Eerd argues that formatting should be considered in the social and technical context in which the code is written. He gives screen size as one example: the formatting style that works on a small screen may not be the style he prefers on a large screen.

He also points to code layouts that make debugging harder because the logical pieces are disjoint and it is unclear where a breakpoint should go [20:29]. Other layouts can be confusing when a programmer switches between languages such as Python and C++. The point is not that one universal layout is always wrong, but that layout affects how the reader navigates and interprets the code.

Van Eerd distinguishes between compact code intended to remain inconspicuous and expanded code that deserves special focus [21:45]. He changes styles according to the circumstances. His criticism of fixed formatting is that it can be like forcing every sentence to rhyme: the pattern may be consistent, but it can interfere with understanding.

He is particularly critical of rigid use of `clang-format`, saying that he would rather be able to write code according to the communicative need of the moment [22:26]. He also cautions against taking flexibility too far. The practical position is not “format randomly,” but rather “do not let an automatic or habitual layout override the structure you are trying to communicate.”

The formatting section ends with an intentionally extreme example: an award-winning piece of obfuscated code that calculates pi [22:38]. Van Eerd says that changing the diameter parameter makes the circle larger and produces a more accurate figure. This is presented as a humorous boundary case, not as a recommended style. It illustrates that code can be technically clever while still being difficult to read.

Graphs, ambiguity, and the cost of unfamiliar paradigms

The graph sequence from [23:22] uses humor to continue the argument about complexity. Van Eerd jokes about the relationship between bugs and lines of code, first treating it as if every line introduced a bug and then suggesting that the real curve becomes worse as code grows. He mentions adding an `if` as a possible fix while noting that the additional line can also obscure big-picture understanding.

These graphs are not accompanied by measurements or empirical data; the speaker introduced them as mostly being for laughs. Their purpose is to make the audience question apparently simple relationships and to show how a familiar graph format can imply more certainty than the content supports.

A rabbit-versus-duck image and several self-referential slides extend the same theme: what a reader sees depends on the pattern and context supplied by the presentation. A slide or projector interruption then becomes a segue into self-referential code [25:48].

Van Eerd demonstrates a class that refers back to itself using a CRTP-style arrangement. The derived class is passed into its base class as a template parameter, allowing the base to obtain a `self` representation [26:25]. He considers changing a pointer to a reference but notes that C++ rules require substantially more code to make the apparently simple improvement work.

The conclusion is not that the technique is impossible. It is that a codebase using an unfamiliar `self` paradigm would surprise many maintainers, introduce another way of writing C++, and create additional learning costs. Van Eerd also says the approach does not work cleanly with multiple inheritance [27:42]. His broader warning is that too many paradigms are difficult to maintain. A pattern should justify the new concepts and conventions it imposes.

Functions and subroutines as cognitive isolation

At [28:22], Van Eerd jokingly dates postmodern programming to 1951, when David Wheeler—identified in the talk as Stroustrup’s thesis adviser—invented the subroutine. He cites a short report defining a subroutine as a self-contained part of a program and an entity of its own.

The report’s significant claim is that subroutines are usually advantageous even when a program does not strictly need them. A programmer can concentrate on one section at a time while the rest of the program does not intrude [29:03]. The subroutine creates a boundary around a problem, making it possible to reason about and test that part in isolation.

Van Eerd acknowledges reuse and libraries as additional benefits. Code can be moved from one place to another, although he says that library dependency management is particularly difficult in C++. Reuse matters, but he explicitly places it after separation of concerns and isolation.

He then describes a codebase containing a linear sequence: step one does some work, followed by steps two, three, and four [30:39]. The original author might see no reason to extract step one into a function because it is already short and appears only once. Van Eerd challenges that assumption. A meaningful function name can identify the responsibility of the step; he gives the idea of a name such as “validate that” instead of leaving the operation as an anonymous block.

He also asks how many copies of the same code must exist before it should become a function. The audience suggests three, but Van Eerd’s answer is one [31:41]. Duplication is not required before a responsibility can be named and isolated. Even a single occurrence can benefit from a function boundary if the boundary makes the surrounding code easier to understand.

A related joke describes a coworker suggesting that a repeated explanatory comment be placed inside a lambda and called from common locations [32:13]. The underlying idea is to express a repeated responsibility as executable, named structure rather than relying on duplicated comments.

Van Eerd identifies the main obstacle as fear of losing local context. When working inside a large problem, the programmer’s mind holds a detailed map of how all the pieces fit together. Extracting a function can feel like discarding that map. His counterargument is that the function boundary is precisely what reduces the amount of context that must remain active in the programmer’s head [32:56]. Each function can handle one smaller problem, leaving fewer unrelated details in working memory. The primary benefits are focus, separation, and isolation; reuse is secondary.

Multiple readers and the final communication model

At [33:51], Van Eerd presents his final tweet: two readers of the same source may see different values because of what each reader read and wrote previously. He explains the idea at several levels. An English student interprets a text in light of prior reading. A programmer interprets source code in light of prior knowledge and habits. A CPU thread reading memory, especially in lock-free code, encounters a value whose meaning depends on what other threads previously read and wrote [34:12].

The example reinforces the talk’s main definition of context. The same text or memory location does not have a complete meaning independent of its reader and history.

The conclusion then turns back on the presentation itself. At [35:55], Van Eerd explains why he used short-form social-media posts, references to Dr. Seuss and Bruce Lee, graphs, and self-referential slides. Twitter, Mastodon, and Blue Sky are described in terms of character limits, which parallels programming as communication inside a constrained environment. Rhyming adds another constraint. References can make a point compactly for an audience with shared experiences but can fail when the audience does not share those references.

He says the graphs follow the pattern expected of a technical talk even when many do not contain meaningful data. Deconstruction is meant to give the audience a new perspective by making it examine its own assumptions. Bugs are described as a manifestation of ambiguity and misplaced patterns rather than the fundamental source of the problem [38:38].

The final conclusion arrives at [41:31]. Writing code means communicating with the compiler, but the compiler is only an intermediary. The programmer also wants the computer to perform an operation, the end user to receive the intended behavior, other developers to understand and maintain the code, and perhaps an AI system to interpret it. These audiences must be considered at the same time. Van Eerd’s final instruction is to stop thinking about programming too narrowly and to change the way one thinks, because code tends to reflect the programmer’s mental model.

Key Takeaways

  • “Postmodern programming” is presented as a mindset for questioning habitual patterns, not as a C++ language version or a specific library technique.
  • Meaning in code depends on context and contrast. A name is understood partly through the alternatives that were not chosen.
  • Separate state maintenance from decisions about behavior. In the menu example, mouse code owns mouse-related state while separate code decides what the menu should do.
  • Renaming a condition can improve human understanding without changing compiler behavior. Machine equivalence does not imply equal communicative quality.
  • Treat `vector` as the default container according to the speaker’s recommendation. When another container or pattern is chosen, document the reason.
  • API design should begin with how client code will call the interface. Examples and tests expose the real contract before implementation details spread.
  • Prefer names that prevent false assumptions. `has_value` is appropriate when its semantics match related APIs, while `valueless_by_exception` deliberately avoids suggesting that it behaves like an ordinary value check.
  • Consistency makes meaningful exceptions visible. A family of consistent names helps readers notice when one operation has genuinely different semantics.
  • Formatting is communication. Layout can support debugging, emphasis, and readability, while rigid automatic formatting can sometimes obscure intent.
  • Extract a cohesive responsibility into a function even when it appears only once. The main benefit is reducing mental load through separation and isolation, not merely enabling reuse.
  • New paradigms such as the self-referential CRTP example carry learning and maintenance costs. Elegance is not enough reason to add another way of writing code.
  • Code addresses several audiences at once: the compiler, the computer, end users, developers, and possibly AI systems.

Practical Notes

For a state-and-decision refactoring, begin by identifying what the state actually represents and listing which parts of the program read and write it. In the talk’s menu example, the mouse-handling code is the only code allowed to change the mouse-related state. The menu logic reads that state and makes the behavioral decision. Keep resets and updates with the state owner, and use a short comment to record the separation of responsibilities. The exact variable name is less important than making ownership unambiguous.

For container selection, begin with `vector` rather than selecting a more specialized container from habit. If a `list` or another container is required, record the reason beside the choice. The talk does not provide a full decision table for every container, so the comment should explain the actual constraint that made the default unsuitable rather than merely repeating the container’s name.

For API design, start from intended call sites. Write representative examples or tests before settling the implementation, then inspect whether the names lead a caller toward the right interpretation. Use a familiar name such as `has_value` only when the operation really has the expected meaning. If the state or failure mode differs, a more specific name may be preferable even if it requires documentation or a lookup. Keep related operations consistent so that deliberate exceptions stand out.

For function extraction, look for a coherent responsibility rather than waiting for code to be duplicated. A sequence such as “step one” may deserve a function because the function name tells the reader what that step accomplishes and keeps its internal context out of the caller. Use a lambda when the responsibility is local and needs to be shared within a limited scope. The talk recommends testing extracted responsibilities in isolation, but it gives no universal function-size threshold or naming formula.

For formatting, choose a layout based on what the code needs to communicate. Expanded formatting may give an important condition or debugging location more focus; compact formatting may keep an incidental detail from dominating the reader’s attention. `clang-format` is specifically criticized when treated as a rigid authority. The transcript provides no configuration or command, so the actionable guidance is to review formatter output rather than blindly accepting it when it damages emphasis or breakpoint placement.

Avoid introducing a self-referential or CRTP-based convention merely because it is technically possible. The example adds code, requires readers to understand another paradigm, creates pointer-versus-reference complications, and is said not to work cleanly with multiple inheritance. If such a pattern is necessary, document the reason and consider whether the team can maintain it without making ordinary code surprising.

Finally, review code for all of its audiences. Ask whether the compiler receives the intended instructions, whether the computer will perform the required work, whether the user receives the intended behavior, whether another developer can infer the responsibilities, and whether an AI system would be led toward the same interpretation. The talk supplies principles and examples rather than build commands, complete implementations, or a prescribed architecture.
