What This Video Is About

This video provides an in-depth technical exploration of how modern Artificial Intelligence agents operate by focusing on three core concepts: function calling, Model Context Protocol (MCP), and tool usage. The central theme is to move beyond the "mythical understanding" of AI agents—where a user simply interacts with a polished application—to understand the underlying mechanics that allow these systems to perform complex tasks.

The presentation details how Large Language Models (LLMs) infer when and which external tools to use based on system prompts and user intent. It addresses common pitfalls, such as over-trust in tool outputs or selecting inappropriate tools. Furthermore, it distinguishes between standard function calling within a single application context and the Model Context Protocol (MCP), which is necessary for building distributed AI applications that interact with external services.

Detailed Walkthrough

The session begins by establishing the foundation of LLMs, noting that they operate by predicting the next token in a sequence based on input tokens [0:15]. The speaker contrasts this basic prediction capability with the functionality of an agentic application. When a user asks a question like "What's the weather like in Oslo?", a direct query to the LLM without tools results in a response stating it lacks real-time data due to its training cutoff [5:00].

To solve this, tool definitions are provided to augment the request. For OpenAI, this mechanism is called function calling. The model receives a definition for a function, such as "get weather," which accepts parameters like an object containing a "city" string. Instead of answering directly, the LLM responds with a structured message indicating it has finished reasoning and requests the agent application to execute that specific function call [5:00]. The agent then simulates executing this by returning a response (e.g., "nice and sunny in Oslo"), which the LLM uses to formulate its final answer.

The process is more complex when user confirmation is required, such as for booking flights or hotels. In this flow, the LLM infers the necessary tool call based on the system prompt and user request but can be configured to ask the user for approval before execution [7:30]. It is crucial to understand that the LLM itself does not execute the tool; it only decides which tool to call, and the surrounding agent framework handles the actual invocation via APIs from providers like OpenAI or Anthropic [8:15].

The speaker demonstrates this using a pizza ordering agent. Even without explicit instructions in the system prompt, the model infers that it can fulfill complex requests (e.g., "three large veggie pizzas, one with extra olives") by making multiple calls to the `add pizza to order` tool [13:00]. This showcases the model's ability to reason through a request into discrete, actionable tool calls.

The discussion then transitions to Retrieval Augmented Generation (RAG), evolving from its naive early form—where a search service was called once based on the user query—to Agentic RAG. In agentic RAG, the LLM acts as an intelligent orchestrator; it decides *if* and *how* to use the search tool [20:30]. A key advancement is "query rewrite," where the LLM composes a refined search query based on conversational context (e.g., rewriting "Tell me about the race" into "information about the Canadian Grand Prix") before querying the service [21:45].

The speaker details token management issues in RAG, such as "token blow" when feeding large document snippets into the chat history. A solution presented is using a summarization tool—which employs another LLM to summarize page content based on a specific interest (e.g., "Oslo history") before it enters the prompt, significantly reducing token consumption [23:45].

The concept of MCP is introduced as the communication layer between an agent and an external MCP server, distinct from the internal model context (which includes system prompts, history, etc.) [40:15]. While function calling works within a single application boundary, MCP enables distributed applications. The speaker demonstrates this with an `MCPDJ` server that provides tools like listing tracks or playing music, which can then be integrated into clients like Semantic Kernel to enhance tools such as GitHub Copilot [46:30].

Finally, the session covers security and reliability concerns. Models are non-deterministic, meaning identical inputs can yield different outputs [15:00], and their behavior changes across providers (e.g., Azure AI Foundry vs. OpenAI) [1:00:00]. Furthermore, models can be tricked by tool responses—even intentionally flawed ones—or exploit vulnerabilities in external MCP servers to perform unintended actions like generating non-scannable QR codes using image generators instead of code interpreters [30:00].

Key Takeaways

  • AI agents infer tool usage based on user intent and the definitions provided, rather than explicit instructions in the system prompt.
  • Function calling allows LLMs to interact with external capabilities (like weather APIs) by outputting structured calls instead of direct answers.
  • The Model Context Protocol (MCP) is a communication standard enabling agents to interact with distributed MCP servers, going beyond single-application function calls.
  • Agentic RAG elevates traditional RAG by allowing the LLM to intelligently decide when and how to use search tools, including rewriting user queries for better results.
  • Token management in RAG can be optimized by using specialized summarization tools that pre-process large document content based on specific interests.
  • LLMs are non-deterministic; their behavior can change over time or between different model providers, necessitating robust testing and caution when building agents.
  • Tool trust is a critical vulnerability; models may accept incorrect outputs from tools without questioning them, even if the error is obvious.
  • MCP allows for extending agent capabilities by connecting to external services (like music players in VS Code) that are not part of the core LLM functionality.

Practical Notes

  • Tool Implementation: When building agents, you interact with model providers via APIs or SDK abstractions, which manage chat history and tool definitions [9:00].
  • Agent Framework Usage: The speaker utilized LangChain, Semantic Kernel, and a custom agent framework to build the pizza ordering demo. These frameworks allow for swapping LLM providers (e.g., AI Foundry vs. OpenAI) [10:30].
  • Tool Definition: Tools must be defined with clear names and parameters; however, the model's ability to infer usage can sometimes succeed even with minimal parameter specification [14:00].
  • Security/PII Handling: Sensitive information (like credit card numbers) provided in system prompts or retrieved via RAG can be passed directly into tool methods if not explicitly blocked by content filters, regardless of the model provider used [40:30]. Content filters and PII protection features are customizable within platforms like Azure AI Foundry [42:15].
  • MCP Implementation: An MCP server (e.g., `MCPDJ`) can be built using Python with FastMCP to expose tools (list tracks, play track) that external clients (like Semantic Kernel) can utilize in a distributed manner [46:30].
