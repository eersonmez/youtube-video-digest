Charity Majors discusses the evolution of observability in software engineering.

Scott Hanselman · 2026-06-21T22:07:58+00:00 · "Observabilitying" the Future of Software with Charity Majors

What This Video Is About

This video features an interview with Charity Majors, CEO and co-founder of Honeycomb, discussing the evolution and modern understanding of observability in software engineering. Majors argues that contemporary observability transcends simply collecting metrics, logs, and traces; it fundamentally requires structured, connectable data—what she refers to as "traces with spans"—that is treated as a product problem for controlled software. She contrasts this necessary rigor with older methods, such as manually grepping log files from decades past.

The discussion heavily emphasizes the critical need for robust feedback loops between code and production telemetry, especially in an era where AI rapidly generates code. Majors posits that while AI can accelerate content creation, it cannot replace human experience or judgment. Furthermore, she advocates strongly for investing in junior engineers to develop this essential human intuition, as true engineering skill is not merely about writing code but about understanding the system's behavior under real-world conditions.

Detailed Walkthrough

The conversation begins with Majors establishing a key shift in thinking regarding observability [1:59]. She asserts that modern practice requires moving beyond just collecting disparate signals like metrics, logs, and traces, arguing instead for structured data that is inherently connective, which she equates to "traces with spans" [2:06-2:22]. The core idea presented is that the telemetry generated by software you control—your application code—is a product problem, whereas traditional infrastructure outputs like metrics and logs are merely the exhaust pipe of systems you operate but do not fully control [2:44-2:51].

Majors uses an analogy to describe debugging at scale. Previously, engineers relied on simple print statements (like console.writeline) or energy-intensive methods like taking screenshots of pixels for visual debugging [3:30-3:52]. She contrasts this with using observability across the entire system, which allowed her to solve a problem within minutes by pointing an MCP server to the observed data via an agent loop [3:57-4:05].

A major point she stresses is that treating telemetry as a product investment yields massive payoffs. When data points are properly structured and connected, they can be analyzed in countless ways—across different dashboards or visualizations—but only if they were knit together initially [5:19-5:27]. She contrasts this with the historical practice of FTPing log files and grepping through them [5:38-5:45].

The discussion then addresses the failure of past DevOps efforts to create a single, effective feedback loop between code writers and production reality. An engineer in 2020 might struggle deciding whether data should be a metric, log, or trace, and even if they instrument it correctly, finding and acting upon that data requires significant labor [6:30-7:15]. AI offers the potential to close this loop by providing understanding as code is written [7:35-7:45].

Majors introduces her second edition of "Observability Engineering" [7:52], noting it has been entirely rewritten and expanded. She frames observability not just as knowing what's wrong, but as reasoning from first principles about how to understand the code itself, differentiating this from monitoring which merely tracks established grooves in a system [10:49-11:04].

The conversation pivots to the role of AI. Majors cautions against the notion that because AI generates code quickly, engineers can simply let it "wash over" them without understanding. She argues that if you are creating massive amounts of code and trying to sleep through it, production failures will inevitably occur [8:16-8:34]. The key is establishing better feedback loops; otherwise, implementing AI agents on systems lacking observability is likely doomed [12:09-13:12].

Finally, Majors discusses the necessary human skills for senior engineers in the age of AI. She argues that technical transformations succeed only when backed by trust and credibility within an organization. When partnering with external vendors, she advises building trust through reciprocity rather than assuming immediate belief [16:35-17:05]. Furthermore, she strongly advocates for investing in junior engineers because developing judgment—the ability to think critically beyond the visible artifact of code—is a human endeavor that AI cannot replicate [26:44-27:14].

Key Takeaways

  • Observability is fundamentally a product problem concerning controlled software, not just an infrastructure concern.
  • Modern observability requires structured, connectable data (traces with spans) rather than fragmented metrics and logs alone.
  • The primary goal of modern engineering practice should be establishing strong feedback loops between code creation and production telemetry.
  • AI's ability to generate code faster necessitates better human understanding and system observation; letting code "wash over" you is insufficient for reliability.
  • Junior engineers are vital because developing judgment—the capacity for critical thinking beyond the immediate artifact—is a uniquely human skill that cannot be replaced by agents.
  • In organizational change, senior engineers must build trust and credibility internally, while building reciprocity when dealing with external vendors.
  • The unit of work in software engineering is shifting from transactions to traces, requiring new levels of observation up the stack.
  • Testing in production ("Schrödinger's production") is an unavoidable reality; safe experimentation requires tooling like feature flags and progressive delivery.

Practical Notes

  • Data Structure: Focus on making data "connectable" rather than just collecting signal types (metrics, logs, traces).
  • AI Implementation Risk: Implementing AI agents or agentic loops on brownfield systems lacking observability is highly risky ("deeply screwed").
  • Production Safety: Treat every deployment as a production experiment. Utilize tools like feature flags and canary deployments to safely roll out changes incrementally.
  • Engineering Focus: Shift focus from merely writing code to understanding the system's behavior across all intersections of software, infrastructure, process, time, and users.
  • Organizational Skill Building: Invest in junior engineers by creating safe environments where they can fail and experiment, thereby developing necessary judgment.
