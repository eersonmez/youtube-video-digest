This talk explores methods and vulnerabilities in building secure AI agents.

NDC Conferences · 2026-06-26T12:55:23+00:00 · HitchHacker's Guide to Building Secure Agents - Kenneth Yeung & Kasimir Schulz

What This Video Is About

This video presents a deep dive into the security vulnerabilities inherent in building and deploying AI agents. The speakers, Kenneth Yeung and Kasimir Schulz from Hidden Layer, demonstrate risks using three custom-built agent examples. The central theme is that traditional security models are insufficient for agentic systems because these agents possess capabilities—such as accessing private data, communicating externally, and processing untrusted content—that create unique attack surfaces.

The presentation moves beyond standard application vulnerabilities to address how LLMs can be manipulated through prompt injection or by exploiting the features they interact with (like image rendering or URL handling). A key concept introduced is the "lethal trifecta," which posits that when an agent has access to private data, external communication capabilities, and exposure to untrusted content, defending it becomes nearly impossible.

The talk provides concrete examples of attacks against applications like PageTicket and discusses hardening strategies for agents, covering system prompt design, context window management, and the limitations of sandboxing. It serves as a guide for developers building agentic systems to adopt a proactive threat modeling approach that accounts for LLM-specific risks.

Detailed Walkthrough

The presentation begins by introducing three custom-built agents: PageTicket, a help chatbot/social media platform agent, and an example based on tools like Cloud Code or Cursor.

PageTicket Demonstration [0:15]: PageTicket is described as having two components: "pages," which are flexible, collaborative live documents supporting HTML, images, and knowledge tracking (with traditional XSS protections in place), and "tickets," used for task management across an organization. The system includes a search function that can operate via keyword search or more complex embedding-based semantic search. An AI agent is built to summarize these search results or answer direct user questions by feeding all workspace documents and tickets into its system prompt within a context window [5:00].

Initial Security Analysis of PageTicket [12:34]: The speakers analyze the system prompt handling, noting that appending raw workspace content directly into the system prompt elevates everything to the system level, preventing granular privilege enforcement. A major risk is that the workspace content can be controlled by numerous sources (users editing tickets/documents), giving attackers no real control over what enters the agent's context.

Attack Vectors on PageTicket [13:45]: Two main attack vectors are identified: direct prompt injection within the user question, or indirect attacks where an attacker modifies content in a ticket or page that the model reads. Since initial demos were scoped to only Ken's data, they needed a way to access private information. They discovered that exfiltration could occur via markdown rendering because this process sends data to remote sources when attempting to render images [10:00].

Zero-Click Attack via iFrames [10:30]: The speakers demonstrate a zero-click attack where an attacker embeds an iframe on the PageTicket page that renders the same page with search enabled. This automatically triggers a direct prompt injection, allowing data exfiltration without user input. The mechanism relies on GET requests for link previews or image rendering; the model can be prompted to replace static data placeholders with content from secret tickets [10:00].

Mitigation Challenges and Creative Exfiltration [13:50]: Simply blocking markdown is insufficient because custom parsers convert markdown into HTML, allowing attackers to use standard HTML image tags for exfiltration. Furthermore, other features like custom profiles or links can also be leveraged for data leakage. The speakers cite external examples of "volume leak" on Android devices where controlling volume rockers allowed an LLM (Gemini) to infer conversation topics based on packet sizes and timing with high accuracy [15:00].

The Lethal Trifecta [15:45]: This concept, derived from Simon Wilson's work, is presented as the agentic evolution of the CIA triad. It states that if an agent has access to private data, the ability to externally communicate (exfiltrate), and exposure to untrusted content (prompt injection vector), defense becomes nearly impossible [15:45].

PageTicket Takeaways [20:30]: For PageTicket specifically, recommendations include restricting image and URL rendering to trusted domains controlled by the organization. If dynamic URLs are necessary, they must remain static and not be modified by the model itself. Developers must consider all tools the model controls—what it can alter or read automatically—as potential exfiltration points [20:30].

Help Chatbot Agent Vulnerabilities [19:45]: The second demo focuses on a help chatbot agent integrated with a social media platform, designed to automate customer support tasks. The speakers examine how client-side storage of chat history (instead of server-side) enables "client side forgery" using AI techniques. An attacker can manipulate the conversation flow by pretending to be the AI's response, effectively trick the model into revealing its system prompt through policy puppetry [24:00].

System Prompt Hardening and KV Cache Costs [30:00]: The speakers discuss hardening system prompts by explicitly defining what the agent *can* and *cannot* do. They also address operational costs related to token usage, specifically mentioning the Key-Value (KV) cache. When a part of the system prompt changes, it invalidates the KV cache for that context, forcing recalculation of all previous tokens, which increases latency and cost significantly [30:00]. Static information should be placed near the top of the system prompt to maintain caching efficiency.

Agent Tool Usage and Persistence [45:00]: The final example examines agents like Cloud Code or Cursor, which have extensive tool access (file system read/write, shell execution). The speakers demonstrate indirect prompt injection by embedding instructions within external data sources, such as a `memory.md` file that the agent is designed to update upon summarizing external content. This grants permanent persistence on the user's machine with all associated tools [45:00].

General Agent Hardening Strategies [30:30]: Two primary hardening components are discussed: 1) Hardening the System Prompt, by explicitly instructing the model on its boundaries (what to deny, what it is allowed to do), which raises the noise level of malicious queries. 2) Guardrails and Tool Output Monitoring. Guardrails should not only check user prompts but also tool outputs. Furthermore, sandboxes protect the system environment but do not secure the data flowing through them; sensitive files needed by the agent remain a threat surface [50:00].

Key Takeaways

  • The "lethal trifecta" defines high-risk agents as those possessing access to private data, external communication capabilities (for exfiltration), and exposure to untrusted content vectors.
  • Direct prompt injection is not limited to user input; it can be triggered indirectly through features like image rendering or URL previews in applications like PageTicket.
  • Traditional security measures like XSS protection are often bypassed because LLMs can manipulate static rendering mechanisms (e.g., using HTML tags instead of markdown).
  • Client-side storage of chat history, while potentially cost-saving, allows for sophisticated AI-driven attacks that can force the model to reveal its system prompt through policy manipulation.
  • System prompts must be carefully structured; placing dynamic or frequently changing data early in the prompt invalidates the KV cache, leading to increased latency and operational costs.
  • Hardening agents requires defining explicit boundaries in the system prompt (what is allowed vs. disallowed) rather than relying solely on preventing injection attempts.
  • Guardrails must monitor not only user prompts but also tool outputs, as malicious activity can be hidden within the agent's interaction with its environment.
  • Sandboxing protects the execution environment of an agent but does not inherently secure the sensitive data that must flow into or out of that sandbox for the agent to function.

Practical Notes

  • PageTicket Vulnerability: Image rendering and URL handling are primary vectors for exfiltration; restrict these features to trusted, organization-controlled domains. If dynamic URLs are required, ensure they cannot be modified by the model itself.
  • System Prompt Optimization: To manage costs associated with KV caching, place static instructions at the beginning of the system prompt so that subsequent token additions only require calculating attention for the new tokens rather than recalculating the entire context.
  • Agent Hardening (Prompt): System prompts must include explicit guidance detailing allowed actions and prohibited queries (e.g., "I cannot perform arbitrary file deletion"). This raises the noise level of malicious inputs, aiding guardrail detection.
  • Agent Hardening (Execution): Implement guardrails that monitor tool outputs, not just user input, to catch attacks where the model is tricked into performing unauthorized actions via its tools.
  • Sandboxing: While using sandboxes or dev containers for autonomous agents (like Nano Claw) provides environmental isolation, developers must recognize that any files passed into the sandbox can contain sensitive data and thus remain a threat surface.
