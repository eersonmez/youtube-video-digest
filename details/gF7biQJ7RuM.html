Five essential concepts for developing mission-critical software systems.

NDC Conferences · 2026-06-24T13:21:07+00:00 · Five things every developer should know about building mission-critical systems - Loek Duys

What This Video Is About

This video details five critical concepts necessary for developing software systems that must operate reliably under demanding, mission-critical conditions, using a bank payment API as the running example. The core challenge addressed is ensuring system robustness when dealing with failures in downstream services, network issues, or high load. A simple payment flow between a mobile app and a core banking system can become fragile; this presentation outlines architectural patterns to transform such a basic setup into a highly resilient, fault-tolerant system capable of handling real-world operational stresses without failing catastrophically or losing data.

The concepts covered move from ensuring individual operations are safe (idempotency) to managing service dependencies gracefully (circuit breakers), decoupling services entirely (outbox pattern), gaining deep insight into distributed behavior (observability), and finally, automating the validation of these complex systems under stress (automated load testing). The goal is to prevent unexpected failures that might otherwise require emergency intervention.

Detailed Walkthrough

The presentation begins by establishing a simple system: a mobile device interacts with a payment API, which in turn communicates with a core banking system holding accounts and balances [0:00]. When the core bank fails (e.g., returns an error), the initial instinct is to implement retries on the payment API side [2:15]. However, this simple retry mechanism only works for transient errors—those that resolve themselves over time, like a temporary deadlock or overload [3:45].

A major flaw in basic retries emerges when the core bank successfully processes the command but the response is lost due to network issues. In this scenario, the payment API might retry and cause a double booking [5:00]. The solution introduced here is idempotent services. By adding a unique identifier, called an idempotency key, to every incoming command, the core bank can recognize if it has already processed that specific request, ensuring commands are executed exactly once even upon retries [6:30].

When the downstream service (core bank) is completely offline, continuous retrying by the payment API leads to user frustration as they wait indefinitely for a timeout [8:15]. The circuit breaker pattern addresses this. When integrated into the payment API's HTTP request flow, it monitors failures. If issues persist, the circuit moves from a closed state (requests pass through) to an open state (requests are immediately blocked and fail fast), providing early feedback to the user [9:30]. After a waiting period, it enters a half-open state where a few test requests are allowed through; if they fail, the circuit opens again. This mechanism enables auto-healing by allowing the system to recover once the downstream service is back online [12:45].

The next step tackles tight coupling introduced by the circuit breaker and retries. The outbox pattern decouples services by preventing immediate calls to the core bank. Instead, when a command arrives at the payment API, it performs local state changes (e.g., updating local balance) and saves the message intended for the core bank into an "outbox" table within its own database as part of a single transaction [13:45]. The API immediately returns a 202 Accepted status to the caller, providing immediate feedback while background processors handle downstream communication. This ensures outbound messages are never lost due to crashes or downstream unavailability [16:00].

With asynchronous processing introduced by the outbox pattern, understanding system state becomes difficult; logs alone may not reveal silent failures (e.g., if the outbox processor crashes but the API still accepts requests) [17:30]. To solve this, observability is required through a combination of end-to-end tracing and metrics. End-to-end tracing allows developers to follow a single request from the mobile device, through the payment API, into the outbox processor, and eventually to the core bank [19:30]. Tools like OpenTelemetry are recommended for capturing this telemetry across service boundaries by ensuring context (like trace ID) is preserved when moving data between services or background jobs [21:45].

The complexity of managing multiple concurrent outbox processors introduces new challenges, such as race conditions where two processors might handle the same record, or ordering issues if a dependent operation runs before its prerequisite [23:00]. Solutions include leader election, where processors compete for a lock to designate one leader responsible for processing, and partitioning, which divides operations by entity (e.g., account ID) so that concurrent operations on different accounts do not interfere with each other [24:15].

Finally, when the core bank recovers after an outage, the backlog of messages from the outbox can instantly overload it again. To prevent this "thundering herd" problem, a corresponding inbox pattern is implemented within the core bank API. Instead of processing commands immediately upon receipt from the outbox processor, they are durably placed into an inbox queue [25:00]. An inbox processor then handles these commands sequentially at its own pace. When combined with idempotency keys, this guarantees both message delivery and exactly-once command execution even during recovery surges [31:00].

The final concept involves automated resilience testing. Instead of waiting for production failures, developers can simulate chaos locally using a tailored proxy server that inject faults (e.g., errors) into requests [35:00]. For load and stress testing, tools like K6 are used across multiple service instances. To manage this complexity, orchestration tools like Aspire can automate the setup of all dependencies (API, proxy, databases) locally [38:15]. Furthermore, a custom "load test support API" can be built to poll internal states—checking if inboxes and outboxes are empty—to confirm that transactions have fully settled before asserting correctness or analyzing performance metrics from tracing tools like Jaeger [40:30].

Key Takeaways

  • Idempotency: When implementing retries for downstream calls, the receiving service must be idempotent. This means processing the same request multiple times yields the same result as processing it once, typically achieved by using a unique idempotency key sent with the command [6:30].
  • Circuit Breakers: Use circuit breakers to prevent cascading failures when a downstream service is unavailable. They provide early feedback (fail fast) instead of forcing clients to wait for timeouts during an outage [9:30].
  • Outbox Pattern: To decouple services and guarantee message delivery, write intended outbound messages into a local "outbox" table within the same transaction as local state changes. This allows immediate response to the caller while background jobs handle downstream communication asynchronously [14:00].
  • Observability Stack: Comprehensive system health requires combining logs with end-to-end tracing (e.g., using OpenTelemetry) and metrics (e.g., outbox depth). Tracing is crucial for finding silent failures in distributed systems across service boundaries [21:45].
  • Inbox Pattern: To manage surges when a previously failed downstream service recovers, implement an "inbox" pattern on the receiving end. This queues incoming commands so they can be processed sequentially by a background worker rather than overwhelming the system immediately [25:00].
  • Scalable Background Processing: For multiple outbox processors, use leader election to ensure only one designated processor handles tasks at a time, and employ partitioning (e.g., by account ID) to allow safe concurrent processing across different data subsets [24:15].
  • Automated Resilience Testing: Use specialized tools like proxy servers for fault injection during development. Combine load testing with custom support APIs that verify internal system states (like empty inboxes/outboxes) before declaring a test complete [38:15].

Practical Notes

  • Idempotency Implementation: Commands must carry a unique identifier, such as a transaction ID, which the receiving service uses to check if the operation has already been successfully recorded before executing business logic [6:30].
  • Circuit Breaker Flow: The request flow through a payment API should be structured as: HTTP Request -> Circuit Breaker Policy -> Retry Policy -> Downstream Service Call. If retries fail, the circuit breaker transitions to an open state [10:00].
  • Outbox Workflow: 1) Receive command. 2) Update local state AND insert message into outbox in one transaction. 3) Return 202 Accepted immediately. 4) Background processor reads from outbox and attempts delivery. 5) If the core bank is down, the background job retries indefinitely without blocking the user [16:00].
  • Telemetry Context: When moving data between services (e.g., API to Outbox Processor), ensure the telemetry context, specifically the trace ID, is captured and rehydrated in the subsequent process to maintain an end-to-end view [21:45].
  • Load Testing Orchestration: Tools like Aspire can orchestrate local development environments by defining dependencies (API, proxy, DB) in a configuration file, allowing for automated setup of complex testing scenarios [38:15].
  • Data Consistency Check: When running load tests, assertions should verify not only performance metrics but also data correctness, such as ensuring that the net balance change after a series of transactions is zero [45:00].
