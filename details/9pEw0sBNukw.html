Scott Hanselman talks with Zed founder Nathan Sobo about Zed, editor design, and DeltaDB as a new layer for collaborative coding history.

Scott Hanselman · 2026-06-23T07:30:27+00:00 · The space between the Commits with Zed and DeltaDB's Nathan Sobo

What This Video Is About

Scott Hanselman interviews Nathan Sobo, founder of Zed and creator of Atom, about the design philosophy behind Zed and the emerging idea behind DeltaDB: a version-control-like layer that records not just committed snapshots of code, but the process of programming between those commits. The conversation starts with editor history and developer-tool monocultures, then moves into why Sobo believes the next frontier is not merely a faster editor, but a richer model of software creation itself.

The core argument is that Git is excellent at versioning checkpoints: commits, snapshots, branches, and a directed acyclic graph of project history. But modern software development increasingly happens through ongoing conversations with AI agents, experiments across work trees, abandoned edits, partial refactors, prompts, corrections, and contextual decisions that never make it into commit history. DeltaDB is presented as a Git-compatible superset that captures this “white space” or “dark matter” between commits.

This matters because agent-assisted coding changes what “source history” means. If a developer asks an agent to implement something, corrects assumptions, steers architecture, accepts some edits, rejects others, and then commits only the final result, teammates see the artifact but not the reasoning path. DeltaDB aims to make that process available as a collaborative artifact: something teammates and agents can inspect, join, resume, summarize, or query later.

The discussion also addresses tradeoffs. Capturing everything can become noisy or invasive, and Sobo explicitly acknowledges privacy and workplace-surveillance concerns. DeltaDB’s early internal version shares entire agent conversations, but the intended direction is finer-grained sharing controls, such as sharing from a particular moment, summarizing parts of a conversation, or hiding early messy reasoning while preserving useful context.

Detailed Walkthrough

The conversation opens with the central thesis: Git versions “checkpoints of the artifacts,” while DeltaDB wants to version “the process between those checkpoints” [0:00]. Hanselman describes this missing layer as abandoned edits, prompts, agent steering, and small decisions that happen between two commits. Sobo agrees, but expands the idea beyond mere archival. DeltaDB is not only meant to coordinate this information, but to make it available to multiple participants across different time scales: real-time collaboration, later review, and asynchronous continuation [0:22].

Hanselman then introduces Nathan Sobo as the founder of Zed and says Zed is working on DeltaDB [0:49]. The early portion of the interview frames Sobo as someone willing to challenge technology monocultures. Hanselman compares today’s browser ecosystem, where Chromium dominates, with past eras when many browsers competed [1:23]. He then draws a parallel to code editors: there used to be many popular editors, but Visual Studio Code became dominant after Atom, Electron, and Microsoft’s editor work converged [2:27].

Sobo recounts Atom’s origins at GitHub. Atom was created by a small team and was originally tied to GitHub cofounder Chris Wanstrath’s interest in building a better editor [2:56]. In the process of building Atom, the team also created Electron: a shell around Atom that made it possible to build desktop apps using browser technology [3:35]. Sobo describes Electron as “dressing up a web browser as if it’s a desktop app” [3:39]. Microsoft then adopted Electron for Visual Studio Code after developing browser-based editor technology such as Monaco [3:55].

Sobo says VS Code took over through feedback loops: extensions went there because users were there, and users went there because extensions were there [4:32]. But he personally did not enjoy using VS Code all day. He had written Atom in Vim, wrote Zed in VS Code because of tooling integration, but ultimately felt that using a “web browser dressed up as a desktop app” for long coding sessions was not good enough for him [4:57]. This dissatisfaction led to Zed.

The Zed philosophy is described as a willingness to rebuild fundamental layers rather than accept inherited limitations. Sobo says Zed required reinventing the stack they had previously built on with Electron [6:33]. He criticizes browser-based desktop apps for RAM use, lowest-common-denominator abstractions, legacy APIs, and insufficient control for building “something truly exceptional” [6:49]. His motivation was not merely to compete with VS Code, but to build a tool he personally would be much happier using [6:09].

Hanselman asks about building products for oneself versus designing by consensus [8:16]. Sobo clarifies that Zed is not exclusively for him; once he built a team, took investment, and gained users, he acquired obligations to teammates, investors, and the community [8:50]. But he sees himself as “user zero” [9:16]. His product process starts with building something he genuinely wants and holding it to a high standard. He says past experience taught him that when he builds something he truly believes is good, other people often like it too [9:50].

The interview then pivots from editors to source control. Hanselman notes that older ecosystems had many systems: Source Depot, SourceSafe, CVS, SVN, Mercurial, and Git. Now Git is taught as the default [10:45]. He frames DeltaDB as Zed’s attempt to question another settled foundation: version control. Sobo says the DeltaDB challenge is different from competing with VS Code because Git is modular and easy to build on top of [11:30]. Unlike Electron, where Sobo felt the team had to start over, Git already solves many problems well.

Sobo praises Git’s core model: taking snapshots of code, chaining them together in a causal graph, storing blobs in pack files, and transferring bulk data efficiently [12:15]. He does not present DeltaDB as a total replacement for Git. Instead, the problem he has long cared about is that software collaboration happens too far downstream, around snapshots and artifacts left behind by coding, rather than around the actual moment of writing code [12:42].

This concern dates back to Atom. Sobo says when he pitched GitHub on building a code editor, part of the vision was to move software interaction closer to the act of writing code itself [12:49]. He came from a pair-programming background, and Zed’s team already uses collaboration features to talk about software while writing it [13:21]. The current DeltaDB work extends that idea: represent the fine-grained versions of code that exist between commits, refer to them stably, and turn the code itself into the canvas for collaboration [13:53].

Hanselman summarizes DeltaDB as versioning the “white space” between commits: abandoned edits, prompts, agent steering, and decisions that do not appear in Git history [14:50]. Sobo agrees and emphasizes that DeltaDB’s purpose is to make those interactions available to multiple participants, whether synchronously or asynchronously [15:13].

The agent workflow is central. Sobo describes the current status quo as a developer having a one-on-one direct-message-like session with an AI agent [15:20]. The byproduct of that private exchange is a set of code snapshots that the user commits and pushes to GitHub [15:32]. The conversation that produced those snapshots is usually absent, or perhaps attached later to a pull request by other tooling [15:40]. DeltaDB instead versions the conversation along with edits in embedded work trees, continuously [15:57].

This enables multiplayer interaction around agent sessions. A developer might work with an agent, generating code and context; those work trees and the conversation can then be stored so that a colleague in another time zone can join three hours later, have the work trees mirror to their own disk, and continue from the same thread [16:27]. The colleague could even ask the agent what the original developer was discussing [16:42]. Sobo contrasts this with pull request bodies, which are not designed to preserve or interact with the live conversation that produced the code [16:50].

Hanselman compares this with existing agent tools that have their own session concepts: GitHub Copilot, Codex, Claude, markdown files, or tools that store conversation history in a SQLite database [17:18]. Sobo’s point is that the conversation itself is a valuable artifact, filled with context and intermediate activity that people may want to join [17:32]. In DeltaDB, if a participant is invited into a conversation, they can access the exact state of the code in that conversation down to the millisecond [17:55]. Sobo states the design shift clearly: in Git, the repository is collaborative but the work tree is personal; DeltaDB makes work trees collaborative artifacts [18:12].

Hanselman gives his own example of working with OpenClaw on Windows and introducing teammates to work trees [18:34]. Work trees make it possible to parallelize work, but they remain local to his machine. Teammates see the before-and-after code, but not the agent conversation, assumptions, corrections, or “conversational stack tracing” that shaped the result [18:51]. He asks whether recording all this becomes useful context or noisy slop [19:15].

Sobo responds that Zed is conscious of developer privacy and that storing every edit can sound uncomfortable [19:33]. The purpose of storing fine-grained data is to index it, not necessarily to force users to replay everything from beginning to end [19:41]. A replay feature could be built, but that would often be noisy [19:50]. In practice, when Sobo returns to an agent conversation, he does not read the entire transcript from the top; he asks the agent to catch him up [20:18]. The value is that the data is available for reference, and the conversation becomes a multiplayer entity that humans and agents can navigate [20:34].

Hanselman then reframes DeltaDB as an “event sourced database for the act of programming,” emphasizing that programming is social and multiplayer for many teams [21:05]. Sobo says DeltaDB stores a directed acyclic graph, like Git, but records and indexes every single thing that happened [21:36]. This index enables new capabilities. DeltaDB can understand every edit at the delta level, associate code with the conversation that created or referenced it, and locate a character from one version in later versions without relying on heuristic diffing [21:45].

Sobo explains that because DeltaDB has a causal understanding of everything that occurred after a point, it can locate a character in any subsequent version “losslessly” [22:14]. If code was deleted, DeltaDB can determine where it would be; if conversations referenced that code, the system can find those references [22:36]. This could answer questions such as: in this file over the last 30 days, which 300 agent conversations referenced this particular code? [22:44]. Sobo acknowledges that the volume of data may be too much for a human interface and may be most useful to an agent querying other context windows [22:59].

Hanselman asks what exactly counts as a delta: one keystroke, a 500-line patch, formatter changes, or something else [23:23]. Sobo answers that if someone is editing manually, one keystroke is a delta [23:45]. If someone pastes 10,000 characters, that entire paste is also one delta [23:54]. DeltaDB therefore tracks both tiny and large units of change as deltas, depending on the editing operation.

The next issue is semantic meaning. Hanselman points out that developers usually do not care that characters 400 through 526 were replaced; they care that, for example, “the authentication check was moved below the cache lookup” [24:08]. Sobo says the value of delta-level tracking is that annotations and references can anchor to specific characters or inserted text [24:29]. A character inserted by a delta has a unique identity: it came from a particular insertion and offset [24:46]. DeltaDB stores enough metadata to find the equivalent logical position in later versions [25:01].

When Hanselman asks how DeltaDB decides which moments are semantically meaningful, Sobo says Git commits still exist because the Git commit DAG is a subset of the DeltaDB causal DAG [25:31]. DeltaDB also plans to introduce “moments” or “waypoints” in conversations with agents: lightweight markers that name a fine-grained version in time without creating a full snapshot [25:44]. Hanselman clarifies for listeners that DAG means directed acyclic graph [26:05].

One feature Sobo names is “portals” [26:31]. In an agent transcript, whenever the agent references code, DeltaDB can let the user step through that reference to the exact code as it existed at that moment [26:31]. Unlike Git, this does not require the system to have a commit for every second. DeltaDB models the underlying history in terms of deltas [26:53]. Once a user jumps to a point in past code, DeltaDB can provide a timeline that keeps the cursor at the same logical position while scrubbing through time [27:08]. If the code was moved to another file, the cursor would move with it [27:24].

Hanselman identifies this as something structurally difficult or impossible with Git alone: going back in time and following a semantic chunk of code as it moves from one location or module to another [27:35]. Sobo says Git can sometimes reconstruct what happened from snapshots, but it does not feel fluid. By continuously tracking and indexing every change, DeltaDB can answer such questions more directly [28:08].

Sobo admits DeltaDB is hard to explain because it is a foundational abstraction that could enable many features [28:15]. Hanselman compares this to Git’s early reception: initially confusing, later obvious [28:44]. He also mentions conflict-free replicated data types as another idea that can seem difficult at first but later feel inevitable [28:57].

Sobo emphasizes that Git users should not need to think much about DeltaDB [29:12]. Its capabilities are meant to be transparently superimposed on top of Git. DeltaDB will watch the work tree on disk, so even tools outside Zed that edit files can be picked up [29:28]. Sobo says it can conflict-free diff to someone else’s file system, even if both people are outside the tool [29:36]. Users can still commit, push, and use Git-based workflows while receiving DeltaDB features “for free” [29:53]. Hanselman summarizes it as Git-compatible, “Git plus plus,” and Sobo agrees that DeltaDB is a superset of Git, tracking commands and everything in between [30:09].

The final major section concerns privacy and control. Hanselman asks who gets to see the internal reasoning trail: the author, the team, or the company [30:22]. He raises concrete risks, such as accidentally pasting keys or connection strings, and warns about the thin line between code provenance and workplace surveillance [30:31]. Sobo agrees, saying he has the same insecurity about his early agent conversations: he may type “I have no idea what I’m doing” and not want that panic preserved as shared code history [30:51].

In the current pre-alpha internal version, Sobo says the model is simple: share the entire conversation with the agent [31:16]. This works internally, but it is not the final desired control model. The plan is to allow constraints, such as sharing from a particular moment in the conversation [31:31], hiding some of the paper trail, or exposing a compacted/summarized point instead [31:39]. Sobo says the goal is to capture everything securely and privately, then give users fine-grained controls over what gets unveiled to peers [31:54]. He explicitly says he does not want to create workplace surveillance [32:12].

Hanselman suggests a useful model: some parts of history could be verbose, while other parts could be summarized [32:20]. He notes that summarization is one of the cheaper and less invasive uses of language models, imagining a “TL;DR” of the last two hours before committing [32:29]. Sobo acknowledges that skeptics may ask whether that is just a commit, but argues that modeling the process continuously and making meaningful moments lightweight could produce a more fluid experience than always requiring discrete commits and summaries [32:43].

The conversation ends with the access path for DeltaDB. Hanselman says people can go to zed.dev/deltadb, enter an email and GitHub username, and request early access [33:22]. He summarizes the promise as being able to rewind to any edit, branch to any moment, and trace code back to conversations [33:30].

Key Takeaways

  • DeltaDB is positioned as a Git-compatible superset, not a wholesale Git replacement. Sobo repeatedly says Git already solves important problems well: snapshots, commit graphs, efficient storage, and transfer. DeltaDB builds on top of Git by tracking what happens between commits.
  • The core missing artifact is the process of programming. Git preserves committed snapshots, but it usually loses abandoned edits, prompt history, agent steering, assumptions, corrections, and intermediate decisions. DeltaDB treats that “between commits” history as valuable.
  • Agent conversations are becoming part of source history. In current workflows, a developer may have a private session with an AI agent and only push the resulting code. DeltaDB aims to version the conversation and the code changes together so teammates can inspect, join, resume, or query that process.
  • Work trees become collaborative artifacts. In Git, a repository is shared but a work tree is typically local and personal. DeltaDB’s model allows an invited collaborator to receive the exact state of a work tree tied to a conversation, even asynchronously.
  • DeltaDB records fine-grained deltas. A manual keystroke can be a delta; a pasted 10,000-character block can also be one delta. The point is not necessarily human replay, but indexing the evolution of code precisely.
  • The system can follow code through time more precisely than Git diffs. Because DeltaDB records causal changes continuously, it can locate a character or code region across later versions, even if moved, deleted, or pasted elsewhere. Sobo describes this as lossless compared with “squinting through a diff.”
  • “Portals” connect agent references to exact historical code states. When an agent references code in a conversation, DeltaDB can let the user jump to the exact code as it existed when that reference was made.
  • Waypoints or moments may supplement commits. DeltaDB plans lightweight markers in agent conversations that name fine-grained points in time without requiring full Git commits for every meaningful step.
  • The UI challenge is deciding how much history humans should see. DeltaDB may capture and index enormous amounts of context. Sobo suggests much of it may be more useful to agents than directly to humans, with humans asking agents to summarize or orient them.
  • Privacy and surveillance are acknowledged as serious risks. Sobo says early messy conversations, accidental secrets, and personal uncertainty should not automatically become shared workplace history. The intended design includes fine-grained sharing, sharing from a moment, compaction, and summarization.
  • Zed’s broader philosophy is to challenge settled foundations. The conversation links DeltaDB to Zed’s editor strategy: Sobo was dissatisfied with Electron-based editor limitations and wanted to rebuild foundational layers rather than accept dominant tooling as finished.
  • Early access is available through Zed’s site. The transcript states that users can request access at zed.dev/deltadb by providing an email and GitHub username.

Practical Notes

DeltaDB is described as something developers should be able to use while continuing normal Git workflows. Sobo says users can still commit and push, and that Git’s commit DAG remains a subset of the DeltaDB causal DAG. In practical terms, DeltaDB is presented as “Git plus plus”: it tracks Git commands and also tracks what happens between them.

The intended architecture includes continuous tracking of work tree changes on disk. Sobo says DeltaDB will watch the work tree, meaning edits made by external tools can still be picked up. This matters in agent workflows because not all edits may come from inside Zed; external agents, formatters, scripts, or other tools may modify files.

DeltaDB’s collaboration model is based around conversations, embedded work trees, and shared state. A developer can have an agent conversation tied to code changes, store that in the cloud, and allow a colleague to join later. The colleague’s disk can receive the associated work trees, allowing them to continue from the same state rather than reconstructing it from a branch, pull request, or patch.

The product introduces or plans several concepts:

  • Deltas: fine-grained edits such as keystrokes, pastes, or larger inserted units.
  • Causal DAG: a directed acyclic graph that includes Git commits but also finer-grained change history.
  • Work tree collaboration: turning local work trees into shareable artifacts.
  • Portals: links from an agent conversation into the exact historical code referenced at that moment.
  • Waypoints or moments: lightweight markers in time, less formal than full commits.
  • Indexing: the mechanism that makes captured history queryable rather than merely replayable.

The key implementation tradeoff is storage and privacy versus utility. Sobo’s argument is that capturing everything enables indexing, tracing, and later summarization. But he also agrees that raw capture can be noisy and potentially invasive. The transcript does not provide storage formats, database internals, command-line syntax, pricing, or installation instructions.

There are no specific commands shown in the transcript. The only concrete access instruction is to visit:

  • `zed.dev/deltadb`

The transcript says early access requires an email and GitHub username.

Risks and caveats supported by the transcript include:

  • Capturing every edit can be noisy if treated as a replay log.
  • Agent conversations may include embarrassing, uncertain, or private reasoning.
  • Developers may accidentally paste secrets such as keys or connection strings.
  • Without careful controls, the system could feel like workplace surveillance.
  • UI design is still being figured out, especially how much conversation and historical reference data to surface to humans.
  • The internal version at the time of discussion is described as pre-alpha and currently shares entire conversations, with finer-grained sharing planned but not finalized.

A practical workflow implied by the discussion would look like this: a developer works in Zed or another tool while DeltaDB watches the work tree; the developer interacts with an agent; edits and conversation history are continuously versioned; the developer can still create normal Git commits; a teammate can later join the conversation, receive the relevant work tree state, ask the agent for a summary, inspect code through portals, and continue work from that point.
