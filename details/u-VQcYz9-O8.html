Strategies for building resilient frontend components within a design system.

NDC Conferences · 2026-06-23T10:11:18+00:00 · Building Better Components - A Journey to a More Resilient Frontend - Daniel Yuschick

What This Video Is About

This video outlines strategies for constructing frontend components within a design system that are resilient to constant content and product evolution. The central premise is that component libraries alone are insufficient; true resilience comes from establishing robust patterns, conventions, and cross-team trust between design and development teams. Since content is perpetually changing, the goal is to build frontends that accommodate this change without accumulating significant technical debt or breaking existing functionality.

The discussion focuses heavily on leveraging TypeScript interfaces as a primary mechanism for achieving component stability. By carefully structuring how props are defined—moving away from sprawling boolean flags toward categorized union types and adhering strictly to established web standards—developers can create components that scale gracefully alongside evolving product requirements. Ultimately, the speaker argues that a design system is fundamentally an exercise in building organizational trust through predictable patterns.

Detailed Walkthrough

The presentation begins by framing the challenge: how to componentize frontends effectively when content is always changing [2:37]. The speaker shares experience leading design systems at OP (Osuuspankki) and Posti, noting that while components are the visible output, the underlying system architecture is crucial for longevity.

The core technical discussion revolves around managing TypeScript interfaces to prevent them from becoming bloated due to evolving needs like new themes or sizes [5:31]. A common anti-pattern identified is relying on numerous boolean props (e.g., separate flags for high contrast mode or extra-large size) [6:41]. This leads to complex defensive logic within the component and makes extensions difficult. The solution proposed is grouping related properties into categories using string literal union types [7:59]. For instance, instead of multiple booleans controlling label rendering, theme, and size, a single `labelType`, `theme`, and `size` prop can be used, allowing new options (like "high contrast" or "extra small") to be added simply by extending the union without breaking existing code [8:36].

Another pattern discussed is avoiding the "data gate model," where a boolean flag like `hasError` controls the visibility of an error message string. The recommendation is to simplify this by using a single prop, such as `errorMessage`, and assuming that if the prop is provided, the component should render in its appropriate state (e.g., toggling `aria-invalid`) [12:00].

A critical concept introduced is establishing a universal terminology system across the entire design system [14:26]. This means defining shared properties—like `disabled`, `busy`, or size definitions using two-letter T-shirt values (e.g., 's', 'm', 'l')—so that if a developer learns how to use one component, they intuitively know how to use all others [14:31]. This convention must be enforced across interactive controls and static media alike [17:28].

The final major technical point is the directive to "do not reinvent HTML" [22:20]. When building components that abstract native HTML elements (like `<button>` or `<input>`), developers should separate standard HTML attributes from custom component props. Deviating from established HTML conventions creates friction because application teams must learn your component's specific rules instead of relying on universal web standards [23:16]. For instance, a button should be treated as a semantic interactive container capable of holding various children; scoping it only to a single `text` prop breaks this convention and limits its adaptability [32:57].

Finally, for complex interactions like popover menus or dialogues, the video suggests using centralized component stores (e.g., via custom hooks in React) [38:02]. This store manages state and behavior across different files—such as keeping a trigger button in one file synchronized with the menu logic in another—by providing a single reference point for actions like `open` and `close` [43:09].

Key Takeaways

  • Component libraries are only part of the solution; building a design system is fundamentally about establishing trust between organizational teams [1:26, 47:07].
  • Group similar component properties into categories using string literal union types instead of relying on numerous boolean flags to prevent interface bloat [8:02].
  • Establish a consistent terminology system that governs shared props (like `size` or `variant`) across all components in the design system, ensuring predictable behavior regardless of the specific element [16:43].
  • Do not reinvent native HTML conventions; when building abstractions over standard elements like `<input>`, separate inherent HTML attributes from component-specific properties to maintain compatibility and reduce learning curves [27:50].
  • Simplify data flow by removing "gate props" (e.g., `hasError`) in favor of single, comprehensive props that imply the desired state when present (e.g., `errorMessage`) [13:05].
  • Use centralized component stores or hooks to manage complex interactive states (like popover open/close status) across multiple files and disparate UI elements, keeping them synchronized [38:40].

Practical Notes

  • Prop Structuring: When designing props for components like inputs or buttons, use union types for constrained values. Example: Instead of `isLarge`, `isExtraSmall`, define a single `size` prop that accepts `"small" | "medium" | "large" | "extra-small"` [8:12].
  • HTML Abstraction: When building components wrapping native elements, use TypeScript's ability to extend framework types (e.g., extending HTML input attributes) so your component interface inherits all standard browser attributes automatically [28:03]. This keeps your custom props clean and focused only on unique behavior.
  • State Management for Interactions: For components spanning multiple files (like a trigger button and its associated menu), implement a centralized hook/store (e.g., `usePopoverMenu`) that holds the state (`isOpen`, `ref` to trigger) and exposes methods (`open()`, `close()`) to both elements [39:06].
  • Collaboration: Align on conventions (like default sizes or variants) between design and development teams early in the process; this alignment is crucial for UI consistency across large applications [18:24, 20:13].
