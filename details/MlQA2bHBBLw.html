Daniel Nikpayuk's lessons from implementing a functional type system and metacompiler in constexpr C++.

NDC Conferences · 2026-07-22T10:00:18+00:00 · Lessons learned while coding my first functional type system in constexpr C++ - Daniel Nikpayuk

What This Video Is About

Daniel Nikpayuk presents a fast survey of roughly a decade spent experimenting with functional type systems, template metaprogramming, `constexpr` C++, continuation-passing-style composition, and compile-time compiler construction. He explicitly frames the talk as a survey rather than a complete tutorial: the material is dense, and he expects that some details will be easier to revisit or discuss afterward.

The technical project is a functional type system built on top of the untyped lambda calculus and implemented in compile-time C++. Nikpayuk’s system is not limited to recording type information. It is intended to record values as well, serialize its internal state, represent function values as assembly-like bytecode, and restrict bytecode memory use to the stack rather than the heap. These constraints shape the system’s data structures and execution model.

A major purpose of the talk is to compare two ways of making C++ perform computation during compilation. Template metaprogramming repurposes C++ grammar and compiler behavior to execute side programs, while `constexpr` programming uses ordinary C++ code in compile-time contexts. The limitations of both approaches motivate Nikpayuk’s use of continuation-passing-style composition, compile-time controllers, variadic register packs, flat-array records, and serialized metadata.

The project also serves a longer-term goal. Nikpayuk identifies himself as a self-taught C++ programmer from Canada’s western Arctic and says he wants to build a high-performance multimedia programming language that can support telling and retelling his people’s stories. He treats design, including cultural design, as a study of ways of thinking that can persist even when technologies and environments change.

Detailed Walkthrough

The speaker’s motivation and the scope of the talk

At [0:42], Nikpayuk divides the talk into six parts: his background, an explanation of the talk’s scope, template metaprogramming, `constexpr` programming, his functional type system, and lessons learned.

He has programmed in C++ since 2005, although he says he does not currently work in the technology industry. His academic background is mathematics and economics. He explains that C++ is useful for his long-term project because a multimedia production language will generally need system-level performance, making C++ a practical choice for writing its first compiler [0:59-1:48].

The technical subject is introduced as the construction of a functional type system in `constexpr` C++. Before discussing his implementation, Nikpayuk establishes what he means by a functional type system.

Functional type systems as records of types and values

For the purposes of the talk, a functional type system is one built on top of the untyped lambda calculus [2:00]. Its atomic types include booleans, characters, and integers. Its compound types include products, co-products, and lists. Nikpayuk notes that the untyped lambda calculus can be paired with several alternative theoretical type systems, whose relationships are represented by the lambda cube.

In practical programming, he says, these systems are often built on list-oriented languages because list-style programming resonates with the lambda calculus [2:46]. A type system should be understood not merely as a catalog of types but as a recordkeeping system that stores types and values belonging to those types [2:54]. Real implementations add constraints that are absent from the abstract theory.

Nikpayuk’s own system extends the basic types with comparison types, including an identity type, and logical types such as dependent products and dependent pairs [3:13]. It also imposes three implementation constraints:

  • The entire type system must be serializable, so it can be transmitted between technological media, saved, and reloaded.
  • Function values must have an internal representation resembling assembly-program bytecode.
  • The bytecode instructions may request memory only from the stack; they may not allocate from the heap.

He says these restrictions are consequences of the continuation-passing architecture he developed, rather than arbitrary limitations imposed before the design began.

Compile-time programming as a separate time scope

Before focusing on `constexpr` C++, Nikpayuk defines compile-time programming more broadly [4:20]. Once a language has types, a program can be divided into information that is invariant and completely known at compile time, and information that can only be known during runtime execution.

For this talk, a computation is compile-time programming when it takes runtime-invariant data as input and returns runtime-invariant data as output [5:14]. Static types are the most obvious example. Modern C++ extends this idea by allowing compile-time manipulation of value data as well.

The examples he mentions include a compile-time integer initialized to 5 and a `static constexpr` unsigned array with a fixed length [5:42]. The broader point is that C++ can evaluate ordinary value-level computations before the program runs, not just resolve types.

Cultural design and persistent ways of thinking

The talk then takes a short detour into cultural design [6:10]. Nikpayuk introduces an Inuktitut phrase translated in the transcript as “weather is thought” and discusses the connection between weather and thought in classical Inuit cosmology.

He describes Arctic weather as changing rapidly: clear skies can become completely covered by fast-moving clouds within minutes. He compares this changing weather with human thought. Weather is connected to air, and air is connected to breath. Since breath passes through the body and is used to speak, he describes speech as carrying thought and will into the world. This is why, in the cultural context he describes, words are treated carefully and as having real power [7:29-8:22].

He extends the idea into design. Arctic ice and snow are forms of water and are malleable, so the material environment offers relatively little permanence. The things with lasting meaning are therefore patterns of thought that persist. Nikpayuk calls these enduring ways of thinking “technologies of mind” and presents the talk as an exploration of design principles that may persist even as particular tools change [8:27-9:37].

Template metaprogramming as a compiler-side programming language

Nikpayuk characterizes template metaprogramming as a paradigm shift from ordinary C++ [9:44]. In a conventional programming language, programmers describe types, values, and functions. Additional grammar generally exists to organize those elements or to validate their use. He therefore treats functions as the central focus of a programming language [10:08-11:02].

A compiler is itself a program. It receives character strings as input, branches and loops while translating them, and produces another representation. Template metaprogramming exploits this fact by using C++ language features in unintended ways to redirect compiler behavior. The result is a side program that executes during the main program translation [11:02-11:51].

The example at [11:51] uses a compile-time conditional type selection. A dispatch value determines whether a type alias becomes `double` or `int`, after which a value is defined using the selected type. The important conceptual step is to treat parts of C++ grammar as if they were functions that accept types and values as compile-time arguments.

This creates a higher-order programming language layered on top of standard C++, one that evaluates during compilation rather than during runtime [12:22-12:56]. Template metaprogramming can improve performance by resolving information early and can dispatch type traits and attributes at compile time. Its costs are architectural and ergonomic: the compiler does not understand these grammar elements as an integrated metaprogramming system, so programmers inherit more responsibility for type checking, and the resulting syntax can become complicated.

As of C++17, Nikpayuk treats variadic templates, alias templates, and class templates as three kinds of metafunctions. Each has different input, output, calling, and specialization behavior.

The three forms of C++ metafunction

Variadic metafunctions most closely resemble ordinary functions. They can accept a mixture of parameters and return a value. Nikpayuk recommends `auto` when a value is needed as input. His square example accepts a value such as 2.0 using template-style call syntax rather than ordinary parentheses [14:15-14:42].

This form is weakly typed if it uses only `auto`: the parameter is constrained to be a value, but not to a more specific type. In C++20, a concept can strengthen it. His example applies an integral concept to the square metafunction, causing an invocation with a floating-point value to produce an error [14:42-15:24].

Nikpayuk also shows that specialization can express pattern matching familiar from languages such as Haskell. A factorial definition has a base case mapping 0 to 1 and a general case that recursively handles the remaining input. In C++, the same structure is expressed with specialization [15:32-16:22].

Alias templates are type-producing metafunctions. Unlike value-oriented metafunctions, their output is restricted to types. The example constructs an integer array type with a compile-time length, conceptually producing something like an integer `std::array` whose size is supplied as a template argument [16:30-16:52]. Alias templates do not support pattern matching through specialization in the way class templates do.

Class templates always construct classes. This distinguishes them from aliases, which can return an already existing type. Class metafunctions can also be specialized, so they support pattern matching. The example defines a general class metafunction taking a type and then provides a specialized case for a const-qualified type [16:59-17:48].

Higher-order metafunctions, recursion, and currying

Nikpayuk next examines whether these metafunctions have the same traits as ordinary functions. Higher-order functions accept other functions as inputs or return functions as outputs. In template metaprogramming, alias and class metafunctions can be passed as template-template parameters. A higher-order metafunction can receive another metafunction and a type, apply the metafunction to that type, and then perform additional work [17:54-18:59].

He specifically notes two limitations. Variadic/value metafunctions are not supported as inputs to the higher-order metafunction pattern he is describing, and there is no direct support for returning a metafunction from another metafunction. The `apply` pattern is presented as an important higher-order operation and as a way to avoid repeating similar algorithms.

Recursion introduces an important compiler-evaluation problem. Nikpayuk revisits factorial and implements the conditional choice with a ternary operator. Both GCC and Clang report that template instantiation depth has been exceeded [19:27-20:10].

He explains the failure using eager and lazy evaluation. For an expression in which a function receives the results of two other expressions, eager evaluation computes both arguments before calling the function. Lazy evaluation would pass the expressions in a paused form and evaluate them only if their results were needed. In his template-metaprogramming example, the ternary expression is evaluated eagerly. Even when the input reaches the base case of zero, the recursive expression involving factorial of `n - 1` is evaluated. That produces factorial of negative one, which continues recursively until the compiler’s template-instantiation limit stops it [20:16-21:48].

Class metafunction recursion provides another route. Nikpayuk says it is less performant at compile time but historically significant, and it introduces a class template as a compile-time tuple-like pattern. C++ classes can contain member values, types, and functions. Static members belong to the class rather than an individual object and can be accessed with the scope-resolution operator.

His `metapair` example stores two input types as `first` and `second` member types, which can then be accessed through the resulting class type [22:02-23:19]. The recursive factorial version uses a factorial tuple with a member value. The general case defines the recursive value, while a specialization for zero assigns the value 1. The external factorial metafunction then reads the tuple’s member value [23:38-24:10]. He uses `class` and `struct` interchangeably in the talk, noting that their relevant difference here is their default member visibility.

Finally, he introduces currying: applying a multi-argument function to one argument and receiving a partially applied function. Ordinary C++ functions can implement currying in C++20, but template metaprogramming cannot directly return a metafunction from another metafunction. There are many workarounds, but he does not cover them [24:10-24:47].

Why `constexpr` is a different compile-time model

Nikpayuk contrasts template metaprogramming with `constexpr` programming at [25:04]. Template metaprogramming makes the compiler behave like an interpreter as a side job. `constexpr` programming is designed to run ordinary C++ code during compilation when the language and context permit it.

A `constexpr` function is written much like a normal C++ function. His example is a function that adds two to its input. A `constexpr` class begins with a `constexpr` constructor, and member functions can independently be marked `constexpr` [26:44-35:20].

The keyword does not guarantee compile-time execution. Nikpayuk says the compiler uses compile-time evaluation when it knows it can; otherwise, the code is treated as ordinary C++ and compiled for normal execution. The exact capabilities also depend on the C++ specification version being used.

One major limitation concerns function arguments. From inside a function, an argument does not reveal whether the caller supplied a compile-time value or a runtime value. The function therefore treats the argument as runtime data. Consequently, a function argument cannot simply be passed as a template parameter.

His example defines a `square_plus_one` function that accepts an `auto` value `x` and attempts to pass `x` to the earlier square metafunction. Even when the caller supplies a `constexpr` value, the function receives it in a form that cannot be used as a template argument, so the code fails [27:13-28:12].

The type-name/auto correspondence workaround

Nikpayuk presents a narrow C++17 workaround that he calls the type-name/auto correspondence. The technique converts a type into a value-like representation and then converts it back into the original type.

The mechanism uses a no-op function template named `store`. A type is encoded into the function’s signature. Non-reference types receive an additional pointer during encoding, while single and double references are preserved. The pointer is necessary for the `void` case because C++ does not permit a function with a `void` parameter, while a `void` pointer is permitted [28:21-29:57].

A reverse operation pattern-matches the stored signature and removes the added pointer where appropriate. The implementation uses type inspection such as `decltype`, a restore metafunction, and a matching step that normalizes some cases as const references for simplicity [30:04-31:15].

The practical trick is to pass a compile-time function pointer as an ordinary function argument, recover its type, restore the original type representation, and then convert it back into a value that is guaranteed to be usable as compile-time data. Nikpayuk demonstrates the technique with an `are_different` example. Two types are restored and used in the comparison, and an integer and a character are supplied through the conversion path under a `static_assert` [31:20-32:25].

He says this workaround was originally motivated not by the argument problem itself but by the desire to give metafunctions uniform parameter kinds. Separate `apply` versions are otherwise needed for a metafunction receiving a type and a metafunction receiving a value, leading to duplicated algorithms. One way to avoid that duplication is to convert all `auto` parameters to type-name parameters or all type-name parameters to `auto` parameters.

Nikpayuk’s own library uses a type-name-to-`auto` conversion policy. He says this makes it possible to pass values both as template parameters and as ordinary function arguments, and that it often improves compile-time performance, although the choice is case-dependent. He adds that C++26 reflection could provide a direct solution to this problem [32:25-33:50].

`constexpr` allocation and compile-time object lifetimes

Another limitation is what Nikpayuk calls the `constexpr` allocation problem. Objects containing heap-allocated memory cannot currently be returned from `constexpr` functions. Even class objects with internal pointer members cannot be returned as `constexpr` values under the conditions he describes [34:00-34:19].

Allocating during the compile-time scope inside a function body is often supported, but the resulting object cannot necessarily be returned and preserved as a compile-time value. Nikpayuk notes that proposals exist to improve this situation and that newer C++ versions support more standard-library containers in compile-time contexts beyond `std::array` and `std::vector` [34:19-34:50].

For classes, the constructor must be marked `constexpr`, and member functions can be marked independently. A `constexpr` object is first constructed according to the class’s normal construction rules. Once construction finishes and the object is assigned as a `constexpr` value, it becomes an immutable compile-time object. He describes the same pattern for functions: the function executes as though it were running, then its result crystallizes into a compile-time object when assigned as `constexpr` data [35:07-36:04].

To reason about this behavior, Nikpayuk divides the program lifespan into compile time and runtime, then divides compile time into meta-runtime and meta-compile-time scopes [36:10-36:41]. In his model, compile-time programming alternates between a scope in which code is executing and a scope in which the resulting value has become an immutable compile-time object. He says this terminology is his own, not standard C++ terminology, but that it helps him design systems that would otherwise be difficult to reason about.

Continuation-passing-style composition

After the basic `constexpr` discussion, Nikpayuk develops a theory of functions centered on composition [37:22]. The goal is to automate the construction and manipulation of function chains.

Manual composition is straightforward: one function’s result is passed to another. Nikpayuk converts ordinary functions into continuation-passing-style, or CPS, functions. Instead of treating a function as simply returning its final result, the CPS form constructs or receives a continuation representing what should happen next. CPS functions have their own composition operator [38:02-38:59].

He refers to a Compiler Explorer implementation of CPS composition for readers who want to inspect the mechanism [39:00]. The presentation then abstracts away from individual functions. Rather than naming a chain such as f0, f1, through fn, the chain can be represented by numerical indices k0, k1, through kn. A controller stores those indices and determines which CPS function should run next.

In the C++-style pseudocode, a function receives a controller value and a way to retrieve the next function. It uses the controller to construct a continuation, calls that continuation with an intermediate value, and carries the controller along the compile-time execution path [39:21-40:57]. A compile-time byte array can therefore act as the controller for the composition.

Nikpayuk extends this idea toward register machines. A more powerful CPS machine can treat function arguments as registers, with controllers encoded as tables rather than simple arrays. The result begins to resemble a bytecode assembly language [41:06-41:31]. He identifies this as the subject of an earlier talk on C++ metacompilers and skips the finer details here.

Variadic packs are the key C++ feature needed to simulate an arbitrary number of registers as CPS function arguments [41:40-42:16]. The machine must be able to read from and write to any register in the pack, requiring variadic indexing. C++26 provides direct pack-indexing support, including array-like selection of a particular item such as the fifth value. In his C++17 library, he relies on carefully constructed workarounds. A CPS register-assembly code example is left on screen but not explained in detail [42:49-43:18].

Rebuilding the metacompiler around the functional type system

The supplied transcript becomes disrupted around [43:35-45:04]. It records repeated pauses, audio checks, and apologies, so the technical explanation in that interval cannot be reconstructed reliably. The next coherent section resumes by restating the type system’s central constraints.

Nikpayuk says the system must support serialization, meta-assembly, and stack-allocated memory only [45:04-45:30]. These constraints arose from the specialized CPS paradigm. He developed the CPS design first and created the functional type system afterward to support it.

The initial implementation was a proof of concept intended partly to help him learn compiler theory. He conducted original research into CPS abstract machines and built lexers and parsers for the first time. Version 1.0 was difficult to maintain. Version 2.0 cleaned it up but remained messy. The metacompiler relied on a wide range of `constexpr` and template-metaprogramming tricks while trying to preserve a generic design.

Eventually, Nikpayuk concluded that the architecture had reached its limits and that further incremental cleanup would not solve the problem. He therefore rebuilt the metacompiler using a `constexpr` functional type system [46:15-47:07]. He emphasizes that there is not yet a large amount of design documentation or code to show, so the next example focuses on the internal representation.

Flat-array records and the “concord” representation

Nikpayuk’s internal records are stored in a flat array. He says users do not need to understand this representation to use the library, but he explains it to demonstrate the underlying design [47:17].

In the example, the first record is always metadata. The value 104 represents the length of the flat array, while 28 represents its current length. The initial zero is an error code indicating no error, and another zero indicates whether the second record is a table of contents.

The remaining records follow a regular layout. Records grow from left to right. The left side contains the record entry; in the shown example, its initial value of 5 is the total length of that record. The right side contains metadata that points to the next related record. The indexing is relative and refers backward to earlier records [48:04-48:43]. Nikpayuk calls these record systems “concords” and uses that term interchangeably with the type-system representation.

He gives three reasons for choosing this design.

First, flat arrays avoid the `constexpr` allocation problem. Concord objects are pre-allocated on the program stack, and the class definitions contain no internal pointer members [49:05-49:20]. This makes the representation compatible with the requirement that the type system and bytecode use stack memory rather than heap allocation.

Second, flat arrays are readily serializable. Nikpayuk gives a context-free grammar represented as a string literal that can be marked `constexpr`. A compile-time parser generator translates that grammar into a parser transition table for a particular metacompiler [49:27-49:57].

The performance measurements illustrate the benefit of preserving intermediate data. His original proof-of-concept metacompiler did not scale, but it took about six seconds on average to translate the grammar at compile time. The current library is slower for the initial lexing step: converting the grammar string into words and tokens takes about nine seconds, averaged between GCC and Clang. If the token list is written to a file and read back during the next compilation instead of regenerated from scratch, the operation takes under one second [49:57-50:51]. He suggests that the C++26 `embed` feature might improve this further.

Third, flat arrays are naturally compatible with meta-assembly bytecode. Nikpayuk previously built a proof-of-concept virtual machine that could directly evaluate meta-assembly bytecode and is researching how to integrate that capability into the current library [50:58-51:33]. More generally, he sees concords as a way to model many kinds of compile-time programs in C++, with meta-assembly capable of generating functions during compilation.

Possible applications

One application is embedded domain-specific languages. If compile-time bytecode can generate C++ functions, the same infrastructure could make it easier to define a variety of specialized languages inside C++ [51:39-51:52].

Another possibility is an alternative to C++ concepts. Nikpayuk points out that functional type theory can represent mathematical claims together with their proofs. The example claim is that for all rational numbers a and b, there exists a rational number between them [51:53-52:12]. He previously discussed a universal compile-time data structure as a possible alternative to concepts. He does not claim that proof-oriented typing is universally better; he says proof theory has tradeoffs that may make it useful in contexts where concepts are not the best fit.

His own intended application is to use these techniques and the library to build the compiler tools required for his multimedia programming language [52:35-52:50].

Lessons learned

Nikpayuk closes by describing the work as the result of ten years of trial and error [52:50]. His first lesson is that modern C++ is a multiparadigm language with considerable expressive power. The tradeoff is a large collection of features whose interactions can be difficult to manage.

The second lesson is that compiler tricks are enjoyable but standard language features should be preferred whenever they can solve the problem. He notes that he did not discuss the `const` keyword in detail and says it generally provides better type safety, feature support, and performance for functions that only need compile-time behavior [53:28-53:50].

The third lesson is that compiler tricks are still worth learning because `constexpr` support has expanded slowly. `constexpr` functions and classes can also take template parameters, but no C++ specification supports every language feature in compile-time contexts. As a result, template-metaprogramming techniques remain necessary in some situations [53:55-54:28].

Finally, he recommends organizing one’s collection of techniques around narratives and theories rather than treating them as disconnected tricks. In this talk, the theory of functions serves as the organizing “technology of mind.” He ends by returning to the possibility that ways of thinking, rather than particular material technologies, are the things most likely to persist in a changing world [54:28-55:02].

Key Takeaways

  • A functional type system, as defined for this talk, is built on the untyped lambda calculus and includes both atomic types such as integers and booleans and compound types such as products, co-products, and lists.
  • A practical type system must record values as well as types. Nikpayuk’s version additionally includes identity-style comparison types, dependent products, and dependent pairs.
  • The system’s unusual constraints come from its CPS architecture: it must be serializable, represent functions as assembly-like bytecode, and restrict bytecode memory requests to the stack.
  • Template metaprogramming effectively creates a second programming language on top of C++ by treating grammar features as compile-time functions. It can improve early resolution and dispatch, but it shifts type-checking responsibility to the programmer and can become syntactically complex.
  • Variadic, alias, and class templates have different roles as metafunctions. Variadic templates are suited to value-oriented computations, aliases return types without specialization-based pattern matching, and class templates support specialization and therefore pattern matching.
  • Eager evaluation can make recursive template code fail unexpectedly. In Nikpayuk’s factorial example, a ternary expression evaluates the recursive branch even at the base case, eventually exceeding GCC and Clang’s template-instantiation depth.
  • `constexpr` executes ordinary C++ at compile time when permitted, but the keyword is not an unconditional guarantee. The compiler may compile the same code for ordinary runtime execution if the context requires it.
  • A `constexpr` function cannot generally treat an argument as a template argument, even if the caller supplied a compile-time value. Nikpayuk’s C++17 type-name/auto correspondence workaround encodes types through function signatures and reconstructs them later.
  • Compile-time object design must account for allocation restrictions. Heap-backed objects and objects containing internal pointers cannot necessarily be returned as persistent `constexpr` values, making flat, pointer-free records useful.
  • CPS composition turns a sequence of functions into a controller-driven execution path. Compile-time byte arrays or tables can select the next function, while variadic packs simulate the registers of a bytecode-like machine.
  • Flat-array “concords” provide one representation for serialization, stack allocation, compile-time parsing, and possible bytecode execution. Caching serialized tokens reduced the reported lexing step from about nine seconds to under one second.
  • The larger design lesson is to prefer standard C++ features where possible, retain compiler tricks where feature gaps require them, and organize techniques around a coherent theory rather than an unrelated collection of hacks.

Practical Notes

The implementation path described by the talk starts by defining a type-and-value model rooted in the untyped lambda calculus. The design must then choose how to represent compound values, functions, comparison types, and dependent structures while preserving serialization and stack-only constraints.

For template metaprogramming, select the metafunction form according to the result needed. Use variadic/value-oriented templates for computations that return values and use `auto` when value parameters are needed. Add an integral concept or a similar C++20 concept when `auto` is too weakly typed. Use alias templates for straightforward type construction such as an integer array with a compile-time size. Use class templates when specialization and pattern matching are required.

Recursive metafunctions should make the base case structurally distinct from the recursive case. The factorial example shows why a seemingly natural ternary implementation can exceed the compiler’s template-instantiation limit through eager evaluation. Class-template recursion is an alternative, but Nikpayuk reports that it has worse compile-time performance.

When moving to `constexpr`, mark functions and constructors appropriately, but do not assume that every call will execute during compilation. A function body cannot normally use an ordinary argument as a template parameter. If working within the C++17 limitations described in the talk, the `store`/restore technique provides a narrow type-to-value and value-to-type bridge, using function signatures, `decltype`, pointer encoding for non-reference types, and special handling for references and `void`.

The type-name-to-`auto` normalization policy is an architectural choice rather than a universal rule. Nikpayuk prefers it because values can then be passed as both template parameters and function arguments, and because it can improve compile-time performance in some cases. The best choice is dependent on the particular algorithms and parameter flows.

For data representation, flat arrays avoid internal pointers and make records easier to serialize. The described pipeline begins with a `constexpr` string literal containing a context-free grammar, lexes it into words and tokens, and produces parser transition data for a metacompiler. Persisting the token list between compilations is an important practical optimization: the talk reports roughly nine seconds for lexing from scratch and under one second when reusing serialized tokens.

The CPS architecture should be designed around an explicit controller. A simple compile-time byte array can select the next function in a composition chain. A more advanced register-machine design uses tables and variadic packs to model bytecode instructions and registers. C++20 supports the variadic-pack examples discussed, while the speaker’s own C++17 implementation requires indexing workarounds. He refers to direct pack indexing, reflection, and `embed` as C++26 capabilities that could reduce some of this complexity, but he presents them as future or newer-language support rather than as dependencies of the current implementation.

The main risks are compile-time cost, compiler-specific limits, incomplete `constexpr` feature coverage, template grammar complexity, and the difficulty of returning persistent values that contain heap-backed state. GCC and Clang both appear in the reported timing averages, but the transcript does not provide compiler versions, build commands, flags, hardware details, or a reproducible benchmark harness. No command-line workflow is supplied.

The proposed proof-oriented alternative to concepts should be treated as an exploratory design direction. The talk establishes that functional type theory can represent claims and proofs and gives a rational-number example, but it does not present a complete replacement for C++ concepts or compare their performance and usability in detail.
