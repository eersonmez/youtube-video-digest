Advanced .NET testing techniques to improve test effectiveness and coverage.

NDC Conferences · 2026-07-08T08:54:13+00:00 · .NET Testing Techniques You Didn’t Know You Needed - Dante De Ruwe - NDC Copenhagen 2026

What This Video Is About

This video presents advanced testing methodologies for developers working with .NET to move beyond basic example-based testing and significantly improve test effectiveness and code coverage. The central theme is building trust in the codebase by ensuring tests are not just present, but actively guard against regressions and uncover subtle bugs that standard testing might miss.

The presentation covers several sophisticated techniques—including randomized testing, property-based testing, snapshot testing, and mutation testing—along with metrics like branch coverage and mutation score to evaluate test quality. It addresses common pitfalls in testing, such as incomplete tests or flaky tests, providing concrete strategies for mitigation. The goal is to help developers write fewer, but far more effective, tests that document system behavior and provide a robust safety net against future changes.

Detailed Walkthrough

The discussion begins by establishing the value of testing beyond just finding bugs; it serves as living documentation of system behavior [0:00]. Tests are also crucial for providing fast feedback loops during incremental development and protecting against regressions through Test-Driven Development (TDD) [3:00].

The speaker outlines a decision tree based on test outcomes:

  • Failing tests after code change: This is generally good, indicating the tests caught a break in functionality, provided requirements haven't changed [5:00].
  • Tests passing after code change: If this was unexpected, the tests are incomplete and need fixing. If expected (e.g., refactoring without changing behavior), it indicates successful refactoring [5:00].
  • Tests failing with no code change: This signals a flaky test, which requires investigation [10:00].

To address incomplete tests, strategies include identifying missing edge cases or forgotten assertions. The speaker introduces Randomized Testing as an alternative to manually listing every edge case. By generating random inputs instead of relying solely on predefined examples, randomized testing guards against human bias in test creation [10:00]. For complex data structures, the library Bogus can generate realistic fake data based on defined rules (fakers) [12:34].

However, introducing randomness can lead to flaky tests, especially when dealing with time zones or transition days. A war story from an electricity management company illustrates this, where randomized date generation caused issues around daylight saving time transitions [15:00]. Flaky tests degrade confidence in the entire test suite. Mitigation strategies for flakiness include using abstractions for time (like `TimeProvider`), ensuring deterministic data via seeding (e.g., passing a seed to a random number generator), or increasing sample sizes until statistical certainty is reached [20:00].

When randomized testing fails to provide clear expectations, the solution shifts to Property Based Testing. Instead of asserting specific outcomes for inputs (like 1+1=2), property-based testing asserts general properties that must *always* hold true for any valid input. For an addition function, this means checking commutative (a+b = b+a), associative, and identity properties [25:00]. The library FsCheck is introduced as a tool for this, which automatically generates random parameters and runs many tests per definition to test these invariants [25:00].

To handle situations where an output structure changes unexpectedly (e.g., in complex XML or API contracts), Snapshot Testing (also called golden master testing) is used. Instead of writing numerous assertions for every field, the framework records the expected output once—the "snapshot"—and subsequent runs verify that new outputs match this stored baseline [35:00]. The library Verify by Simon Crop is mentioned as a popular .NET tool for this purpose, which can be applied to OpenAPI specifications to guard against unintended contract changes [35:00].

Finally, the video discusses Metrics to assess test quality. Line coverage measures executed lines but is insufficient. Branch Coverage is better as it checks if all possible paths in conditional logic have been tested (e.g., both sides of an `if/else` statement) [40:00]. The most advanced metric discussed is the Mutation Score. This involves introducing small, deliberate bugs (mutations) into the code and checking if existing tests are capable of catching them. A high mutation score indicates that tests are truly effective at detecting incorrect behavior, not just executing lines of code [40:00]. Striker is presented as a primary .NET tool for performing mutation testing [45:00].

Key Takeaways

  • Tests serve multiple purposes beyond bug finding, including documenting system behavior and providing fast feedback loops during development.
  • Randomized testing guards against human bias by generating random inputs rather than relying solely on predefined edge cases.
  • Property based testing shifts the focus from specific outcomes to general mathematical properties that must always hold true for a function's implementation (e.g., commutativity).
  • Snapshot testing is ideal for tracking complex data structures or API contracts, verifying that the output structure remains consistent across changes without requiring exhaustive manual assertions.
  • Flaky tests undermine trust in the test suite and should be addressed by using deterministic data generation via seeding or increasing sample sizes.
  • Branch coverage is a superior metric to line coverage because it verifies that all logical paths (like `if` and `else`) have been exercised during testing.
  • Mutation score measures the actual effectiveness of tests by checking if they can detect deliberately introduced, small bugs in the code.

Practical Notes

  • Randomized Data Generation: Use libraries like Bogus to create realistic fake data for randomized tests by defining rules within a faker object (e.g., specifying country codes).
  • Time Handling: When dealing with time-sensitive logic, use abstractions like `ITimeProvider` or ensure all internal code operates strictly in UTC, mapping offsets only at input/output boundaries to prevent issues related to daylight saving transitions.
  • Property Testing Implementation: Use libraries such as FsCheck (available for C# via .NET packages) and define properties based on mathematical invariants of the operation being tested.
  • Snapshot Testing Tool: The Verify library by Simon Crop is recommended for tracking changes in complex outputs like XML or OpenAPI specifications.
  • Mutation Testing Tool: Striker is a primary mutation testing tool available for the .NET space; it can be run via `dotnet tool global install` and provides dashboards to analyze which introduced bugs were caught (killed) versus those that survived.
  • Decision Point on Metrics: Do not treat metrics like line coverage or mutation score as absolute targets, but rather use them as indicators of test quality and areas needing improvement.
