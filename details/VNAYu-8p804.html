Building a real-time collaborative text editor for airgapped defense environments.

NDC Conferences · 2026-07-03T10:49:29+00:00 · Building a real-time collaborative editor - Mihaela Gheorghe-Roman - NDC Copenhagen 2026

What This Video Is About

This video details the development of a highly specialized collaborative text editor designed to function within airgapped and disconnected defense environments where internet connectivity is unavailable or strictly prohibited due to security protocols. The core challenge addressed was enabling multiple users across disparate, potentially offline systems to work on shared documents in real time, requiring robust mechanisms for merging changes once servers reconnect after extended periods of isolation.

The solution goes significantly beyond standard rich text editing by integrating complex security requirements, specifically granular access control down to individual paragraphs or links based on user security clearance levels. Furthermore, the system must maintain compatibility across different versions and deployment types—ranging from laptops to specialized hardware inside military vehicles—while ensuring all users converge on a consistent document state despite concurrent edits occurring offline.

Detailed Walkthrough

The presentation begins by establishing the context of the application's users: personnel operating in various domains (land, air, sea) during mission planning phases [5:00]. These users require real-time access to documentation containing organizational structures, asset details, and target information. A critical requirement is granular security clearance, meaning a user's rank dictates access not just to the document generally, but to specific sections, paragraphs, or embedded objects within it [10:00].

The development process started by defining what constitutes a rich text editor—a sequence of characters identified by an index and value. Initially, this was reduced to two basic operations: insert and delete, as updates are treated as combinations of these primitives [10:00]. Advanced features were then added, including styling (bold, italics), handling embedded objects like links, pictures, and videos, and implementing classification security at the object/paragraph level [10:00].

When evaluating existing market frameworks (like Froala or TinyMCE), the team found that while some offered rich text editing capabilities, none provided a viable solution for the collaborative aspect under their specific constraints [15:00]. The decision to build in-house was driven by licensing restrictions on server counts and the inability of these commercial tools to allow proprietary defense code sharing [15:00].

The core problem—real-time collaboration across disconnected nodes—was tackled by aiming for consistency and responsiveness. Initial attempts, modeled after Git conflict resolution (where manual merging is required), were deemed unusable due to the need for automated, seamless operation [20:00]. The team shifted focus to achieving consistency through mathematical principles of commutativity and idempotency, leading them to adopt Operational Transformation (OT) algorithms [35:00].

The OT approach requires transforming indices before applying operations. For example, if User One inserts "collaborative" at position 14, and User Two simultaneously tries to insert "editor" at position 14, the system must transform User Two's index based on User One's insertion so that both resulting documents are consistent (e.g., "collaborative editor") [25:00]. For deletions, OT ensures idempotency by checking if an operation has already been applied to prevent cascading deletes beyond user intent [35:00].

The complexity increased with advanced objects and operations. The system extended its JSON action/operation format beyond simple text insertion/deletion to handle object types (tables, lists, images) and complex structural changes like renaming or splitting paragraphs (wrapping operation) [40:00]. Furthermore, cursor synchronization was implemented to track user positions accurately even when insertions shift the document structure [45:00].

Finally, security and usability features were layered on. Undo/Redo functionality is managed via a stack manager that reverses operations by inverting them (e.g., reversing an insertion with a deletion) [45:00]. The most stringent requirement involved classification: sensitive information can be encrypted at the system level, rendering it invisible to users without the appropriate clearance, even if they are technically part of the document structure [50:00].

Key Takeaways

  • The editor is specifically engineered for airgapped and disconnected defense environments where internet connectivity is absent or restricted.
  • Conflict resolution relies on implementing Operational Transformation (OT) algorithms to ensure consistency between concurrent edits without requiring manual user intervention.
  • Security clearance can be applied with high granularity, restricting visibility down to specific paragraphs or embedded objects within a document.
  • Operations are managed via custom JSON objects termed "actions" or "operations," which define the type of change (e.g., insert, delete, rename).
  • The system supports complex data structures beyond plain text, including tables, lists, images, and videos, by leveraging tree-based parsing concepts similar to HTML.
  • Cursor synchronization is necessary to accurately track user positions within a document that is dynamically changing due to concurrent edits or object insertions.
  • Undo/Redo functionality is achieved through a stack manager that reverses operations by applying their inverse transformation.
  • Data security is managed at the system level using encryption, ensuring restricted content remains inaccessible but present in the data model for authorized users.
  • The architecture supports decentralized operation; each node can act as a master/owner of a document during synchronization events.

Practical Notes

  • Technology Stack: Backend utilizes Scala and Java, with Microsoft SQL and PostgreSQL databases managed within a Kubernetes cluster. Frontend is built using Angular (with some legacy AngularJS components being migrated).
  • Deployment: Deployment in air-gapped environments is manual or via automation scripts; installation must be triggered by personnel physically present at the deployment site (e.g., inside a vehicle).
  • Security Implementation: Encryption for classified content occurs at the system level, not the user level. Interoperability protocols include additional security layers to manage data transfer across air gaps.
  • Backwards Compatibility: A two-year compatibility policy is maintained; when newer features (like tables) are present in one version and absent in another, information is translated into a displayable format for older systems (e.g., tabular data rendered as tabbed lines).
  • AI Usage Policy: AI is restricted to providing suggestions or alternative scenarios; human judgment must always confirm any changes made based on AI input due to strict defense technology policies and model restrictions.
