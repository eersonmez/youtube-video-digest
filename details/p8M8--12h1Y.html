The implementation of an autonomous development workflow at Nordic Corporate Bank using an AI agent named Nil Skorg.

NDC Conferences · 2026-06-29T11:37:13+00:00 · How We Built an Autonomous Development Workflow at Nordic Corporate Bank -

What This Video Is About

This video details the implementation of an autonomous development workflow at Nordic Corporate Bank, a small financial institution with 26 employees. The core subject is the deployment and evolution of an AI agent named Nil Skorg within Azure to automate significant portions of the software development lifecycle (SDLC). Initially, the agent was integrated into Azure DevOps to handle coding, testing, and documentation. Over time, the workflow has been advanced to include an AI-driven requirement validation stage, aiming to mimic the critical thinking of a senior developer.

The implementation showcases how this agent can drastically increase code output—moving from 141 pull requests in one month previously to 572 after Nil Skorg was deployed. The system operates as a multi-stage pipeline utilizing various models for research, planning, implementation, and review. While the AI handles much of the execution, the human team remains responsible for high-level architecture, risk assessment, and strategic prioritization.

Detailed Walkthrough

The process began in November 2025 after inspiration was drawn from external talks on agent coding. The development team made a deliberate choice to build an independent, cloud-based AI agent rather than running it locally. Nil Skorg was deployed in Azure and integrated with Azure DevOps using webhooks, gaining its own user identity in Entra and email address [6:43].

The initial capabilities of Nil Skorg included answering questions, writing code, creating and running tests, generating pull requests (PRs), reviewing PRs, arguing within PR threads, and creating documentation like ADR records. A key feature is the memory system, adapted from a framework called Squad; when the agent completes a task, it assesses its output for reusable knowledge and updates its own memory file to facilitate self-improvement on new tasks [8:08].

The workflow evolved significantly by addressing bottlenecks. Initially, coding was the bottleneck, but once Nil Skorg took over implementation, requirement gathering became the primary constraint. To solve this, instructions were given to Nil Skorg regarding epic specifications—requiring a description, acceptance criteria, and a certain confidence level for implementation [10:57]. If these requirements are insufficient, the agent is instructed to ask clarifying questions back to the product owner rather than proceeding blindly [11:23].

The interaction between the AI and human stakeholders is demonstrated through examples in Azure DevOps. For instance, when a Product Owner (like Kaya) submits an epic without technical detail, Nil Skorg can either stop due to hard rules or proceed by self-generating descriptions and acceptance criteria before starting implementation [12:51]. This allows for rapid iteration; one example showed an epic moving from creation to verification within the same day [14:30].

The underlying architecture is a complex system built around Nil Skorg. It starts with triggers (manual mention, task assignment, or CI build failures routed via Azure Service Bus) which create tasks stored in a Cosmos DB hub [32:05]. These tasks spin up a worker running as a Docker image within an Azure Container App Job, powered by the GitHub Copilot SDK [32:39].

The core of the development process is the implementation workflow, which consists of five stages, each pinned to a specific model and reasoning level. The first stage is a research phase where the most expensive model performs planning and codebase understanding [36:02]. This is followed by a cheap sanity check using a fast model [36:17], then the implementation phase [36:28], a mechanical build/test/commit/PR creation stage using a fast model [36:48], and finally, a self-review pool. The review pool involves multiple models—one set for initial review, a faster cheaper model for another review, and GPT5 as the consolidator that verifies findings before assigning it to a human [37:01]. Crucially, Nil Skorg is instructed to act like a senior developer who must be critical and push back against assumptions [37:58].

Key Takeaways

  • The AI agent, Nil Skorg, was deployed in Azure and integrated with Azure DevOps via webhooks for task management.
  • A multi-stage pipeline utilizes different models (some expensive, some cheap/fast) across research, planning, implementation, and review to mimic senior developer critical thinking.
  • The workflow has evolved from simple coding assistance to an AI-driven requirement validation stage that prevents premature implementation errors.
  • Nil Skorg maintains a self-improving memory system by assessing completed work and updating its internal knowledge base.
  • The system architecture relies on Azure services: API Gateway (App Service), task storage (Cosmos DB), and execution environment (Container App Jobs).
  • Human oversight remains critical for high-level architectural decisions, risk assessment, and final approval of PRs, even as the agent automates coding.
  • To ensure predictable behavior, the system uses guardrails like regex expressions to filter technical jargon when communicating with non-technical product owners.
  • The development process is becoming less about writing code and more about defining high-value problems and managing the AI's output.

Practical Notes

Implementation Steps/Architecture:

  • Trigger Mechanism: Tasks can be initiated by mentioning Nil Skorg, assigning tasks to him, or receiving exceptions via Azure Service Bus (Application Insights monitoring).
  • Task Hub: An API Gateway reads triggers and stores the task in a Cosmos DB instance, which acts as the central hub.
  • Execution: A worker, built on the GitHub Copilot SDK within an Azure Container App Job, is spun up to process the task.
  • Workflow Control: The implementation workflow has five stages: Research (most expensive model for planning), Sanity Check (cheap/fast model), Implementation, Mechanical Build/Test/Commit/PR Creation (cheapest model), and Self-Review Pool (multiple models culminating in GPT5).
  • Guardrails & Determinism: To prevent unpredictable behavior, the system uses:
  • Specific instructions for how to create agent skills, optimizing them for AI consumption rather than human reading.
  • Regex expressions applied to output directed at Product Owners to strip technical file paths or jargon.
  • Context management limits (e.g., compaction borders set at 68 and a hard stop at 92) and response caps on tools to prevent context flooding.

Risks/Caveats:

  • The current bottleneck has shifted from coding to requirement definition, which the team is actively trying to automate by having Nil Skorg validate epics.
  • Full autonomy (removing humans entirely from PR verification) has not yet been achieved and requires further development.
  • Developers must accept a shift in role; they are moving away from deep code diving toward higher-level developing, architecture, and risk judgment.
