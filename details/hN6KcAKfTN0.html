Benchmarking and optimizing the Carbon compiler, focusing heavily on hash table performance.

NDC Conferences · 2026-07-23T11:00:29+00:00 · Benchmarking and optimizing the Carbon compiler - Chandler Carruth - NDC Toronto 2026

What This Video Is About

This video details a deep dive into the process of benchmarking and optimizing the C++ implementation of the Carbon compiler, focusing heavily on performance bottlenecks within its data structures, specifically hash tables used by the lexer. The central theme is that effective optimization requires rigorous, context-aware benchmarking rather than relying on general performance metrics. The speaker uses the lexer's identifier hash table as a case study to illustrate how misleading incorrect benchmarks can be and how critical it is to tailor tests to real-world usage patterns within a specific application like a compiler.

The presentation moves beyond simple timing by employing hardware performance counters and sophisticated source code generation techniques to simulate realistic compilation workloads. The goal is not just to make the hash table faster in isolation, but to understand its impact on overall compile time across different phases of compilation (lexing, parsing, semantic analysis). Ultimately, it serves as a guide for software engineers on how to engineer robust benchmarks that accurately reflect production behavior and drive meaningful performance improvements.

Detailed Walkthrough

The speaker begins by profiling a handwritten Carbon example and discovers that the lexer is significantly slower than other compiler components [0:35]. The primary culprit identified was the identifier hash table, which stores every unique identifier encountered during compilation for interning purposes. This finding was surprising as lexers are often considered simple parts of compilers.

To address this, a custom wrapper API was defined to benchmark specific operations like `insert` and `lookup`, rather than general hash table performance [5:00]. The lookup operation was specifically designed to require accessing the stored value (e.g., checking if it's non-empty for strings), ensuring the test reflects real usage, as a simple existence check is insufficient [5:38].

The benchmarking process utilizes Google's benchmark library. Key parameters controlled in the benchmarks include the size of the hash table and the number of distinct keys being looked up [9:00]. A crucial technique discussed for accurate measurement is suppressing compiler optimizations on loop indices during the benchmark execution, as compilers are adept at optimizing loops in ways that invalidate performance measurements [10:34].

For measuring latency—the time from starting an operation to receiving a result—a specific trick is employed. Instead of incrementing the loop index `I`, the value added to it is dynamically derived from the result of the hash table lookup itself. This forces the CPU to execute the full lookup and examine the resulting value before proceeding, preventing speculative execution from masking true latency [13:42].

Another meaningful test for insertion speed is cycle detection, analogous to depth-first search in graph algorithms. The benchmark inserts a sequence of keys until it attempts to insert one that already exists (a collision), thereby testing the failure path of the hash table's insertion logic [15:00].

When benchmarking the entire compiler, the granularity shifts from individual keys to source files. Source code is generated on the fly using specialized generators whose API shapes (number of methods, fields, etc.) can be controlled via parameters like class and type use configurations [34:00]. The generator supports deterministic sampling based on histograms derived from real-world distributions, rather than uniform random selection, to ensure source code represents reality [45:00].

The compiler benchmark measures throughput metrics such as bytes per second, lines per second, and tokens per second. Instead of timing the start and end of a phase in isolation, the measurement captures the total time taken to reach the end of that specific phase (e.g., after lexing) [35:40].

Finally, optimizations discussed include table-based dispatch for the lexer. This technique builds a function pointer table indexed by the first byte read from the source file; instead of using traditional switch statements or gotos, it branches directly to the appropriate token handling routine based on this index, which is noted as being significantly faster than compiler-generated alternatives [1:00:00].

Key Takeaways

  • The lexer's identifier hash table was identified as a major performance bottleneck in the Carbon compiler during initial profiling.
  • Custom wrapper APIs must be created to benchmark specific, relevant operations (like lookup hit vs. miss) rather than general API performance.
  • Benchmarks should use techniques like suppressing loop optimizations on indices to prevent the compiler from optimizing away the measurement target [10:34].
  • To measure true latency in tight loops, the next iteration's input must be dependent on the result of the current operation (e.g., using lookup results to determine the next key) [13:42].
  • Cycle detection provides a meaningful way to benchmark insertion speed by forcing the hash table into its collision/failure path [15:00].
  • When benchmarking compilers, throughput metrics like lines per second are often more relevant than microsecond latency measurements for large workloads.
  • Source code generation must use deterministic sampling based on histograms derived from real-world distributions to ensure benchmark validity across runs [45:00].
  • Table-based dispatch is a highly effective optimization technique in the lexer, replacing switch statements with direct function pointer lookups indexed by input bytes [1:00:00].
  • It is crucial to invest significant effort into engineering high-quality benchmarks, as they are necessary to discover and validate performance optimizations.

Practical Notes

Hash Table Benchmarking:

  • Use Google's benchmark library for implementation.
  • Define wrapper APIs for `insert` and `lookup`. The lookup must access the stored value (e.g., check if a string is non-empty).
  • Control test dimensionality by varying hash table size and number of distinct keys looked up.
  • For latency measurement, dynamically derive the next loop index from the result of the current operation to prevent speculative execution from masking true latency.

Compiler Benchmarking:

  • Benchmark granularity should be set at the source file level rather than the key level for compiler phases.
  • Source code generation must control API shape (e.g., number of methods/fields per class) and use weighted sampling based on histograms to mimic real-world distributions, avoiding uniform random selection [45:00].
  • When measuring compilation phases, measure the total time taken to reach the end of a phase; do not attempt to start and stop timers mid-phase as this adds noise.

Optimization Techniques:

  • Table-based dispatch involves creating an array of function pointers indexed by input bytes (e.g., the first byte of a symbol) to achieve faster routing than switch/goto statements [1:00:00].
  • The speaker suggests that for hash tables, specializing the implementation rather than trying to create a general-purpose abstraction is often more effective.
