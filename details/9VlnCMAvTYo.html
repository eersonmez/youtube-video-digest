This talk presents a case study of stabilizing and scaling a legacy embedded smart-metering concentrator.

NDC Conferences · 2026-07-22T08:30:07+00:00 · Boosting Software Efficiency : A Case Study of 100% Performance Improvement in ... - Gili Kamma

What This Video Is About

This talk is a case study in rescuing and scaling a legacy embedded smart-metering concentrator. In 2017, Gili Kamma inherited a Linux/C++/Qt device used in water-metering deployments. Wireless water meters in homes transmitted consumption measurements to the concentrator, which collected and processed them before forwarding data to a cloud or billing system over cellular or Ethernet. Because the business depends on complete measurement data, missing readings could mean lost revenue and contractual fines.

The product had worked with fewer than 1,000 meters, but sales began offering it for much larger neighborhoods after a hardware calculation suggested that it could support up to 7,500 meters. In real deployments, especially those with more than 5,000 meters, the device suffered unexplained resets, crashes, and occasional data loss during network problems. These failures did not occur in the office, making them difficult to reproduce and diagnose. The device had only 32 MB of RAM, an old Linux environment that could not be changed, and C++ code written in a largely pre-modern style despite being compiled as C++17.

Kamma’s solution was not a single rewrite or hardware upgrade. He measured allocations, built a small internal load simulator, reproduced the production crash at 6,000 simulated meters, and then reduced peak memory by changing message scheduling, batching, string representation, and allocation strategy. He also added reset diagnostics, nonvolatile acknowledgment-based queuing, UART recovery, proactive monitoring, and automated long-running tests. The reported result was eight months without resets, support for up to 10,000 water meters, and releases every two to three months instead of the previous three-engineer, six-month release effort.

Detailed Walkthrough

The system and the scale problem

Kamma’s story begins in 2017, when he joined a meter company and took responsibility for a water-metering product [1:27]. A water meter installed in a home measured consumption and transmitted its readings wirelessly at a configured interval ranging from seconds to several minutes. In a neighborhood, hundreds or thousands of meters could be transmitting measurements.

The concentrator sat in the middle of this system, installed high up near a large antenna. It received wireless readings from the meters, performed processing and business logic, and sent the resulting data to a backend system through a cellular modem or Ethernet connection [2:04]. The backend was responsible for functions such as billing.

The commercial importance of reliability was direct: the product’s business value was the data. Losing measurements meant losing money, and the company could be fined for missing data [3:16]. The concentrator originally worked well with fewer than 1,000 meters. Later, sales wanted to use it in larger projects. An engineer calculated that the hardware should support as many as 7,500 meters, and the product was sold on that basis [3:32].

In practice, larger deployments were unstable. Customers saw crashes and missing data, while the development team received complaints without being able to explain the failures. The specific symptoms included memory problems, unexplained resets, and data loss during some network outages but not others [4:05]. Nothing failed in the office, where Kamma had only about 300 real water meters available for testing.

The technical constraints made conventional solutions difficult. The device had 32 MB of RAM that could not be upgraded. It ran an old Linux kernel and a fixed software image. The application was C++17, but its coding style was closer to C++11 or earlier, with no smart pointers. It was built with Qt, which Kamma described as making embedded programming approachable but also encouraging frequent object allocation without automatically reclaiming those objects [5:13].

His initial goal was to eliminate data loss and make the product stable [6:35]. His first plan was conventional: understand the code, find the logs and main modules, reproduce the failures on a desk, and then fix the causes [6:58].

The first hypothesis: leaks and fragmentation

The code contained many explicit `new` and `delete` operations, which immediately suggested memory leaks. Kamma also considered memory fragmentation. He explained fragmentation with a simple example: start with 100 free bytes, allocate blocks of 40, 20, and 40 bytes, then release the two 40-byte blocks. Although 80 bytes are free in total, there is no contiguous 50-byte region, so a request for 50 bytes can still fail [7:46].

That theory seemed to explain the resets. A failed allocation could return a null pointer; code that used the pointer without checking it could then trigger an uncontrolled reset. Kamma spent two weeks cleaning up small leaks, expecting the larger crashes to disappear [8:56].

The measurements eventually showed that this explanation was incomplete. There were some minor leaks, but not enough to account for the production behavior, and he found no meaningful memory fragmentation. The important lesson was that the first plausible explanation was wrong. He needed instrumentation and a way to create the production load instead of relying on intuition.

Instrumenting allocation behavior

Kamma could not use a standard memory profiler because the Linux kernel was too old [9:20]. Instead, he implemented a profiler inside the application by overloading the C++ `new` and `delete` operators.

The profiler tracked several dimensions of allocation behavior:

  • Allocations per second, distinguishing a light workload such as two allocations per second from a potentially destructive rate such as 2,000 allocations per second.
  • Current and maximum numbers of live allocations.
  • Current and maximum bytes allocated.
  • Current and maximum allocation counts grouped by allocation size.

A stable maximum is expected when the program releases everything it allocates. If the maximum number of allocations or maximum allocated bytes continues to grow, that indicates a leak. Grouping allocations by size makes it easier to identify a leak associated with a particular object size. It also helps investigate fragmentation: a system allocating only uniform 100-byte blocks is less likely to fragment than one allocating many different sizes [10:36]-[12:39].

The implementation stored allocation metadata directly in the allocated block. On every allocation, the custom `new` reserved the requested size plus four extra bytes, wrote the allocation size into those four bytes, updated the profiler, and returned a pointer four bytes after the start of the block. On `delete`, the code moved back four bytes to recover the stored size, updated the profiler, and released the original block [12:42]-[14:07].

This instrumentation confirmed that the leaks he found were minor. After two weeks of cleanup, the major production failure still remained. The profiler had prevented him from continuing indefinitely in the wrong direction, but it had not yet reproduced the real problem.

A ten-line internal load simulator

The second tool was a load simulator designed to reproduce thousands of meters in the office. Rather than generating RF traffic with special hardware, Kamma reused real messages received from the approximately 300 meters available in the office [14:39].

Each message contained an opcode, a unique water-meter ID, and a measurement value. The simulator was built inside the product itself. A file on the Linux device enabled or disabled simulator mode, so switching modes did not require a separate build. When a real message arrived, the application placed it on its normal processing queue. If simulation was enabled, another thread duplicated the message, changed its meter ID to a unique fake ID, and put the fake message on the same queue as the real one. This duplication could be repeated up to 100 times, depending on configuration [15:46]-[17:30].

The implementation was only about ten lines of code. It required no signal generator, no additional RF hardware, and no separate test build. It also exercised the same queue and processing path used by real meter messages, rather than testing a disconnected mock path.

Kamma used aggressive timing configurations to apply more pressure than normal operation. For example, a message normally sent once per hour could be sent every five minutes; a ten-minute message could be sent every four minutes; and a five-minute message could be sent every two minutes [17:41]. With this configuration, the device consistently crashed on his desk when simulating 6,000 meters [17:41]-[19:04].

Reproducing the crash and finding the memory bottleneck

The simulator transformed an intermittent field problem into a repeatable local failure. Kamma combined it with a conditional debugger breakpoint that stopped when a pointer was null [20:16]. Instead of investigating after the reset had already damaged the program state, he could stop immediately before the failure and inspect the call stacks, variables, and threads.

The cause was not a large persistent leak. Two threads were simultaneously constructing large outgoing messages, each approximately 5 MB. The second thread failed because the system could not provide enough memory for both messages [21:02].

The first fix was to replace the parallel message-building path with sequential processing. Kamma removed one thread so that the device prepared and sent message A, then prepared and sent message B. The peak requirement fell from approximately 10 MB to 5 MB, which stopped the immediate crash [21:43]-[22:14]. However, this was not the final solution because the target was still 7,500 meters rather than merely avoiding the 6,000-meter failure.

Further investigation showed why the problem appeared only at larger sites. Message size grew with the number of meters being monitored. The application collected the readings, created a large JSON message, encrypted it, and compressed it. At 5,000 meters, one message required approximately 5 MB at its peak. At 1,000 meters, the equivalent message required approximately 1 MB. Two 1,000-meter messages could fit in the available memory, while two 5,000-meter messages could not [22:32]-[23:38].

The next change was to process the readings in batches of up to 1,000 meters. The concentrator created and compressed one batch, using approximately 1 MB at peak, then moved on to the next batch rather than constructing one massive message [23:43]-[24:12].

Kamma then reduced the memory footprint of the message path itself. Qt character storage used two bytes per character in the relevant code, while the C++ representation he used for the message path used one byte. Changing the message preparation and message representation reduced the batch requirement from approximately 1 MB to half a megabyte [24:24]-[24:57].

At that point, the message buffer no longer needed to be dynamically allocated. Kamma replaced the dynamic allocation with a static array. Since that path no longer depended on a successful dynamic allocation, it could not fail in the same way [25:07]. Together, serializing message construction, splitting messages into batches, reducing representation size, and using a static buffer more than doubled the number of meters the unit could support.

Preserving evidence across resets

Solving the memory bottleneck did not eliminate the need to understand failures in deployed devices. The product had normal application logs stored in nonvolatile memory, so those logs survived a reset. However, the old Linux environment did not provide remote access to them [25:34].

The product also had an event mechanism that collected selected events in RAM and periodically sent them to the backend. That worked during normal operation, but it failed precisely when it was most needed. During a low-memory crash, the application could not allocate the event data or build the outgoing event message. After the reset, the RAM-based event queue was empty, so the backend showed only that the unit had reset, not what happened immediately beforehand [26:04]-[27:03].

Kamma’s workaround was deliberately simple. When the unit woke up after a reset, it sent the last 100 lines of the application log to the backend through the existing event service. He described using `tail -n 100` to capture the recent log lines, placing them into a short-log string, and sending them as an event along with the reset information [27:12].

He chose to send those lines every time rather than adding conditions to decide when they were necessary. This was not a complete crash dump, but it was a high-value, low-effort diagnostic improvement. He characterized it as an approximately 80-percent solution for 20 percent of the effort and acknowledged that there were edge cases where the last 100 lines would not be sufficient [28:02].

Decoupling data collection from network availability

Another failure mode involved occasional data loss during network problems. Measurements and transmissions waiting for further processing remained in RAM. When connectivity was unstable, the backlog grew and consumed more memory. A reset also erased everything still in RAM [28:28].

The solution was to separate data production from data transmission. One thread handled the application logic: it determined when data should be sent, created the message, and placed it into a nonvolatile queue. A second thread handled sending. It repeatedly checked the queue, attempted to transmit the next message, and removed that message only after the remote side acknowledged successful receipt [29:17]-[30:14].

If sending failed because of a timeout or a bad acknowledgment, the sender left the message in the queue, waited about a minute, and retried it. The system no longer needed separate logic for every possible network state. It performed the same queue-and-retry loop regardless of whether the network was currently available.

Kamma gave two reasons not to rely on a preliminary network-status check. First, connectivity can change immediately after it is checked. Second, the local network can be functioning while the backend or the route to the backend is unavailable [30:27]-[31:07].

When data was collected once per hour, deleting each item only after acknowledgment limited the maximum loss to one hour under the stated design. A 23-hour outage could be followed by successful transmission when connectivity returned. He also used a two-week unpaid-SIM scenario to illustrate that the queue could preserve data through a prolonged outage and send it later [31:16]-[32:15]. After this change, the product no longer experienced significant data loss of the earlier kind.

Recovering from a silent RF receiver

A separate embedded failure caused the concentrator to remain connected to the backend while receiving no meter readings. The concentrator was connected by UART to a neighboring RF receiver. That receiver handled the antenna, converted RF frames into messages, and sent them over the serial connection [32:25].

The problem appeared after the system had been operating for weeks. The concentrator stopped detecting meters, did not recover on its own, and continued reporting to the backend, making it appear alive. A controlled reset of the concentrator restored operation, which suggested that the problem was in the concentrator’s software path rather than in the hardware itself.

The first workaround reset the software after one hour of silence. Kamma later reduced the threshold to four minutes, which reduced the visible outage and made customers happier, but he recognized that this only contained the symptom [33:56]-[35:07].

Months later, log analysis revealed that the RF receiver—not the concentrator—had reset. The receiver then emitted data that left the concentrator’s UART driver out of alignment. The receiver continued sending valid frames afterward, but the concentrator could not recognize them correctly.

The root fix was to reinitialize the UART when silence was detected instead of resetting the entire software unit. A full reset had worked only because startup reinitialized all drivers. Reinitializing the UART alone restored the relevant state with a smaller interruption [35:16]-[36:12]. This episode illustrates the difference between a temporary recovery mechanism and a diagnosis of the actual failing component.

Turning monitoring into a proactive process

At first, the team learned about failures from customer complaints. Every few days, someone would report a unit that was not working. Engineers would inspect the event service and backend, usually fail to understand the cause, and reset the unit. Kamma said that resetting solved the problem about 99 percent of the time, but it did not explain why the problem occurred [36:20]-[37:14].

He changed the process by actively examining event data for patterns. The existing interface showed only about 20 lines and was uncomfortable for broad analysis. A student working nearby wrote a script to retrieve all events and save them to a local text file. Kamma then wrote a Python script to parse the text and export it as CSV, allowing the data to be sorted, filtered, and graphed in Excel [37:22]-[38:20].

This made longitudinal analysis possible. For hourly transmissions, the team could calculate gaps over days, weeks, or months rather than checking only a short screen view. They could also graph periodic memory-usage reports over long periods and identify gradual changes that were not obvious in a short test [38:32]-[39:39].

Kamma initially ran the analysis manually for about ten minutes every morning. When he found a suspicious unit, he involved support and asked for customer contact, then used the information to validate software changes. As the number of devices and monitored parameters grew, individual graphs no longer scaled to tens, hundreds, or thousands of units.

The solution was to produce one summary row per unit. The summary made outliers easy to spot: a unit running a different firmware version, a device with 87 errors while others had none, or a unit whose last-seen timestamp was nine hours older than the rest [40:12]-[42:02]. Those indicators did not always explain the cause, but they directed engineers to the unit and its detailed logs. In the reported examples, long last-seen gaps were usually caused by power or network problems.

The process was later automated to run overnight and email summarized results in the morning. It remained only partly automatic because a person still reviewed the reports and contacted support when necessary [42:14].

Replacing the manual release process with continuous tests

The previous release process consisted of a 100-page manual test document. Three engineers had spent six months on the earlier release effort [42:33]. Kamma concluded that manually repeating that process alone was not practical, so he converted the manual tests into automated tests, removing some and modifying others. The initial rewrite took about a week, with a student team implementing the tests under his direction [43:12]-[43:45].

The team created two test pipelines: one for short tests that completed in minutes and another for long tests that ran for hours. This kept long-running tests from blocking quick feedback. They also ran the pipelines continuously, 24 hours a day, seven days a week.

The purpose of repeated execution was not to discover a different logical result each time. It was to prove that the test and product were stable rather than flaky. If a test failed even once in a way that indicated flakiness, Kamma removed it from the pipeline and returned it to the student for further work. The goal was that a green result would reliably mean green and a red result would reliably mean red [44:09]-[45:05].

For system testing, approximately ten devices ran the load simulator with aggressive configuration continuously, while another ten ran normal configurations continuously. Monitoring checked more than error conditions. It also verified positive behavior, such as whether an operation expected once per hour continued to occur at the correct rate over long periods [45:16]-[46:01].

Results and broader engineering lessons

The reported results were achieved after eight months: no resets and support for up to 10,000 water meters instead of 5,000 [46:10]. After two years, the project had automated tests and partly automated monitoring. Releases took two to three months rather than the previous three-engineer, six-month effort, and the team could add features and improvements that had previously been too risky to introduce into an unstable product [46:24]-[46:48].

Kamma’s closing principles were to measure everything, avoid assumptions, use simulators early, split large messages, prefer fixed-size storage where appropriate, decouple logic from network operations, and avoid unnecessary threads. He emphasized that robust software should control its behavior even when inputs and the environment are unpredictable. It may be acceptable not to complete a requested operation when resources are unavailable; it is not acceptable for the device to crash [46:57]-[48:19].

The question-and-answer session adds important context. Hardware could not be upgraded because this was an already-deployed product installed at thousands of sites. Kamma argued that the issue was not simply that 32 MB was too little; the system had originally been written and tested for roughly 1,000 meters and had never been validated at the much larger scale later sold to customers [48:36]-[51:46].

He also said he deliberately did not introduce fixed-size memory pools or refactor the entire application. Such a solution might have been valid, but it would have required more code and more changes to a working deployed system. He preferred targeted improvements that minimized the amount of code touched [56:30]-[58:08]. Management support came largely from the customer and business pain: the original team was frustrated, the next generation was already in development, and Kamma first made clear that he could not safely add features until the legacy product was stable [58:08]-[1:00:37].

Key Takeaways

  • A hardware capacity calculation is not the same as demonstrated system capacity. The concentrator was judged capable of supporting 7,500 meters, but its software had been proven at much smaller loads.
  • Instrument the system before committing to a diagnosis. Allocation rate, live and maximum allocation counts, total bytes, and per-size allocation data exposed that minor leaks and fragmentation were not the main failure.
  • A small simulator embedded in the real processing path can be more useful than a sophisticated external test setup. Duplicating real messages and changing only the meter ID reproduced the 6,000-meter crash with about ten lines of code.
  • Reproducibility makes debugging actionable. A conditional breakpoint on a null pointer allowed the failure to be inspected before the reset corrupted the evidence.
  • Peak memory, not just total RAM, was the bottleneck. Two concurrent 5 MB message constructions required about 10 MB at once; serializing them reduced the peak to about 5 MB.
  • Message size must be controlled at scale. Batching readings into groups of up to 1,000 meters prevented one large JSON/encryption/compression operation from consuming the available memory.
  • Fixed-size or static storage can eliminate an entire class of dynamic-allocation failures when the relevant message size is bounded.
  • Critical data should be stored in nonvolatile memory and deleted only after an acknowledgment. This made network outages survivable and limited loss to the collection interval under the stated hourly example.
  • A recovery timeout can hide a failure without fixing it. The four-minute silence reset reduced customer-visible downtime, but UART reinitialization was required to address the actual frame-alignment problem.
  • Monitoring must look for expected behavior as well as errors. Missing hourly transmissions, abnormal error counts, firmware mismatches, and stale last-seen timestamps provided useful early warnings.
  • Long-running tests are valuable for detecting instability and flakiness, not merely for checking whether a test passes once.
  • Targeted changes can be preferable to a large rewrite in a deployed legacy system. Kamma chose limited code changes over fixed-size pools or wholesale refactoring because implementation time and regression risk mattered.

Practical Notes

A reproduction workflow based on the talk starts with the real constraints: fixed RAM, real message formats, actual processing queues, meter-count targets, and production-like reporting intervals. The simulator should be inside the application or directly connected to the same queue used by real messages. The described switch was a file-existence check on the Linux device, allowing simulator mode to be enabled without creating a separate build.

For allocation instrumentation, overload C++ `new` and `delete`. The described implementation allocates the requested size plus four bytes, stores the requested size in the prefix, updates the profiler, and returns the user pointer after the prefix. On deletion, move back four bytes, recover the stored size, update the profiler, and release the original pointer. Record both current and maximum values, and group statistics by allocation size so that leak investigation and fragmentation analysis use evidence rather than a general impression of “high memory use.”

Use production-shaped messages and progressively aggressive rates. In this case, real messages contained an opcode, meter ID, and measurement value. Fake messages kept the same structure, received unique IDs, and entered the normal queue. The simulator multiplied real traffic by up to 100 and was used to find the load at which the unit failed. The talk’s example rates were deliberately more frequent than normal: hourly work was run every five minutes, ten-minute work every four minutes, and five-minute work every two minutes.

When peak memory is the problem, the sequence used here was to remove unnecessary concurrency in message construction, split large messages into batches, reduce the representation used by the message path, and replace dynamic buffers with a static array. Serializing the work sacrifices parallel message preparation, but the talk does not quantify a throughput cost; its measured benefit was lower peak memory and more than twice the supported meter count.

For reset diagnostics, preserve the existing application log and send a short tail of it after startup or reset. The described command was `tail -n 100`, sent through the existing event service. Sending it unconditionally simplified the implementation. The result is useful context, not a complete diagnostic record, so the speaker treated it as a pragmatic 80/20 solution rather than a perfect crash-reporting system.

For network reliability, separate the logic thread from the sending thread. The logic side creates data at the required schedule and writes it to a nonvolatile queue. The sender retries the oldest item, removes it only after a successful acknowledgment, and waits before retrying timeouts or bad acknowledgments. Do not make correctness depend on a network-status check performed before sending; the status can change immediately, and a functioning local connection does not guarantee that the backend is reachable.

For serial-input recovery, distinguish a full software reset from reinitializing the affected driver. The initial one-hour silence reset and later four-minute threshold were containment measures. Once logs showed that the adjacent RF receiver had reset and disrupted UART framing, reinitializing the UART restored operation with a smaller interruption than resetting the entire unit.

For monitoring, export machine events into an analyzable format. The workflow described was event retrieval to text, a Python transformation to CSV, and Excel for sorting, filtering, gap calculation, and graphs. Individual graphs worked for a few units, but summary rows were more effective at production scale. Useful summary fields included firmware version, error counts, periodic memory reports, and the last time the backend saw each unit.

For testing, separate short and long-running pipelines so that hours-long tests do not block minute-scale feedback. Run both normal and aggressive simulator configurations continuously. Treat a flaky test as a test defect: remove it from the trusted pipeline after a single questionable failure, fix it, and reintroduce it only when stable. Monitoring should check positive behavior, such as whether hourly operations continue to occur, not only whether an error was logged.

Several boundaries are important. The hardware was fixed and could not be upgraded. The speaker did not use fixed-size allocation pools because he considered them too much code for this deployed product, and he did not perform a wholesale refactor. The four-minute UART silence threshold was initially only a nicer failure mode, not a root-cause fix. The monitoring process was only half automated because a person still reviewed summaries and involved support. Finally, the nonvolatile queue’s stated loss guarantee assumes the hourly collection example and deletion only after acknowledgment; the talk reports no significant data loss after implementation but does not claim that every possible failure can be eliminated.
