Spec Kit presents Spec-Driven Development as a future approach for software engineering.

NDC Conferences · 2026-07-06T09:01:30+00:00 · Is Spec Kit the future for software development - Johan Smarius - NDC Copenhagen 2026

What This Video Is About

This video presents Spec-Driven Development (SDD) as a potential paradigm shift in modern software engineering, contrasting it with current trends like agentic or "vibe coding." The central argument is that relying too heavily on low-level technical implementation details—whether by human developers or AI agents—leads to poor long-term maintainability because the focus drifts away from understanding what the product is supposed to do. SDD advocates for prioritizing a deep, machine-readable understanding of the *product* through detailed specifications before diving into code generation.

The presentation introduces Spec Kit, an open-source toolkit designed to operationalize this spec-driven approach. By using structured markdown requirements, developers can leverage AI agents to guide the entire software development lifecycle, from initial requirement gathering and planning to final implementation. While acknowledging that SDD is not a universal solution—especially for projects with highly defined business rules—it offers a rigorous framework for maintaining traceability between intended product behavior and actual code execution.

Detailed Walkthrough

The speaker begins by addressing current trends in AI-assisted coding, noting that while agentic environments are useful, they often lack documented requirements [2:51]. The traditional view that "the code is the documentation" is challenged because revisiting old code years later without clear intent leads to misinterpretations [3:05]. This problem is exacerbated when using AI agents; if nothing is explicitly stated, the agent guesses, which can be wrong [4:38].

The core critique of current AI approaches (agentic/vibe coding) is that they remain focused on low-level technical requirements, boosting implementation speed but failing to boost product understanding [6:02]. Spec-Driven Development aims to shift this focus back to the product. The speaker notes that SDD itself is not new, referencing older concepts like Rational Unified Process and CASE tooling, which historically failed to produce desired code [7:11].

Spec Kit formalizes this concept by using machine-readable specifications, specifically markdown files, which must be unambiguous [8:13]. The workflow involves five key phases: constitution, specification, planning, task placement, and implementation [18:50].

1. Constitution: This phase establishes the most important non-functional requirements for the project [19:26]. Examples provided include test-driven development, clean code, YAGNI (You Ain't Gonna Need It), usability, scope discipline, quality gates, and traceability [23:02]. A key consideration here is that this file acts as "memory" passed to the AI context on every call; therefore, it must remain manageable to avoid excessive token usage [24:51].

2. Specification: This phase focuses on functional requirements rather than architecture [24:28]. The speaker stresses the importance of adding sufficient detail and explicitly rethinking assumptions in prompts. For instance, when building a simple game like Pong, one must specify edge cases—such as what happens if the ball reaches the end of the field or how the start button functions—because AI agents will guess otherwise [25:31]. The process is circular, meaning developers can always loop back to earlier phases.

3. Clarification (Optional but Recommended): Before proceeding, a "clarify" command can be used to scan all requirements in the spec file for anomalies and ask clarifying questions, presenting preferred options for resolution [31:20]. The speaker advises rerunning this step until all assumptions are cleared out.

4. Planning: Once specifications are solid, technical details are supplied (e.g., choosing .NET 8 with Blazor WebAssembly). Running the "plan" command generates a detailed plan that incorporates quality gates and traceability based on the constitution [35:19]. The agent can make intelligent assumptions, such as replacing an initially suggested Xcode testing framework with XUnit because it is more appropriate for the chosen stack [35:50].

5. Task Placement & Implementation: The final phases involve task placement, where the work is broken down into manageable tasks, and implementation, where code is generated [38:42]. The system can intelligently consider parallelism (e.g., adding multiple projects to a solution) to reduce manual overhead [39:57].

The demonstration shows that what might take a human developer a week of work could be accomplished in about an hour and a half using this process, though the speaker cautions that token usage will increase with iterative refinement [46:21]. Furthermore, Spec Kit supports "reverse engineering" by allowing users to generate specifications from existing brownfield codebases [49:03].

Key Takeaways

  • Spec-Driven Development (SDD) prioritizes product understanding over low-level technical implementation details.
  • Spec Kit is an open-source toolkit designed to support SDD workflows using machine-readable markdown specifications.
  • The core workflow consists of five phases: constitution, specification, planning, task placement, and implementation.
  • The Constitution phase defines non-functional requirements (e.g., clean code, test-driven development) that act as project memory for the AI agent.
  • Functional Specifications must be highly detailed to prevent AI agents from making incorrect assumptions about game logic or application behavior.
  • The "clarify" command is a crucial step to proactively identify and resolve ambiguities within the requirements before coding begins.
  • Planning generates structured outputs like project architecture, quality gates, and complexity tracking based on initial constraints.
  • Spec Kit supports both greenfield projects (starting from scratch) and brownfield projects (reverse engineering existing code).
  • While powerful, this process consumes tokens, especially during iterative refinement cycles; token cost must be weighed against manual labor time saved.

Practical Notes

Implementation Steps:

  • Install Python [13:10].
  • Install the specify CLI tool [13:18].
  • Initialize a project directory using `specify init` and select an AI agent (e.g., Copilot, Claude) [15:15].
  • Define non-functional requirements in the Constitution phase by writing markdown files [20:12].
  • Detail functional requirements in the Specification phase, explicitly defining edge cases and user interactions [25:31].
  • Run the `clarify` command to check for anomalies in the specification [31:20].
  • Supply technical constraints (e.g., ".NET 8 Blazor WebAssembly") during the Planning phase [33:11].
  • Execute the `plan` command to generate a detailed project plan, including quality gates and structure [34:15].
  • Run the final implementation command to generate source code based on the validated plan [42:38].

Architecture Choices:

  • The process supports defining architectural styles such as Clean Architecture [33:44] or specifying frameworks like Blazor and Maui [33:23].
  • Testing strategies can be specified, including XUnit for unit testing and Playwright for end-to-end validation [33:50].

Risks/Caveats:

  • The process is not suitable for all applications; it struggles in environments with extremely clear, rigid business rules.
  • Token usage scales significantly with the complexity and number of iterative refinement cycles performed on the specifications [51:38].
  • Customizations require understanding how to hook into the existing markdown processes, which can be complex when merging changes from new Spec Kit versions [50:54].
