Google DeepMind podcast discussion on large-scale AI agent societies and their safety challenges.

Google DeepMind · 2026-06-23T15:48:06+00:00 · When millions of AI agents meet

What This Video Is About

This episode of the Google DeepMind podcast is a conversation between Hannah Fry and Nenad Tomašev, a senior staff research scientist at Google DeepMind, about the shift from large language models as question-answering systems to AI agents that can observe environments, plan multi-step tasks, use tools, and take actions. The discussion starts with the practical difference between chatting with a language model and delegating work to an agent, then expands into what happens when many agents interact with each other at scale.

The central point is that agents are not just more convenient interfaces for language models. They introduce autonomy: the ability to execute chains of actions, use external tools such as email or web browsers, make bookings, write code, schedule experiments, or delegate subtasks to other agents. That autonomy creates new productivity opportunities, especially in software development and scientific workflows, but it also creates new failure modes because mistakes can have real-world consequences.

A major theme is that individual agent safety is not enough. As agents become numerous, they may form something like an “agentic economy,” where personal assistants negotiate, bid, delegate, verify, and coordinate with one another. This raises problems familiar from human institutions and financial markets, such as fairness, trust, reputation, collusion, flash crashes, and regulation, but with new AI-specific complications such as prompt injection, dynamic cloaking, cognitive monoculture, and correlated failures.

The conversation argues for a cautious, layered approach: humans must remain meaningfully in the loop, agents need verifiable contracts and permission boundaries, web environments need trust and certification mechanisms, and distributed agentic systems may require new forms of alignment beyond aligning a single model.

Detailed Walkthrough

The podcast opens by distinguishing the older idea of an AI assistant from the emerging idea of an AI agent. Until recently, an AI assistant usually meant a large language model: the user asked a question, and the model produced a response. It could not independently perform tasks on the user’s behalf. Fry frames the current shift around AI agents, mentioning open-source tools such as OpenClaw and Google tools including Gemini Spark and “anti-gravity” as examples of a new agentic generation [0:00].

Tomašev explains that agents are not a new concept in AI. Before modern large language models, AI researchers built agents that operated in simulated 3D environments, collected items, and completed tasks. The key conceptual distinction is that an agent observes the state of a world or environment and takes actions within it, whereas a language model simply continues or replies to a prompt [1:18]. Modern agents often use large language models “under the hood,” but the difference is the surrounding harness: the system takes proposed model outputs and turns them into actions in an environment [2:05].

That harness is what gives agents their practical value. A user could theoretically complete many agentic tasks manually by repeatedly prompting a language model and carrying out each step. An agent automates some of that back-and-forth. For a multi-step task, it can make a plan and execute actions, while still requiring approval or human input for sensitive or risky steps [2:22]. The user’s role shifts from typing every instruction to reviewing, deciding, and approving.

Fry uses wedding planning as a concrete example. A language model might suggest caterers or venues, but the human would still need to send emails and do the organizing. An agent, by contrast, could be given access to tools such as Gmail and permission to draft or send messages [3:48]. Tomašev emphasizes the caveat: the agent might send something wrong, so humans still need to verify what it composes [4:06]. In principle, tool access empowers the language model to perform tasks rather than merely describe how to perform them.

The first major practical area discussed is coding. Tomašev says the field is focusing heavily on coding capabilities because many formal processes can be expressed as software or code [4:42]. Coding agents are already being used internally and externally, and they are accelerating software development by shifting human attention toward ideas and design rather than boilerplate implementation [5:04]. Language models can now handle many repetitive implementation details that used to require time, skill, and bespoke knowledge.

However, Tomašev rejects a simple distinction between what agents “can” and “cannot” do. The problem is reliability. Even when an agent can perform a task, it does not do so with 100% accuracy [5:28]. Every action has a failure rate, and the more complex the action, the higher the expected failure rate. Some mistakes will be obvious; others will be subtle. This leads to automation bias: if an agent completes one task correctly, then another, the user may begin to trust it too much, stop checking, and miss important errors [6:19]. Tomašev stresses that keeping humans in the loop is not enough if those humans are disengaged. They must remain attentive and actively verify outputs [6:49].

The conversation then broadens to the long-term significance of agents. Tomašev says it is difficult to imagine a world without “deep disruption” from AI entering fields where it was previously absent [7:01]. Scientists are already using AI regularly, and fields such as mathematics, where AI had recently seemed implausible as a serious contributor, are changing quickly. The unsettling part is the speed of transition: unlike earlier industrial revolutions, society may not have as much time to adapt [7:38].

When Fry asks why build agents at all, Tomašev gives a broad motivation: advancing science, improving health, and improving human welfare [8:24]. He acknowledges that some people argue against granting AI systems autonomy. His response is conditional: if the field can develop safe harnesses that let agents perform complex tasks autonomously, then progress can accelerate because more can happen with the same amount of human input [8:56].

Science becomes the key example. Today, researchers already use language models as collaborators, for reasoning, or for formal derivations, but automating science more substantially requires more than short bursts of reasoning inside a model context window [9:41]. Tomašev points to autonomous research laboratories as a developing area. In such labs, agents might schedule experiments, run them, observe results, analyze feedback, and close the loop [10:13]. The risks are different from software. In software, tests can verify whether code works. In physical science, feedback often requires real-world experiments. If the domain is material design, batteries, or biotech, an agent’s proposed experiment could overheat equipment, damage hardware, or create other consequences [10:22]. Reliable protocols and safeguards are therefore essential.

Fry raises the question of the human role if agents can autonomously test ideas. Tomašev says current systems are not AGI and still have many limitations [11:30]. He characterizes current models as strong at “combinatorial closure” over things humans already know how to do: they are trained mostly on human data, so they can replicate, recombine, and bridge small gaps between human skills [11:45]. But he says they have not yet shown deeply transformative scientific discovery of a kind no human would have thought of [12:10]. That leaves a major role for humans.

The conversation then asks why agents have only recently become broadly usable. Tomašev notes that older narrow agents have existed in areas such as data center optimization, and trading or investment algorithms could also be called agents in some sense [12:24]. The difference now is language. Earlier agents could act in a narrow domain but offered no natural way for humans to interact, communicate, steer, or learn from them. Modern agents are based on language models, so people can talk to them and influence them [13:17].

What still prevents the vision of a universal personal assistant from arriving? Tomašev says the field needs to move beyond designing underlying models and toward coordination, orchestration, and management [13:31]. If users have powerful systems that can do many things, they become managers of agent teams or institutions. Managing agents differs from managing humans because agents make nonhuman mistakes and do not deeply know the user’s preferences. The user still needs to be involved [14:01].

This leads into trust and hallucination. Fry notes that language models still hallucinate, so trusting agents to act on one’s behalf is a major leap [14:39]. Tomašev responds that trust is both given and earned. He describes frameworks that track reputation over time: if an agent is repeatedly unreliable, it should not be trusted; even if it is mostly reliable, it should not be blindly trusted [14:59]. Because language models will hallucinate to some extent, workflows must be designed so hallucinations do not compromise the task [15:23].

Delegation is the next major topic. Tomašev describes complex tasks that may need to be broken into subtasks that no single agent can perform alone [15:46]. An agent may hand off work to another agent through an established agent-to-agent protocol. But delegation introduces failures: the delegating agent or human must manage, handle, and ideally preempt them [16:26]. Preemption includes identifying which agents are reliable, what capabilities they have, whether those capabilities can be certified, and how to guard against malicious interactions [16:57].

Fry returns to the event-planning example. Managing a large event involves bookings falling through, delays, and things not arriving on time. Tomašev says an agent delegating to a group of agents would face similar coordination problems [17:17]. He distinguishes true delegation from mere parallelization. Many current multi-agent systems simply split work into chunks and run agents in parallel, which may speed things up if each subtask is independent and reliable [17:47]. But that is not the same as an intelligent delegation framework with structured task assignment, communication, verification, and recovery [18:20].

Verification becomes a central design issue. In software, unit tests can check that code performs a specified function in isolation [18:40]. In real-world tasks, verification may be subjective or hard to formalize. Fry gives the example of choosing “nice tasting wine” [19:04]. Tomašev connects this to reward hacking: a system might technically satisfy a request while violating the spirit of the request [19:17]. This makes it important to define contracts between delegator and delegate as formally as possible and to emphasize verifiability [19:34]. Some tasks are reversible and can simply be retried. Others spend money or take actions that cannot easily be revoked, requiring more care [19:45].

The conversation then reverses the usual delegation direction: agents delegating to humans. Tomašev draws from his background in medical AI. In medical imaging and radiology, narrow machine learning models can identify pathology in scans, draw boxes around suspicious regions, and hand results to a human radiologist for review [20:33]. Such systems can perform at very high levels, sometimes superhuman on narrow tasks, but still have failures and require expert review [21:04]. Researchers have explored AI-human teams in multiple directions: humans consult AI when uncertain, humans review AI suggestions continuously, or AI systems make predictions and defer to humans when uncertain [21:15]. Tomašev says this setup, where AI defers to a human in cases of uncertainty, has worked well in narrow domains [21:58]. For general agents, it similarly makes sense for AI to recognize when sensitive actions require human approval [22:31].

The safety discussion then turns to cybersecurity and “agentic traps.” Tomašev calls the topic both scary and fascinating [22:47]. At scale, if individual interactions are not completely reliable, a system with many interactions will statistically fail [23:10]. Agents operate in an environment, and for web agents that environment is the web. If the environment is poisoned, agents may encounter traps laid by malicious people or malicious agents [23:46]. These traps can compromise the agent’s behavior.

Fry gives the example of a wine-buying agent visiting a merchant website that contains prompt injection designed to change the agent’s goals [24:11]. Tomašev confirms this is one possible attack. A key issue is that web pages contain elements not visually rendered to humans. If an agent consumes the raw page format rather than pixels as a human would see them, it may ingest hidden tokens that cause it to do something unintended [24:30]. Another attack is dynamic cloaking, where a site displays different content to humans and agents. A site can infer from behavior whether the visitor is human or agentic, then alter content only for agents with particular intentions in order to induce jailbreaking [25:05].

The risk escalates when agents have access to wallets or payment tools. Tomašev says people experimenting with agents have already encountered issues when giving them wallet access [25:35]. Early prototypes may run in trusted environments, but once agents operate on the open web, attackers have more incentive because the surface area is larger [25:53]. He notes that agentic use of the web may be exceeding human use for the first time, with agents both generating and consuming web content [26:23].

Mitigation requires more than making the agent itself safer. Tomašev compares the problem to older web security problems: computer viruses, malicious attachments, untrusted pages, and the need to certify resources [27:10]. Machine learning has also long faced adversarial examples, where changes imperceptible to humans can affect model behavior [27:33]. Agents can be attacked through subtle image or encoding changes, including small pixel changes or modifications to least significant bits [27:43].

Tomašev’s answer is “defense through depth” [28:15]. There will not be one solution. Safety requires layered mitigations: certify and test web page content, establish trust for resources, build agent-side mitigations, build model-side mitigations in the underlying foundation models, keep meaningful human controls, and carefully restrict permissions so that even a jailbroken agent can do minimal damage [28:27].

The conversation then moves to “formal agentic economies.” Tomašev imagines personal assistants with persistent memory of users, knowledge of their preferences, and some delegated agency and budget [29:31]. Such assistants could negotiate on behalf of users, forming localized economies. Fry asks how this might work for scarce goods such as concert tickets, with many agents rushing to buy at once [30:15]. Tomašev says the outcome is a design choice. If equal access is desired, each agent could be given the same budget across repeated auctions, and each agent could allocate that budget based on its user’s preferences, schedule, travel constraints, and desire to attend [31:04]. The goal would be fair outcomes at population scale, though the transcript does not claim such a system already exists.

Fry then compares the risk to high-frequency trading and stock-market flash crashes [32:35]. Tomašev says financial markets have already dealt with similar risks and offer mitigations to learn from [32:52]. But agents introduce a specific issue: a small number of highly represented language models may underlie many agents, including models such as Claude, GPT, Gemini, and open-source models [33:19]. These models may have similar opinions and take similar actions. Tomašev calls this “cognitive monoculture” [33:38]. If hundreds of thousands or millions of artificial decision-makers behave similarly, failure points become correlated [33:49]. One mitigation is to diversify agent decision-making, potentially through system prompts or other design choices, though most users do not currently customize agents that way [34:04].

Related risks include agentic groupthink and collusion [34:34]. In auctions, groups can coordinate bids to gain advantage. With agents, the issue may be harder to detect because they could coordinate indirectly through the environment rather than by communicating directly [34:36]. Tomašev says anti-collusion measures will be needed [35:01].

The discussion then reflects on why careful, slow releases may be necessary. Tomašev compares agents to self-driving cars: impressive demos came long before safe deployment on streets, because the “last mile” takes most of the work [35:23]. For agents, technical solutions are not enough. Policy and broader societal understanding are also needed, because current systems are being integrated into human structures rather than operating as fully autonomous agentic economies [35:52].

The final conceptual shift is from individual generalist agents to societies of specialists. Tomašev says multi-agent systems become most interesting when agents can do different things in different ways, not merely parallelize the same work [36:41]. He argues that although researchers aim for general and capable agents, the economic endpoint may not be one giant general model. Specialists may be faster, cheaper, more reliable, and easier to certify for specific skills [37:21].

He uses chess as an example. Gemini and other models can play some chess, but for chess one would still use a chess engine because it is faster, more accurate, cheaper, and focused on one task [37:50]. Humans are similar: no single human can do everything humanity can do [38:18]. AGI is often framed as human-level intelligence, but people sometimes imagine it as “humanity-level intelligence,” meaning the combined abilities of all humans [38:21]. Tomašev suggests a future with a general orchestration layer as connective tissue, calling specialized models for specific tasks [38:54]. Fry describes this as distributed intelligence rather than a single AGI-like entity [39:42].

This has major safety implications. Tomašev says alignment approaches today often focus on one model: observe its behavior and align it with what is permissible, preferable, or desirable [40:05]. But if 10,000 agents interact in complex ways, it becomes hard to define or align “the system” [40:31]. Agent A may interact with B today, C tomorrow, while C subdelegates to D and D consults a human [40:43]. Economic incentives may provide one starting point for aligning distributed agentic societies, but only if agent economies are designed so that profit maximization does not cause harm [41:00]. Individual agent safety remains a prerequisite, but more work is needed on safeguarding groups of agents [41:23].

Fry closes by emphasizing that focusing on a single agent misses the bigger picture. The more consequential future may be an agentic society of specialists, generalists, delegators, and detail-focused agents. She frames the lasting idea as a shift from replicating “human-level intelligence” to possibly replicating “humanity-level intelligence” [41:44].

Key Takeaways

  • Agents differ from ordinary language models because they act in an environment. A language model replies to a prompt; an agent observes a state of the world, plans, and takes actions through a surrounding harness, often using a language model to formulate those actions.
  • Tool access is what makes agents practically powerful and risky. Giving an agent access to Gmail, browsers, booking systems, wallets, or lab equipment allows it to complete work, but also means errors can have real-world consequences.
  • Coding is currently one of the strongest use cases. Software tasks are relatively formal, can be expressed as code, and can often be verified with tests. This makes coding agents useful for reducing boilerplate and accelerating implementation.
  • Human oversight must be active, not symbolic. Agents have failure rates. Automation bias can cause users to stop checking once an agent has performed well several times, allowing subtle mistakes to slip through.
  • Scientific agents require physical-world safeguards. In science, closing the loop often means running experiments, not just passing tests. Domains such as material design, batteries, and biotech require protocols to prevent damage or unsafe outcomes.
  • Delegation is more complex than parallelization. Many current multi-agent systems simply split work into chunks. True delegation requires choosing reliable agents, defining contracts, verifying results, handling failures, and managing dependencies between subtasks.
  • Verifiability is central to safe delegation. Software can often be checked with unit tests, but real-world tasks may involve subjective goals, such as choosing “nice” wine. Poorly specified goals create reward-hacking risks where an agent satisfies the letter but not the spirit of a request.
  • Agents may need to delegate decisions back to humans. In medical imaging, narrow AI systems can make predictions and defer uncertain cases to human experts. General agents should similarly seek human approval for sensitive or uncertain actions.
  • The open web is a hostile environment for agents. Agentic traps, hidden prompt injections, dynamic cloaking, adversarial inputs, and malicious websites can manipulate agents, especially if they consume raw web content rather than only human-visible pixels.
  • Safety requires defense through depth. No single guardrail is enough. Mitigations should include trusted resources, content testing, model-side safeguards, agent-side safeguards, human controls, and strict permission limits.
  • Large-scale agent economies could create market-level risks. Millions of agents bidding, negotiating, or trading could produce fairness issues, flash-crash-like dynamics, collusion, correlated failures, and the need for anti-collusion and market-design measures.
  • Distributed agent societies change the alignment problem. Aligning one model is different from aligning thousands of interacting agents that delegate, subdelegate, and coordinate dynamically. Individual safety is necessary but not sufficient.

Practical Notes

A practical agent system should be designed around the distinction between reversible and irreversible actions. For low-risk tasks, such as drafting text or running a local test, an agent can be allowed more autonomy. For irreversible or costly actions, such as sending emails, spending money, making bookings, changing production systems, or interacting with physical equipment, the system should require human approval or stronger verification.

For coding workflows, the transcript supports a test-centered approach: agents can write code, but verification should rely on unit tests and other checks before accepting changes. This is one reason software engineering is currently a strong agent domain: the feedback loop can be automated more easily than in physical science.

For scientific workflows, the implementation burden is higher. Agents that schedule or run experiments in autonomous labs need safeguards, reliable protocols, and controls around interfaces to real-world systems. Physical experiments can damage hardware or create safety risks, so the workflow cannot be treated like ordinary software execution.

For delegation, the system should track reputation and reliability over time. Agents that repeatedly fail should not be trusted, and even mostly reliable agents should still be verified. Capability certification is another decision point: before delegating a subtask, the delegator should know what the target agent is supposed to be good at and whether that claim is trustworthy.

Contracts between agents matter. The delegator should specify the task, success criteria, constraints, and verification method as clearly as possible. This reduces reward-hacking risk, where a delegate technically satisfies a request while violating the intended outcome.

Permissions should be minimized. If an agent is jailbroken through a malicious webpage or adversarial input, the damage should be limited by what it is allowed to access. The transcript specifically mentions wallet access as a danger area and recommends being mindful of permissions so that a compromised agent cannot cause large harm.

Web-facing agents need layered defenses. Supported measures include testing and certifying web content, using trust mechanisms for resources, adding mitigations on the agent side, adding mitigations to the underlying foundation model, keeping human override controls, and not relying on any single safety mechanism.

Agentic marketplaces or auctions require explicit design choices about fairness. For scarce goods such as tickets, one possible design discussed is giving agents equal budgets over repeated auctions and allowing them to allocate those budgets based on user preferences and constraints. The transcript presents this as a conceptual design choice, not as a deployed system.

Large-scale agent systems should consider correlated behavior. If many agents use the same or similar underlying models, they may make similar decisions, producing cognitive monoculture and correlated failures. Diversifying decision-making is a possible mitigation, but the transcript notes that most users do not currently customize agent behavior deeply.

For future architectures, the conversation suggests a likely role for specialist agents or models. Instead of one large model doing everything, a general orchestration layer could route specific tasks to cheaper, faster, more reliable specialists, similar to using a chess engine rather than a general language model for chess.
