Scott Hanselman interviews Zed founder Nathan Sobo about DeltaDB and rethinking version control for collaborative, agent-assisted programming.

Scott Hanselman · 2026-06-21T23:17:37+00:00 · The space between the Commits with Zed and DeltaDB's Nathan Sobo

What This Video Is About

Scott Hanselman interviews Nathan Sobo, founder of Zed, about why Zed is building DeltaDB: a version-control layer intended to capture the fine-grained process of programming that happens between Git commits. The conversation begins with editor history and technology monocultures, especially the path from Atom and Electron to Visual Studio Code, then shifts into the deeper question of whether Git captures enough of modern software development.

Sobo argues that Git remains valuable and is not being replaced. Git is good at snapshots: commits, branches, pack files, and a causal graph of repository states. But, in his view, software work now contains important context that Git does not model well: manual edits, abandoned edits, agent prompts, agent responses, code references inside conversations, parallel work trees, and collaborative review before code is turned into a pull request. DeltaDB is presented as a superset-like layer on top of Git that records and indexes the deltas between commits.

The reason this matters is that agent-assisted programming is making the “process” more important. A developer may spend hours steering an agent, correcting assumptions, asking questions, and shaping code before producing a commit. Git records the final snapshot, but not the conversation or the sequence of edits that led there. DeltaDB aims to make that intermediate history available for collaboration, review, navigation, and future agent context, while still raising difficult privacy and surveillance questions that Sobo says Zed is actively thinking through.

Detailed Walkthrough

Hanselman opens by introducing Nathan Sobo as the founder of Zed and says Zed is working on “something amazing called Delta DB” [0:13]. Before getting into DeltaDB, they frame the discussion around technology monocultures. Hanselman compares browser consolidation around Chromium with editor consolidation around Visual Studio Code, asking Sobo to walk through the history from Atom to VS Code [0:42].

Sobo explains that Atom was created at GitHub by a small team and was originally a passion project of GitHub founder Chris Wanstrath [2:16]. Atom came from Sobo’s desire to combine things he liked from different editors into one environment, though he says Atom itself never fully achieved what he personally wanted [2:38]. In building Atom, the team also created Electron, which Sobo describes as the shell around Atom: effectively a way of “dressing up a web browser as if it’s a desktop app” [2:53].

He then connects this to Visual Studio Code. Microsoft had been working on browser-based Visual Studio tooling, including the Monaco editor, and adopted Electron in the creation of VS Code [3:14]. Sobo says VS Code took over through feedback loops: extensions went there, users went there because everyone else did, and the ecosystem reinforced itself [3:54]. He is clear that he used VS Code himself while writing Zed because of the tooling integration, but he did not like spending his day inside “a web browser dressed up as a desktop app” [4:17].

That dissatisfaction led to Zed. Sobo says he believed he could build something he would be much happier using, but doing so required a company, resources, and a business model [5:26]. The Zed approach involved “reinventing the whole stack” that Atom had built on top of last time [5:42]. His criticism of Electron is that it consumes a lot of RAM, carries legacy web-platform baggage, and does not expose the level of control needed to build what he considers an exceptional editor experience [5:59]. For Sobo, the editor remained fundamental: even in an agent-assisted future, he still expects to need a user interface for interacting with software and architecture [6:43].

Hanselman then asks about building for oneself versus building by consensus [7:32]. Sobo says Zed is not exclusively for him, because his team and the community matter, but he sees himself as “user zero” [8:06]. The product starts with his own taste and standards, then expands into obligations to teammates, investors, and users [8:32]. He frames Zed’s mission as advancing the state of the art in developer tooling, not merely chasing an existing market [9:21].

The conversation then pivots to source control. Hanselman recalls a time with Source Depot, SourceSafe, CVS, SVN, Mercurial, and Git, then observes that Git has now become the standard taught in schools [9:51]. He asks how Zed’s thinking led to DeltaDB and whether it is challenging Git as the dominant “VHS” of version control [10:14].

Sobo immediately distinguishes DeltaDB from Zed’s fight against Electron-based editors. He says Git is modular and easy to build on top of, unlike Electron, where he felt there was no viable way to fix the foundational performance and architecture issues from within [10:43]. Git, by contrast, solves many problems well: taking snapshots of code, chaining them together in a causal graph, moving blobs around efficiently in pack files, and transferring bulk data [11:29]. DeltaDB is not presented as a rejection of Git’s snapshot model.

The gap Sobo identifies is that Git focuses on artifacts left behind by the programming process, not the moment of writing code itself [12:04]. He traces this concern back to his Atom days and his pair-programming background. In both Atom and Zed, he says, the team has valued talking about software while writing it [12:35]. The problem is that there are many versions of code between discrete commits, and current tools do not provide a stable, collaborative way to refer to those intermediate versions [13:05].

Hanselman paraphrases DeltaDB as versioning the process between Git checkpoints: abandoned edits, prompts, agent steering, decisions, and the “dark matter” between commits [13:50]. Sobo agrees and expands the point. DeltaDB does not just coordinate those intermediate states; it aims to make them available to multiple participants over different time scales, including real time and asynchronous collaboration [14:29].

Sobo describes the current status quo in agent-assisted development as one-on-one direct-message-style sessions between a developer and an agent [14:36]. The result of that private interaction is usually a Git snapshot pushed to GitHub, while the conversation that produced it is absent or perhaps attached afterward to a pull request by some tools [14:48]. DeltaDB’s alternative is to version the agent conversation, embedded work trees, and edits together continuously [15:14]. That makes the original conversation itself a place where other people can participate, either while the code is being written or later.

One concrete example is asynchronous handoff. A developer could interact with an agent, leave work trees tied to that conversation in the cloud, and then a colleague in another time zone could pick up the thread three hours later [15:45]. The work trees would mirror down to the colleague’s disk, and the colleague could ask the agent what the original developer was discussing [15:53]. Sobo contrasts this with a pull request body, where the conversation is separate from the active coding process [16:11].

Hanselman compares this to existing tools such as GitHub Copilot, Codex, Claude, and GitHub Chronicle, each of which has its own notion of sessions or storage [16:35]. Sobo’s point is that the conversation and the between-commit process are themselves valuable artifacts [16:52]. With DeltaDB, Sobo says, users should no longer have the problem of forgetting to push or commit changes in order to share the current state, because the conversation can carry the exact state of the code “down to the millisecond” for all participants [17:07]. In Git, a work tree is personal while the repository is collaborative; DeltaDB tries to make work trees collaborative artifacts [17:31].

Hanselman then raises the question of noise. If DeltaDB captures prompts, corrections, false starts, and all the conversational “stack tracing” of how a human and agent sculpted code, that could be useful, but it could also be slop [17:51]. Sobo acknowledges developer privacy and noise concerns [18:50]. He says the point of storing every edit is not simply to replay a developer’s day from beginning to end, though that could be built. The more important reason is indexing: the data is there for reference and can be used by humans or agents to orient themselves [19:06].

Sobo gives an example from internal use. When he returns to an agent conversation after being pulled away, he does not usually reread the whole transcript from the beginning. He asks the agent to catch him up, and the agent may refer to earlier moments [19:27]. The conversation becomes a persistent, multiplayer entity: someone can join it, read if they want, or ask the agent questions about what happened [19:59].

Hanselman characterizes DeltaDB as an “event sourced database for the act of programming,” which he describes as inherently social and multiplayer for many teams [20:21]. Sobo says DeltaDB stores a directed acyclic graph, like Git, of everything that happened, but with every event recorded and indexed [20:50]. This index enables a different kind of code navigation: a developer can go to code and understand the edit-level history and the conversation that created or referenced that edit [21:04].

Sobo explains that once DeltaDB can locate a character in one version, its causal understanding of subsequent changes lets it locate that character in later versions losslessly [21:27]. This is not based on “squinting through a diff”; DeltaDB knows what happened [21:43]. If code was deleted, it can figure out where it would be. This could support queries such as: in this file, over the past 30 days, which conversations with agents referenced this particular code? [21:59]. Sobo notes that so much data may be most useful to an agent that can query code and ask other context windows questions [22:14].

Hanselman asks what counts as a delta: one keystroke, a 500-line patch, or a formatter change [22:38]. Sobo answers that if a person is editing manually, one keystroke is a delta; if they paste 10 KB of text, that paste is also one delta covering the entire inserted unit [23:03]. Hanselman then asks how DeltaDB gets from character-level truth to intention-level history, such as “the authentication check was moved below the cache lookup” rather than “characters 400 to 526 were replaced” [23:22].

Sobo’s answer is that the low-level delta is valuable because it provides a stable anchor. If a developer or agent adds an annotation to source code, it can be anchored to a particular character, and that character was part of an insertion delta with a unique identity and offset [23:46]. Because DeltaDB indexes everything that happens after that delta, it can later locate the same logical position in another version [24:07]. For higher-level semantic moments, Sobo says Git commits still exist, and DeltaDB’s causal DAG includes the Git commit DAG as a subset [24:44]. Zed also plans the idea of “moments” or “way points” in agent conversations: lightweight named points in time that are not full snapshots but identify a fine-grained version [24:53].

This leads to one of the clearest product concepts in the discussion: portals. If an agent transcript references code, DeltaDB should let the user “step through that portal” to the exact spot in the code as it existed at the moment the agent made the reference [25:33]. There may be no Git commit for that instant; DeltaDB explicitly models the in-between versions as deltas [26:00].

Sobo then describes another capability that would be extremely hard with Git alone. After jumping to a historical point in code, the user could scrub a timeline while keeping the cursor in the same logical position and ask, “Where is that code now?” [26:19]. If the code was cut and pasted to another file, the view could move with it [26:31]. Hanselman points out that this is structurally different from Git: following a semantic chunk of code backward and forward across moves and modules is not something Git can do fluidly from snapshots alone [26:47]. Sobo agrees that Git can reconstruct some history from snapshots, but DeltaDB’s continuous tracking and indexing should answer these questions more directly [27:15].

Sobo emphasizes that DeltaDB should not require developers to change their Git-based workflow much [28:20]. Its capabilities are intended to be “transparently superimposed on top of git” [28:27]. Zed can watch the work tree on disk, so even external tools that edit files can be picked up and conflict-free diffed to someone else’s file system [28:35]. Developers can still commit and push as usual, including with agents pushing where desired [29:05]. Sobo calls DeltaDB a “superset of git” in the sense that it tracks commits and everything in between [29:15].

The conversation ends with privacy and workplace-surveillance concerns. Hanselman asks who gets to see the internal reasoning trail: the author, team, or company. He notes that developers may paste secrets, keys, or connection strings, and that there is a thin line between code provenance and a system that judges how someone spent their day [29:29]. Sobo says the current internal pre-alpha shares an entire conversation with the agent, which works but can feel uncomfortable [30:21]. He admits that even he feels insecure about sharing his first rough interactions with an agent [29:56].

The planned direction is finer-grained sharing controls. Sobo suggests being able to share from a particular moment in the conversation, hide some of the paper trail, or share after compaction or summarization [30:38]. The ideal is to capture everything securely and privately, then let users decide how captured context is revealed to peers [31:03]. Hanselman suggests that summarization could let some parts remain verbose while others are condensed, and Sobo agrees that having everything first and then taking away detail may lead to more fluid experiences than requiring every point to be manually chosen and summarized upfront [31:25].

Hanselman closes by directing listeners to zed.dev/deltadb, where they can request early access by providing an email and GitHub username [32:26]. He summarizes the promise as rewinding to any edit, branching to any moment, and tracing code back to conversations [32:34].

Key Takeaways

  • DeltaDB is not positioned as a Git replacement. Sobo says Git’s core model of snapshots chained in a causal graph is still sound, and Git remains good at efficient storage and transfer of repository states.
  • The target gap is everything between commits. DeltaDB aims to capture manual edits, pasted patches, agent conversations, code references, work-tree states, and other process-level artifacts that Git does not model.
  • Agent-assisted programming changes what version control needs to remember. When code is produced through an extended conversation with an agent, the transcript, prompts, corrections, and intermediate work-tree states may be as important as the final commit.
  • DeltaDB makes work trees collaborative. In Git, the repository is shared but the work tree is personal. DeltaDB’s goal is to let a work tree tied to a conversation be shared, mirrored, and continued by another participant.
  • The system records fine-grained deltas. A manual keystroke can be a delta; a 10 KB paste can also be one delta. The point is to preserve the causal history of how text entered and moved through the codebase.
  • Indexing is as important as storage. Sobo says the reason to store every edit is to index it, enabling later navigation, lookup, agent assistance, and reconstruction of code context without merely replaying everything linearly.
  • “Portals” are a central user-facing concept. If an agent references code in a conversation, a portal should let the developer jump to the exact code location as it existed at that historical moment.
  • DeltaDB can follow code through time more precisely than Git diffs. Because it records every change, it can potentially track a logical code position even if the code is moved, pasted elsewhere, or deleted.
  • Git commits still matter as semantic milestones. DeltaDB’s causal graph includes Git’s commit graph as a subset, and Zed also plans lightweight “moments” or “way points” for naming fine-grained versions without creating full commits.
  • The workflow is intended to stay Git-compatible. Developers should be able to keep committing and pushing normally while DeltaDB adds extra capabilities on top.
  • Privacy is an unresolved but acknowledged design challenge. Capturing everything risks exposing embarrassing drafts, secrets, or evidence that feels like workplace surveillance. Sobo says finer-grained sharing, summarization, compaction, and secure controls are part of the intended direction.
  • DeltaDB is still early. Sobo refers to an internal pre-alpha and Hanselman points listeners to an early-access request page at zed.dev/deltadb.

Practical Notes

DeltaDB is described as a layer built on top of Git rather than a separate version-control world. A team using it should expect normal Git concepts to remain in place: commits, pushes, branches, repositories, and work trees. The difference is that DeltaDB tracks additional causal history between commits.

The practical workflow described in the transcript looks like this: a developer works in Zed, possibly with an agent, while DeltaDB continuously records edits and conversation context. The developer can still commit to Git. Meanwhile, the conversation and embedded work trees can become collaborative artifacts. Another teammate may join the conversation later, mirror the relevant work trees to disk, ask the agent what happened, and continue work from the same state.

For agent workflows, the product direction is to avoid isolating each developer-agent session as a private DM whose only output is a pull request. Instead, DeltaDB treats the session itself as shareable and versioned. This can support review and collaboration earlier than a pull request, because teammates can enter the same context where the work is being produced.

For navigation, the important implementation idea is stable identity for inserted text. Sobo describes an inserted character as belonging to an insertion delta with a unique ID and offset. Because later edits are also indexed, DeltaDB can locate that same logical position across later versions. This enables features like portals from transcript references to historical code and timeline scrubbing that follows code as it moves.

For teams considering this kind of tool, the major decision point is privacy policy. The transcript supports several risks: accidental capture of secrets, exposure of rough or embarrassing agent prompts, and possible workplace-surveillance dynamics. Sobo says the current internal pre-alpha shares whole conversations, but the intended design includes finer-grained sharing, such as sharing only from a certain moment, hiding earlier paper trail, summarizing sections, or sharing after compaction.

The specific product and access path mentioned is Zed DeltaDB at zed.dev/deltadb. Hanselman says the page lets users request early access by entering an email and GitHub username [32:26].

No commands, configuration files, installation steps, or public API details are provided in the transcript.
