The talk covers Carbon's progress from an experimental C++-interoperable language toward practical adoption.

NDC Conferences · 2026-07-20T12:11:58+00:00 · Carbon: graduating from the experiment - Chandler Carruth - NDC Toronto 2026

What This Video Is About

Chandler Carruth reviews Carbon's progress since the project was announced four years earlier. Carbon is intended as an incremental successor to C++, not as a language that requires abandoning existing C++ systems. The project targets problems Carruth identifies in C++: decades of technical debt, slow and difficult language evolution, textual inclusion and long compile times, pervasive macros, weak generic and package systems, and the absence of built-in memory safety.

The talk demonstrates how Carbon is being designed to coexist with C++. A vending-machine example introduces Carbon syntax for classes, methods, references, access control, packages, and initialization. The example then imports C++ headers, calls C++ APIs, embeds C++ code directly in a Carbon file, and allows C++ code to derive from and call Carbon code. This interoperability is central to Carbon's proposed migration strategy for large, complex C++ codebases.

Carruth also presents a proposed memory-safety model. Carbon would have a permissive mode for mechanical migration and a strict mode that checks memory safety. The design tracks which abstract storage locations references point into, records invalidation effects such as vector reallocation, and models mutex protection and thread sharing through similar type-system effects. The goal is to make safety checking incremental while preserving valid existing patterns wherever possible.

The project is approaching a transition from research experiment to evaluation, but Carruth does not present Carbon as finished or production-ready. C++ interoperability is being implemented rapidly, yet Carbon 0.1 still needs a usable compiler and toolchain, build-system integration, most of the required interoperability, and enough evaluation feedback to identify blockers. Limited real-world production use is described as a later step.

Detailed Walkthrough

Why Carbon exists and what has happened so far

Carruth begins by restating Carbon's motivation [0:00]. He argues that C++ has accumulated more than 40 years of technical debt and that backwards compatibility is usually prioritized over substantial improvements. In his characterization, the standards process is complex, slow, inconsistent, and ineffective at making fundamental changes. The result is a widening gap between C++ and newer languages.

The specific problems he names include dependence on textual inclusion, unbearable compile times, pervasive macros, the continued influence of C conventions, the lack of a type-checked generic system, the lack of a package manager, and the absence of memory safety. Memory safety is presented not as a new concern but as an increasingly urgent industry problem that now represents an existential risk for C++.

The obvious solution would be to evolve C++ directly, but Carruth says that many people have attempted this for decades without achieving the necessary changes. Carbon is therefore being developed as an incremental successor: a language that can coexist with the world's existing C++ code while providing a path to migration. Its stated goals include adding memory safety to existing software, improving compile times by 10x or more, generating smaller and faster binaries, and serving large “brownfield” systems rather than only new projects.

The open-source project has been active since its announcement. Carruth reports more than 5,000 pull requests, more than 90 contributors, 179 merged language proposals, and 159 closed “leads” issues concerning difficult design questions [0:00]. He says the governance model is working openly and transparently. The project publishes weekly “Last Week in Carbon” summaries, has produced 12 editions of its bimonthly “Carbon Copy” newsletter, and maintains an active Discord community.

Carbon's basic language model

The first code example is a vending machine [5:00]. A Carbon class named `vending_machine` has a function that creates an instance from a 32-bit integer price and returns `Self`, meaning the class containing the function. Carruth describes this as a factory function that takes the place of a C++ constructor.

Carbon uses struct-literal-style values to initialize fields. Carruth compares their purpose with C++ list initialization using designators, while emphasizing that Carbon's form is more type-oriented and explicit about what is being initialized.

The example includes a `dispense_item` method. Unlike C++, where the object being operated on is normally implicit, Carbon makes the object parameter explicit. The method takes `self` by reference, which allows it to mutate the vending machine's item count. Another method, `print_profit`, takes `self` as an input parameter rather than a mutable reference. Carruth presents Carbon input parameters as a concise analogue to the read-only, const-reference style commonly used in C++.

Access control is also demonstrated. Carbon does not require an exported API to label every declaration `public`; public visibility is the default in an exported API file. A `private` marker is used when something appears textually in the API file but is not part of the public interface. Carruth argues that this makes APIs easier to read: a reader can stop considering declarations after reaching the private section. The vending machine's price field is used as the private member in the example.

The second part of the example introduces Carbon's package defaults [5:00]-[10:00]. If a file does not declare itself as a library in a named package, Carbon treats it as a program in the default `main` package. Such a program needs a `run` function. The example creates a vending machine, calls a method to obtain an item price, and prints the result. Carruth presents this as evidence that the basic pieces of a programming language—types, variables, methods, calls, and an executable entry point—are in place, before moving to the more important interoperability features.

Calling C++ from Carbon

The next section uses C++ interoperability to solve a limitation in the early Carbon example [10:00]. The code imports the C++ standard-library I/O stream header and accesses it through Carbon's `Cpp` package. Carruth explains that Carbon's own printing facilities initially handled integers but not strings. Rather than add an independent Carbon string system before solving interoperability, the project prioritized the ability to use C++ types and APIs, including C++ strings and streams.

Carbon uses dots where C++ uses double-colon namespace qualification. A C++ name such as `std::cout` can therefore be referenced through Carbon syntax with the namespace punctuation translated across the boundary.

The example initially uses the shift-left operator to write output through C++ streams. Carbon knows that C++ defines overloads for this operator and can invoke the corresponding C++ operation. However, Carbon's shift-left operator is mathematically a shift operation rather than a streaming operation, so Carbon does not support indefinitely chaining it in the same way C++ stream syntax does. Printing multiple values requires awkward parentheses. Carruth uses this limitation to motivate a better helper function.

Carbon supports inline C++ blocks [10:00]-[15:00]. The C++ code is written directly inside the Carbon source file as a multiline raw block string literal. The extra raw-string syntax means that backslashes inside the embedded C++ do not need to be escaped. Carbon treats the block as text and sends it to Clang; Carbon does not need to understand the embedded C++ grammar, tokenization, or implementation.

The example defines a normal C++ variadic template that uses a fold expression to stream multiple arguments and a newline. Once the inline block is closed, Carbon calls that helper through the `Cpp` package. This lets the Carbon program use a C++ implementation without requiring Carbon to reimplement C++ stream behavior or understand every detail of the C++ template.

A central interoperability rule appears here [15:00]. When Carbon calls a C++ API, selecting the API and resolving overloads uses actual C++ rules implemented by Clang. Carbon does not attempt to create an approximate second implementation of C++ overload resolution, which could produce ambiguity or behavior differences. However, the arguments originate in Carbon, so implicit conversions must still satisfy Carbon's conversion rules. If C++ overload resolution would normally request a conversion that Carbon considers unsafe, the Carbon call is rejected. This creates a deliberate division of responsibility: C++ determines which API overload is selected, while Carbon controls whether the argument conversions are permitted.

Calling Carbon from C++

Carruth then reverses the direction of the example [15:00]-[20:00]. The Carbon vending-machine type is made usable as a base class. Carbon types are final by default, so inheritance requires an explicit opt-in. Carruth presents this as an example of Carbon's preference for strong defaults: the common case requires no modifier, while inheritance is marked as an intentional exception.

An inline C++ block defines a C++ `snack` struct containing a price and a C++ `snack_machine` class derived from the Carbon vending-machine class. The C++ constructor initializes the Carbon base subobject by calling the Carbon factory function. A local C++ alias makes the otherwise long Carbon base type easier to reference. The Carbon namespace and member syntax are translated into the corresponding C++ form.

The derived C++ class adds a method that obtains a snack by calling the inherited Carbon `dispense_item` method. The example then reads the snack's price, prints it, and calls the inherited Carbon `print_profit` method. This demonstrates that C++ code can use Carbon members through normal C++ object access and inheritance patterns.

The example is run in Compiler Explorer [20:00]. Carruth says the code is compiled with the latest private Carbon compiler. The compiler at that point still expects a different method syntax—square brackets rather than the round-bracket form shown in the presentation—but he describes this as a purely syntactic difference. The code imports the libC++ standard I/O stream library, defines the helper and methods, derives a C++ class from Carbon, and compiles and runs.

Carruth characterizes C++ interoperability as being in a strong and rapidly improving state. He says that features were landing quickly enough that he had to update his examples during the week of the talk because functionality that did not work the previous week had become available.

Carbon's memory-safety goals

After the interoperability demonstration, the talk moves to memory safety [20:00]. Carruth says the industry has become more urgent about the problem, so Carbon began designing its safety model earlier than originally expected.

The design has three major goals. First, strict Carbon must actually be memory safe, with compile-time memory checks, spatial memory safety at runtime, safe initialization, null-pointer safety, and data-race safety. Carruth compares the intended level of protection with Rust and says memory safety is a non-negotiable requirement.

Second, the system should be smooth enough to represent existing C++ patterns that are already correct. A valid C++ design should be expressible in safe Carbon whenever possible, without forcing every program to be rewritten into a completely different style. The annotations and code changes should be introduced only where needed to establish the safety proof.

Third, the system must be incremental in two dimensions. Teams need to migrate only part of a C++ codebase at a time, and they also need to add safety modeling gradually rather than annotate everything before any checks can be useful.

Carbon therefore distinguishes between permissive and strict modes. Strict Carbon is intended to provide the complete safety guarantees. Permissive Carbon acts as an intermediate migration environment in which code can be mechanically moved from C++ and then gradually given safety contracts. Carruth emphasizes a particularly important property: adding strict checking must not introduce new undefined behavior, even when unsafe code elsewhere is incorrect. This property is needed for an incremental migration strategy.

Temporal safety through places and invalidation effects

Carruth uses use-after-free as the example for temporal safety [25:00]-[30:00]. In C++, a typical use-after-free requires four ingredients: an allocation, a pointer captured into that allocation, an operation that frees or reallocates the memory, and a later use of the now-dangling pointer.

The Carbon version uses a buffer type called `Buff`, serving a role analogous to `std::vector`. The buffer is initialized and allocates storage. A pointer `P` is obtained into that storage. A later `push_back` may reallocate the buffer, and the code then attempts to use `P`. In Carbon, the intended result is a compile-time error because the compiler knows that the `push_back` operation invalidated the memory to which `P` pointed.

To make this possible, Carbon adds storage information to the type system. Carruth calls the abstract locations where objects may live “places” or “place sets.” A `Buff` allocation is modeled as an owned, disjoint set of places. The set is named in the type's API, so the type can state which storage an operation affects without exposing every implementation detail.

Indexing is described as an implementation of a generic interface called `IndexRefWith`. Instead of returning merely a pointer or reference to an integer, the operation returns a reference to an integer known to live in the buffer's place set. The pointer obtained by the caller therefore carries provenance information: it points into the buffer's storage, not just to an arbitrary integer location. Carbon can infer this place information for local variables, so programmers do not need to write the full place annotation every time they declare a local pointer.

The buffer's `push_back` API declares an `invalidate` effect for its storage place set. When the call occurs, the compiler connects the effect to every pointer known to point into that set and marks those pointers invalid. A later use is rejected. Carruth compares the role of these place parameters with Rust lifetime parameters and mentions similar goals in Sean Baxter's safer C++ proposals, while noting that Carbon's particular place-and-set design is intended to integrate with C++ interoperability.

Thread safety through protected places

Temporal safety is only one part of the model. Carbon also proposes thread-safety checking based partly on Clang's existing thread-safety analysis [30:00]-[35:00]. Clang has supported this analysis for more than a decade using annotations originally associated with conventions for mutex-protected code. Carbon takes the idea beyond a commenting discipline by representing the relevant relationships in the language and checking them.

Each mutex is associated with a set of places that it protects. A variable such as a bank-account balance can be declared as guarded by a particular mutex, which places the variable's storage in that mutex's protected set. APIs can state that a particular mutex must be locked before they are called.

Lock acquisition and release are modeled as safety effects, much like invalidation effects. Carbon tracks which specific locks are held and uses place provenance to determine which object or mutex a lock belongs to. References and pointers shared across threads must also be explicitly marked as shared. Thread-sharing APIs can require such a marker or establish the shared state, and access through a shared pointer is restricted to guarded variables when the appropriate lock is held.

Carruth maps a standard Clang thread-safety example into Carbon. A `balance` field is guarded by a mutex, and an API requires that mutex to be locked. After a scoped lock operation, Carbon records that the mutex associated with the current object is held. If the code then accesses a variable protected by a different mutex, Carbon reports an error. The compiler can distinguish “the mutex in this bank account” from “the mutex required for that other bank account” through place provenance. Carruth says this precision is harder to achieve with C++ annotations, especially when templates and other abstractions obscure the relationship.

Migrating from C++ to strict Carbon

Carruth next explains what Carbon's migration strategy is not trying to do [35:00]. The project does not want to add a large volume of safety annotations directly to existing C++ code. C++ lacks the type-system machinery needed to represent these contracts cleanly, and retrofitting the annotations would be difficult, ugly, and difficult to scale.

Carbon also does not aim to prove that arbitrary existing C++ code is safe. Some existing C++ will be provably safe, which is useful, but the system does not need to understand every technically safe C++ program. The target is a practical migration path rather than a universal safety proof for all legacy code.

The described workflow has several stages. First, code should be movable from C++ to permissive Carbon in a mechanical and deliberately boring way. That makes the initial port easier to validate. Next, the team can add safety annotations and contracts in Carbon, where the language is designed to represent them. Strict checking can then be enabled in pieces. The order does not need to be strictly top-down or bottom-up; Carbon's migration model must allow teams to choose the order that fits their codebase.

C++ interoperability provides an alternative to a full port. Code owned by the team can be migrated into permissive Carbon, while dependencies the team does not own can remain in C++ and be wrapped from Carbon. The same process of adding contracts and enabling strict checking can then be applied at the wrapper boundary and to the Carbon code around the dependency.

The mechanical port example includes a type with collections of venues and teams [40:00]. The C++ version uses `std::vector`, while the Carbon version uses `Buff`. A method such as an elimination round explicitly takes `self` by reference in Carbon because it mutates the object; in C++, mutation of the object is implicit in the ordinary method form. Another method that returns a pointer into a collection is analogous to a const-qualified C++ method and takes a read-only input in Carbon.

A caller named `finals` is ported as well. The basic operations—reference parameters, local variables, object method calls, and collection accesses—remain similar. One notable difference is that Carbon requires the call site to carry a matching annotation when a reference is passed to a function. Carruth describes this as a feature often requested for C++ but designed into Carbon from the beginning.

Adding strict contracts incrementally

Switching the ported code from permissive to strict exposes missing safety information [40:00]-[45:00]. A resize operation may reallocate the internal team storage, so its implementation has an invalidation effect. If the method signature does not state that it invalidates a relevant place set, the method itself cannot type-check: its implementation has a safety effect that is not represented in its public contract.

The fix is to add the invalidation effect to the method's signature. Carbon does not need to expose the concrete storage field as part of the public API. Instead, the type can define an abstract place set—for example, a set representing “teams”—and alias the implementation's internal storage to that set. Callers learn the safety-relevant fact without depending on the representation.

The caller then encounters a second class of error. If it obtains a pointer `L` from one collection and calls an operation that invalidates another collection, the compiler needs to know whether `L` might belong to the invalidated storage. If the pointer's place is unknown, Carbon must conservatively reject its use.

The solution is to model the venue storage as a distinct, owned place set disjoint from the team storage. A pointer returned from the venue method is then known to point into the venue set. An operation that invalidates the team set cannot invalidate that pointer, so the later use of `L` is accepted. The caller must also propagate the invalidation effect in its own signature so that its callers receive the same information.

This example illustrates the intended workflow: add one contract, observe the resulting errors, refine the abstract place sets, and gradually make more code strict. The migration and the safety model can be advanced independently until the relevant parts of the program are fully checked.

From a science experiment to evaluation

Near the end of the main talk, Carruth revisits the questions Carbon was originally created to answer [45:00]. These include whether the language design is feasible, whether fine-grained C++ interoperability is possible, whether meaningful memory safety can be layered on top, whether migration can be incremental, whether the compile-time goal of 10x or more is achievable, whether migration can be automated at scale, and whether an open-source governance and evolution process can sustain the project.

His assessment is that the fundamental technical questions are approaching “yes.” He stresses that the work is not complete, but says the project now sees a clear path to answering those questions positively.

That changes the next objective. Carbon 0.1 is intended to mark a transition from experiment toward evaluation. Carruth lists the requirements for that stage: a working compiler and usable tools, a modern Clang-based C++ toolchain, support for the C++ code that Clang supports, integration with existing build systems, and implementation of most of the interoperability needed for meaningful experiments. Missing features may be acceptable if practical workarounds exist. The project also needs enough memory-safety design to make evaluation credible, although the talk does not claim that the complete safety implementation is already finished.

The team has begun early evaluations inside Google and is speaking with technical leads and customer teams. The initial response is described as a mixture of enthusiasm and skepticism. People want to try Carbon interoperability in their own code and build systems rather than evaluate it only through examples. Carruth says the immediate engineering focus is therefore on implementing more interoperability and making the compiler and toolchain easy to integrate.

After positive evaluation feedback and resolution of blockers, the project wants to attempt limited production use. That step is intended to reveal how Carbon works in real systems rather than merely establish that the language is technically possible.

Questions about interop, compile times, OpenMP, and pilot projects

The Q&A begins around [50:00]. Asked about Google feedback, Carruth says the conversations have only recently started. The response has been generally positive but still skeptical about whether the system will work in practice. The most concrete request so far is the ability to try C++ interoperability in real code with real build systems.

He distinguishes Carbon's inline C++ blocks from GPU host/device compilation. Both involve an embedded or crossed boundary, but Carbon does not share one body of code compiled for two targets. Instead, Carbon and C++ remain separate languages with separate grammars, parsers, and compiler stacks. Carbon builds bridges between those stacks. This is different from systems such as CUDA, where host and device code can share more of the compilation model.

When asked whether Carbon is intended to be a faster Rust, Carruth rejects that framing and says he is a Rust supporter. He does say that Carbon is watching compile times closely and that its current compile times are good, but he does not provide the promised benchmark numbers in this talk. He refers the audience to another talk for detailed measurements. Consequently, the 10x compile-time improvement remains a goal discussed in the presentation, not a result established here.

On OpenMP, Carruth says that if Clang supports OpenMP in C++, Carbon can interoperate with that support inside inline C++ blocks. Native Carbon parallelism is a separate design question. He leans toward a fundamental language feature rather than simply reproducing OpenMP annotations, while noting that this would require further design.

A question compares Carbon's explicit C++ boundary with TypeScript's close layering over JavaScript. Carruth says the bright line is intentional. The project wants to prevent C++ complexity and technical debt from leaking into Carbon and wants Carbon's design to retain control over compile times. He contrasts this with his view that JavaScript is evolving more healthily and that TypeScript's relationship with JavaScript therefore makes deeper integration more practical.

Finally, Carruth describes two kinds of ideal pilot project. His personal choice would be moving parts of a compiler into Carbon, reflecting his background in compilers and LLVM optimization. Strategically, he wants a business-critical system that is difficult to move rapidly to Rust, is materially held back by its legacy C++ architecture, and is especially sensitive to memory-safety bugs. He says the project is already speaking with at least two projects that have those characteristics.

Key Takeaways

  • Carbon is designed as an incremental successor to C++, not as a clean-slate replacement that requires rewriting an entire codebase.
  • Its target audience is large, complex, brownfield C++ systems that need a migration path, better compile times, smaller or faster binaries, and memory safety.
  • Carbon's public API defaults are intentionally concise: exported declarations are public unless marked private, and types are final unless explicitly made usable as base classes.
  • The vending-machine example demonstrates explicit object parameters, mutable reference parameters, read-only input parameters, factory-style construction, struct literals, packages, and a program entry point.
  • C++ interoperability is deliberately bidirectional. Carbon can import C++ headers and call C++ APIs, while C++ can derive from Carbon classes and call Carbon methods.
  • C++ overload selection is delegated to Clang's actual C++ rules, but implicit conversions at the call boundary must satisfy Carbon's rules.
  • Inline C++ blocks are treated as raw text and passed to Clang, creating a clear boundary between two compiler stacks rather than blending the languages into one grammar.
  • Carbon's proposed strict memory-safety model uses place sets and effects to track where references point and which operations can invalidate those locations.
  • A buffer operation such as `push_back` can declare that it invalidates the buffer's storage, allowing the compiler to reject later use of pointers into reallocated memory.
  • Thread safety is modeled through protected place sets, mutex lock effects, shared references, and API requirements, enabling Carbon to distinguish the wrong mutex from the correct one.
  • The intended migration path is C++ to permissive Carbon, followed by gradual safety-contract annotation and incremental conversion to strict Carbon; unowned dependencies can remain C++ behind wrappers.
  • Carbon is nearing the end of its feasibility experiment but still needs a usable compiler, broad interoperability, build-system integration, evaluation feedback, and limited production use before its claims can be tested at scale.

Practical Notes

Migration sequence

The workflow described in the talk is to begin with a mechanically simple port into permissive Carbon. The goal is not to make the first port maximally safe or idiomatic; it is to make it boring enough to validate. Once the code is in Carbon, safety contracts can be added gradually. Strict checking can then be enabled for selected components, with invalidation, storage, sharing, and lock requirements propagated through APIs as errors reveal missing information.

Teams do not need to migrate every dependency. Code they own can be ported into permissive Carbon, while third-party or otherwise unowned C++ can remain behind Carbon wrappers. The transcript explicitly says that both approaches should support the same later process of adding safety contracts and checking the Carbon boundary.

Interoperability architecture

The intended architecture has a bright line between Carbon and C++. Imported C++ headers and the `Cpp` package expose existing C++ APIs. Inline C++ blocks allow Carbon source files to contain C++ helper code that is passed to Clang. C++ can also consume Carbon types and derive from Carbon classes.

This boundary is an intentional tradeoff. It makes interoperation explicit and limits leakage of C++'s grammar, complexity, and compile-time costs into Carbon. It also means that Carbon is not trying to behave like a transparent layer over C++ in the style of TypeScript over JavaScript.

The talk does not provide command-line build commands. The concrete toolchain evidence is a Compiler Explorer demonstration using a latest private Carbon compiler, the libC++ standard library, Clang, and C++ header interoperability. Actual project adoption therefore depends on the future compiler and build-system integration work Carruth identifies as a prerequisite for Carbon 0.1.

Designing safety contracts

For an allocation-owning container, the implementation should expose an abstract place set representing its storage. Operations that return references or pointers should associate those results with the relevant place set. Operations that may free or reallocate storage should declare an invalidation effect on that set.

The public contract should expose safety-relevant abstractions rather than concrete fields. In the example, a public “teams” place set can be an alias for whatever internal storage actually holds the teams. This lets callers understand that a resize may invalidate team pointers without coupling them to the implementation layout.

For shared state, variables should be associated with the mutex that protects them. APIs can require that mutex to be locked. References and pointers crossing thread boundaries need to be marked shared, and access through them should be permitted only when the guarded place and corresponding lock state are known.

Decisions and caveats

The proposed strict mode is the safety guarantee; permissive mode is an intermediate migration mode and should not be treated as equivalent to fully checked Carbon. The transcript describes the safety mechanisms as a design and teaser rather than presenting a complete production implementation.

The project does not intend to prove arbitrary existing C++ safe or retrofit large numbers of safety annotations into C++. A codebase that cannot express its safety contract cleanly may need restructuring at the Carbon boundary, and some legacy C++ will remain outside the proof.

C++ interoperability is working in the demonstrated example, but Carruth says that most of the functionality needed for evaluation is still being implemented. The live demonstration uses a private compiler and includes a known syntax mismatch between the presentation and compiler. It establishes that the example works, not that Carbon has a stable production toolchain.

The 10x-or-more compile-time improvement is a stated goal. The talk does not provide benchmark data proving that target. Similarly, the discussion of strict memory safety does not establish that all listed guarantees are already available in the current implementation.

For a pilot project, the speaker's strategic criteria are unusually specific: the software should be business-critical, difficult to migrate rapidly to Rust, materially constrained by its existing C++ architecture, and highly exposed to the cost of memory-safety bugs. OpenMP can be used through C++ interoperability where Clang supports it, while native Carbon parallelism remains an unresolved design choice.
