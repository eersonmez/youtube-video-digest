The video covers Carbon’s development and readiness for incremental migration from C++.

NDC Conferences · 2026-07-22T08:00:34+00:00 · Carbon: graduating from the experiment - Chandler Carruth - NDC Toronto 2026

What This Video Is About

In this NDC Toronto 2026 talk, Chandler Carruth gives a progress report on Carbon, a proposed successor language intended to help existing C++ systems migrate incrementally rather than requiring a full rewrite. The project began four years earlier as an experiment in whether a new language could improve on C++ while remaining closely integrated with the large amount of C++ code already in use.

The motivation is C++'s accumulated technical debt: more than 40 years of compatibility constraints, slow and inconsistent language evolution, textual inclusion, long compile times, pervasive macros, weak support for type-checked generics, no standard package manager, and no built-in memory-safety model. Carruth presents memory safety as an increasingly serious industry problem and argues that C++ has not made material progress toward solving it.

The talk demonstrates Carbon language features, two-way C++ interoperability, and a proposed memory-safety system based on tracking where objects live and how APIs invalidate or protect those locations. Carbon is designed to support a permissive intermediate mode and a stricter, checked mode so that migration can happen in stages. Carruth says the fundamental technical questions are approaching positive answers, but a usable compiler, complete enough interoperability, build-system integration, evaluation by real users, and limited production deployments still remain.

Detailed Walkthrough

At [0:00], Carruth revisits the announcement of Carbon in Toronto four years earlier. He frames Carbon as a response to a problem rather than as an attempt to replace C++ for its own sake. C++ is important and has an enormous installed base, but its evolution is constrained by backwards compatibility and what he describes as an ineffective standards process. The result is that improvements that would require substantial changes are difficult to make.

The problems he identifies include continued dependence on textual inclusion, unbearable compile times, pervasive macros, the absence of a type-checked generic system, and the absence of a package manager. Memory safety is the most urgent concern in the talk: C++ provides no general language-level guarantee against errors such as use-after-free, and Carruth characterizes that gap as an existential risk for the language.

The original idea was therefore an “incremental successor” to C++. Carbon is meant to provide a migration path for large, complex, existing C++ systems, including brownfield systems that cannot realistically be rewritten. The goals include adding memory safety to existing software, improving compile times by a hoped-for factor of 10 or more, producing smaller and faster binaries, and retaining practical interoperability with C++.

The project itself has grown as an open-source effort. Carruth reports more than 5,000 pull requests, more than 90 contributors, 179 merged formal proposals, and 159 closed “leads” issues, which he describes as questions about difficult areas of language design. The project publishes weekly summaries called “Last Week in Carbon,” has produced 12 editions of the bimonthly “Carbon Copy” newsletter, and uses an active Discord server for discussion. He presents this governance and evolution process as evidence that Carbon can evolve openly and transparently, in contrast to the process he criticized for C++.

At [5:00], the talk turns to a Carbon code example centered on a vending machine. The example defines a `VendingMachine` class and a function that creates one. The function accepts a 32-bit integer price and returns `Self`, where `Self` refers to the containing class. This factory function is intended to fill the role that a constructor commonly fills in C++.

The example also uses struct-like literal initialization. Carruth compares this to C++ list initialization with designators, but describes Carbon's form as more type-based and definitive while retaining the convenience of initializing fields directly.

Carbon makes the object parameter explicit. A `dispense item` operation is a method because it receives the object on which it operates, but unlike C++, where `this` is implicit, the object parameter is explicit in Carbon. In the example, the object is passed by reference so that the method can mutate the vending machine's count. Carbon therefore has references as a deliberate language feature rather than relying only on C++-style pointer conventions.

Another method, `print profit`, takes an input `self` parameter instead of a mutable reference. Carruth compares Carbon input parameters to the kind of const-reference parameter that C++ programmers commonly want as a default. The intent is to make the common, non-mutating case concise.

The example also illustrates Carbon's access-control defaults. In an exported API file, declarations are public by default, so the author does not need to add `public` to each function. If a declaration appears textually in an API file but is not intended to be part of the public API, it can be marked `private`. Carruth argues that this makes the public surface easier to read: readers can skip the implementation details as soon as they encounter the private section. The vending machine's price member is used as an example of a private member.

Carbon also has a default package behavior. If a file does not declare itself as a library in a particular package, Carbon treats it as a program in the `main` package. Such a program needs a `run` function. The example's `run` function constructs a vending machine, calls a method to obtain an item's price, and prints the result. Carruth uses this portion to establish that ordinary language fundamentals—types, variables, methods, calls, and program entry points—are already present before moving to the more distinctive parts of Carbon.

At [10:00], the example begins using C++ interoperability. The code imports the standard library iostream header through Carbon's `CPP` package. This lets Carbon code refer to C++ library entities such as `std::cout`. Carbon uses dots for qualified names, while C++ uses double colons; the interoperation layer translates between those naming conventions.

Carruth explains that early Carbon printing facilities could print integers but did not yet handle strings. Scott Hanselman had criticized that limitation in an earlier talk. Carbon's response was not simply to add an independent Carbon string type, because a useful string type would need to interoperate correctly with C++ string types. Instead, the team wanted C++ interoperability in place before designing Carbon strings, even though that made the immediate solution more difficult.

The example calls C++ stream output using the C++ shift-left operator. Carbon supports the operator, but Carbon does not allow it to be chained in the same way as C++ stream expressions because, as a general Carbon operator, shift-left is a mathematical shift operation rather than a streaming abstraction. The resulting parentheses make the syntax awkward, which motivates a better interoperation technique.

At [10:00]-[15:00], Carbon embeds an inline C++ block directly in the Carbon source file. The block is represented as a multiline raw block string literal. The additional hash delimiter avoids having to escape backslashes inside the embedded C++ text. Carbon does not need to parse or understand the C++ grammar in this block. It treats the block as text, passes it to Clang, and asks Clang to compile it as C++.

The embedded code defines a C++ variadic template helper that streams a sequence of arguments and then emits a newline. The helper uses a C++ fold expression, including the parentheses required by C++ syntax. Once the helper has been defined, Carbon code can reach back through the `CPP` package and call it.

This boundary has a deliberate division of responsibility. When Carbon calls a C++ API, the C++ side performs API selection and overload resolution using actual C++ rules. Carbon does not implement an approximate copy of C++ overload resolution, which could diverge from Clang and produce ambiguous or incorrect calls. However, the arguments being passed are still Carbon arguments, so implicit conversions are governed by Carbon's rules. If C++ overload resolution would normally select an overload using an implicit conversion that Carbon considers unsafe, Carbon can reject the call.

This gives Carbon exact C++ overload behavior without requiring Carbon to adopt every C++ conversion rule. It also makes the language boundary explicit: C++ determines which API is being called, while Carbon retains control over the safety and conversion semantics of the call.

At [15:00]-[20:00], the example reverses the direction and accesses Carbon code from C++. First, the Carbon vending machine is changed into a base class. Carbon's default for a type is to make it final. Inheritance therefore requires an explicit opt-out from that default. Carbon supports inheritance with the goal of matching C++ behavior closely enough for interoperation.

The inline C++ block then defines a C++ `Snack` struct containing a price and a C++ `SnackMachine` class derived from the Carbon vending machine class. The C++ constructor initializes the Carbon base subobject by calling the Carbon factory function through a local alias for the base type. The name translation between Carbon dots and C++ double colons continues to work in this direction.

The derived C++ class adds a method for vending a snack. It calls the inherited Carbon method that dispenses an item, obtains the snack's price, prints it, and then calls the inherited Carbon `print profit` method. The example is intended to show more than a C-compatible function boundary: a C++ class can derive from a Carbon class, initialize it using Carbon functionality, call inherited Carbon methods, and use the resulting values.

At [20:00], Carruth describes running the example in Compiler Explorer. The transcript says that the code compiles and runs with the latest private compiler. The compiler version shown there has one purely syntactic difference from the slide example: it still expects the method syntax in square brackets rather than parentheses. Carruth presents this as an implementation-status detail, not a semantic difference.

He says C++ interoperability is in a strong and rapidly improving state, with new interoperability features being implemented frequently. He had to update the examples during the week of the talk because several features that had not worked the previous week had begun working.

At this point, the talk returns to memory safety. Carruth says the industry pressure around memory safety has accelerated, causing Carbon's team to work on its memory-safety design earlier than originally expected. The design has four main requirements.

First, strict Carbon must actually be memory safe, with a goal comparable to Rust's guarantees. Second, it must be smooth enough to represent existing correct C++ patterns rather than forcing every safe C++ design into an entirely different programming model. Third, migration must be incremental in two separate dimensions: the amount of code moved from C++ to Carbon and the amount of safety information modeled in that code. Fourth, moving toward strict checking must not introduce additional undefined behavior, even if unsafe code elsewhere misbehaves.

At [20:00]-[25:00], Carruth describes two modes. Strict Carbon is intended to provide compile-time temporal memory safety and runtime spatial memory safety, along with checks involving type initialization, null pointers, and data races. Permissive Carbon is an intermediate mode that allows code to be migrated and safety contracts to be introduced before the entire codebase is subjected to strict checking.

Temporal safety is the difficult part of memory safety because it concerns whether an object is still alive when a pointer or reference is used. Carruth decomposes a C++ use-after-free into four ingredients: an allocation must exist, a pointer into that allocation must be captured, an operation must free or reallocate the storage, and the dangling pointer must then be used.

At [25:00], he applies the same sequence to a Carbon container called `Buff`, which serves a role similar to `std::vector`. The container is initialized and allocates memory. Carbon code captures a pointer into the buffer. A `push_back` operation may then reallocate the buffer's storage. When the code tries to use the old pointer, Carbon is intended to know that the pointer has been invalidated and issue a compile-time error.

The mechanism is based on “places” or “place sets.” A place represents where an object can live in memory, while a place set groups possible locations. `Buff` declares a named owned place set for the memory it allocates. Because that allocation is owned by the buffer, it can be treated as disjoint from other owned allocations.

The buffer's indexing operation is described as an implementation of a generic interface called `IndexRefWith`. Its result is not merely a reference to an integer. It is a reference to an integer known to live in the buffer's place set. Consequently, a pointer obtained from indexing carries more information than an unqualified pointer-to-`i32`: it carries the fact that the pointed-to object is located in the buffer's named storage.

Local variables do not need to spell out this place information manually. Carruth says Carbon can infer the place annotations for local declarations. The important information appears in the type system and API effects, even when the local syntax remains concise.

The `push_back` API declares an `invalidate` effect. The effect identifies the buffer's place set as storage that may be invalidated by the operation. When the compiler sees `push_back` called on the buffer, it connects that effect to the pointer previously obtained from the buffer. Since the pointer points into the invalidated place set, the later use is rejected.

This makes invalidation part of the API contract rather than leaving it only in documentation. Carruth compares the idea to Rust lifetime parameters and also mentions similar directions in Sean Baxter's safer C++ proposals. Carbon's “places” terminology and related details are chosen to integrate with C++ while addressing the same underlying problem: APIs need to express where values live and what operations can do to that storage.

At [30:00]-[35:00], the talk applies the same general approach to thread safety. Carruth uses Clang's thread-safety analysis, referred to as `-Wthread-safety`, as the baseline. That analysis has existed in open-source Clang for more than a decade and originated from a commenting discipline used at Google for documenting mutex-protected code. Carbon's design turns similar information into checked annotations and type-system effects.

A mutex protects a set of places. A variable can be declared as guarded by a particular mutex, which associates the variable's storage with that mutex's protected set. Lock acquisition and release are modeled as safety effects, similar to invalidation effects. An API can require that a particular mutex be locked before the call is allowed.

Carbon also requires references and pointers shared across threads to be marked as shared. Thread-sharing APIs can require pointers to be shared or make them shared as an effect. Access through a shared pointer is restricted to cases where the accessed variable is guarded by a mutex and that mutex is currently held.

Carruth maps these ideas onto the example used in Clang's thread-safety documentation. A C++ `guarded_by` annotation becomes a Carbon guarded declaration. A C++ lock requirement becomes a Carbon API requirement that a mutex be locked. Shared references must be marked explicitly. Once a scoped lock or equivalent operation has acquired the lock, the safety state records which mutex is held.

The place-based model provides additional precision. If code locks a mutex belonging to one bank-account object but then accesses a value guarded by a different bank-account mutex, Carbon can identify that the wrong lock is held because the lock effect is tied to a particular place and provenance chain. Carruth says this is an area where traditional Clang annotations can have difficulty, especially when templates and other C++ abstractions obscure the relationship.

At [35:00], the talk addresses how existing C++ code can move toward strict Carbon. Carruth identifies two non-goals. Carbon does not intend to add a large volume of safety annotations directly to existing C++ code, because C++ does not have a type system capable of expressing all of the required information and the resulting annotations would not scale well. Carbon also does not intend to prove that arbitrary existing C++ code is safe. Some C++ code may be technically safe but still too difficult for Carbon to understand, and that is acceptable under the stated plan.

The intended workflow is instead to move C++ code mechanically into permissive Carbon. This first step should be as boring and predictable as possible, because a mechanically obvious migration is easier to validate. Once code is in Carbon, safety contracts can be introduced gradually using Carbon's language features. Portions of the code can then be switched from permissive to strict mode and checked incrementally.

The order of migration must remain flexible. Carbon cannot require that all callers be migrated before callees, or that code be migrated strictly from the bottom up or top down. Any such fixed layering requirement could block a large codebase. C++ interoperability and permissive mode together are intended to provide the flexibility needed to migrate in different orders.

Dependencies that a team does not own do not necessarily need to be ported. A C++ library can instead be wrapped in Carbon, after which the same process of defining contracts and increasing checking can be applied at the wrapper boundary.

At [40:00], Carruth gives a mechanical-port example involving a tournament-like API. The C++ version uses `std::vector`, while the Carbon version uses `Buff`; the example includes collections of teams or venues and functions such as an elimination round, a venue lookup, and a `finals` caller.

The Carbon version makes mutation explicit on the method that can resize its internal collection. A non-mutating method corresponds to a const-qualified C++ method and uses an input object parameter. A method can return a pointer to one of the stored locations. The caller has similar local variables and method calls in both languages. One notable Carbon requirement is that a call site must carry an annotation when passing a reference to a function, matching the function's reference parameter. Carruth describes this as a feature that has often been requested for C++.

Switching the ported code from permissive to strict mode exposes the safety information that was previously missing. The resize operation may reallocate the internal collection, so it has an `invalidate` effect. If the method's signature does not declare that effect, the implementation cannot type-check correctly, and callers do not receive the information they need to reason about pointers returned from the method.

The fix is to add the invalidation effect while hiding the storage representation. The type defines an abstract place set, referred to in the example as `hat teams`, and aliases that abstract set to the internal storage place. The method can then expose the fact that it invalidates the teams without exposing precisely how the collection is laid out.

The caller initially produces another error. After calling the elimination-round method, it uses a pointer `L` that could have come from the collection that was invalidated. Carbon conservatively rejects the use because it has no proof that `L` remains valid.

The example is corrected by defining a separate place set for venues, referred to as `hat venues`, and making it disjoint from the teams place set. The lookup method returns a pointer whose type includes the fact that it points into the venues place set. Since the elimination round invalidates the teams set rather than the venues set, the pointer remains valid. The caller also propagates the teams invalidation effect so that its own callers receive the same information.

This example illustrates the two-dimensional migration strategy. Safety annotations can be added gradually, and the code can be made stricter gradually. Once both the APIs and their callers carry the needed effects and place information, Carbon can type-check the full chain.

At [45:00], Carruth explains why he considers Carbon to be approaching graduation from the experimental phase. The original project had several fundamental questions to answer: whether the language could be designed as proposed, whether fine-grained two-way C++ interoperability was possible, whether memory safety could be layered on top, whether that safety model could be adopted incrementally from C++, whether the compile-time goal of 10x improvement was achievable, whether migration could be automated at scale, and whether an open-source governance and evolution model could work.

He says the project is nearing “yes” on the fundamental technical questions, although the work is not finished. The next stage is evaluation rather than pure scientific experimentation. Carbon 0.1 is intended to mark that transition.

For evaluation to be meaningful, Carbon needs a working compiler and usable tools. The compiler must work with a modern Clang C++ toolchain, support the C++ code that Clang supports, integrate with existing build systems, and implement enough of C++ interoperability for users to try Carbon in real code. Missing features may be acceptable only if practical workarounds exist. Carruth also says some memory-safety design is necessary for evaluation because memory safety has become too important to defer entirely.

The team has begun early evaluations inside Google and has started talking with technical leads and customer teams. The response so far is described as cautiously positive: people are interested, but some are skeptical that the system will work in practice. The most concrete request is to try the interoperability in real code with real build systems. The planned progression is to remove the blockers discovered during evaluation, obtain sufficiently positive feedback, and then target limited production usage.

The question-and-answer period adds several constraints and motivations. Asked about Google adoption, Carruth says those conversations have only recently begun. The team is broadening the discussions and following up with groups that want to test the interop model directly.

Asked whether inline C++ is like GPU host/device programming, he distinguishes the designs. GPU systems may compile shared code for host and device, often through one compiler stack. Carbon's inline C++ is an explicit boundary between two languages, each with its own grammar, parser, and compiler. The implementation builds bridges between the two compiler stacks rather than treating the two languages as one shared source language.

On compile times, Carruth does not call Carbon a faster version of Rust and says he is a Rust fan. He does say that compile time is a major concern and that Carbon's current compile times are good. He does not give benchmark numbers in this talk, pointing instead to a separate talk where he planned to show the measurements. The transcript therefore supports the 10x figure as a project goal, not as an achieved result demonstrated here.

On OpenMP, Carbon can interoperate with OpenMP-supported C++ inside inline C++ blocks to the extent that Clang supports it. Native Carbon parallelism is a separate design question. Carruth leans toward providing more fundamental language support rather than simply adding OpenMP-style annotations, and he also mentions CUDA as an example of a system layered on top of C++.

Another question concerns why Carbon maintains such an explicit boundary instead of integrating with C++ as deeply as TypeScript integrates with JavaScript. Carruth gives two reasons. First, the Carbon team wants to prevent C++'s technical debt and complexity from leaking throughout Carbon. Second, the boundary protects Carbon's ability to pursue its own compile-time goals rather than requiring it to support every behavior and interaction that C++ permits. He sees the TypeScript/JavaScript relationship as different partly because JavaScript is evolving in a healthier way and the two communities have a more compatible relationship.

Finally, Carruth describes his ideal pilot projects. His personal dream project would be moving parts of a compiler into Carbon, partly because he is a compiler and optimizer developer and would like to work on LLVM-related systems in a better language. Strategically, he says the best pilot would be business-critical software that is strongly affected by legacy C++ architecture, has no realistic path to rapidly adopting Rust, and is particularly sensitive to memory-safety bugs. He says the team is already talking with at least two projects that appear to have all three characteristics.

Key Takeaways

  • Carbon is intended as an incremental successor to C++, not a greenfield language that ignores existing C++ systems.
  • The project is motivated by C++ technical debt, slow language evolution, textual inclusion, long compile times, macros, weak generic support, lack of package management, and lack of memory safety.
  • Carbon's migration goals include memory safety, potentially 10x or better compile-time improvements, smaller and faster binaries, and support for large brownfield codebases.
  • Carbon demonstrates explicit object parameters, references, factory functions returning `Self`, struct-like initialization, public-by-default exported APIs, private implementation details, and explicit inheritance opt-in.
  • Two-way C++ interoperability is a central design feature. Carbon can call C++ APIs, and C++ can derive from and call Carbon classes and methods.
  • C++ overload resolution remains C++'s responsibility, but argument conversions are governed by Carbon rules. This preserves exact C++ API selection while allowing Carbon to reject unsafe implicit conversions.
  • Inline C++ embeds raw C++ text in a Carbon file and sends it to Clang, creating a bridge between two compiler stacks rather than merging the two languages.
  • Carbon's proposed temporal-safety model tracks named memory places and attaches invalidation effects to APIs such as buffer resizing, allowing use-after-free errors to become compile-time errors.
  • Thread safety uses place-specific mutex effects, guarded variables, lock requirements, and explicit shared references to identify whether the correct lock protects a given access.
  • Migration is designed to proceed from C++ to permissive Carbon, then through incremental safety-contract annotation and finally into strict Carbon. Unowned C++ dependencies can be wrapped instead of ported.
  • Carbon does not intend to annotate all existing C++ or prove arbitrary C++ code safe; the safety model is meant to be applied where Carbon can express and check the required contracts.
  • Carbon is nearing the end of its experimental phase, but real evaluation still depends on compiler maturity, tooling, C++ interop coverage, build-system support, user feedback, and eventual limited production use.

Practical Notes

A practical evaluation path described by the talk begins at the interoperability boundary. Carbon code can import C++ headers through the `CPP` package and use C++ library facilities such as the iostream library. If Carbon syntax is not yet sufficient for a needed operation, an inline C++ block can define a helper in the same source file. The embedded text is passed to Clang, so the C++ block must be valid C++ for the compiler and standard library being used.

The boundary should be treated as explicit rather than transparent. C++ overload resolution chooses the API, but Carbon conversion rules still apply to the arguments. This is useful for preserving C++ APIs while tightening safety, but it also means a call that compiles in C++ may be rejected from Carbon because Carbon does not permit the required implicit conversion.

For code migration, the suggested sequence is to make the first port mechanically simple and permissive. Existing C++ can be translated into permissive Carbon, or a Carbon wrapper can be placed around a dependency that the team does not own. Safety contracts can then be added in Carbon without requiring the original C++ code to carry a large set of annotations. The code can be switched to strict checking in independently chosen increments rather than following a mandatory top-down or bottom-up order.

APIs involving storage should expose invalidation behavior. A collection such as `Buff` needs a named place set for its owned allocation, and operations such as resizing or `push_back` need an `invalidate` effect describing which places may become invalid. Public APIs should expose the abstract storage relationship that callers need while hiding implementation details, as in the example that aliases an abstract teams place set to the collection's internal storage.

For multithreaded code, variables should be associated with the mutex that guards them, lock and unlock operations should update the safety state, and APIs should state which locks they require. References and pointers that cross thread boundaries need to be marked as shared. The transcript uses Clang's thread-safety analysis as a conceptual baseline, but it does not provide complete Carbon syntax for these declarations.

The language defaults are deliberate API-design choices. Exported Carbon APIs are public unless marked otherwise, while types are final by default and require an explicit change when they are intended to serve as base classes. These defaults reduce unintentional API exposure and inheritance, but C++-style class hierarchies must be reviewed for the required opt-ins during migration.

The talk does not provide a public installation command or a complete build recipe. The live example is described as running in Compiler Explorer with the latest private compiler, and the speaker notes that the implementation still had a syntactic difference in method notation. A real project will also need a modern Clang-based toolchain, sufficient C++ interop, and integration with its existing build system; Carruth identifies all three as prerequisites for meaningful evaluation.

The 10x compile-time improvement remains a stated goal rather than a result established in this talk. Carruth says current compile times are good and refers to a separate talk for benchmarks, but no measurements are given here. Similarly, the memory-safety design is explained in detail, but the talk distinguishes the design and implementation work from the future evaluation and production-readiness milestones.

OpenMP support is available through inline C++ only where the relevant Clang C++ support exists. Native Carbon parallelism remains an open design direction, with Carruth leaning toward deeper language-level support rather than simply copying OpenMP annotations. The transcript does not establish that a native Carbon parallelism model has been finalized.
