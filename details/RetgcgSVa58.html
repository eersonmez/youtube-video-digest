This presentation demonstrates how to integrate an AI agent into applications using the GitHub Copilot SDK.

NDC Conferences · 2026-07-10T11:09:31+00:00 · Put an Agent Inside Your App in 10 Minutes or Less with the GitHub Copilot SDK - Daniel Ward

What This Video Is About

This video introduces the concept of building AI agents directly into applications using the GitHub Copilot SDK. The central theme is demystifying what an "agent" is—it is fundamentally a Large Language Model (LLM) augmented with external tools and looping capabilities, moving beyond simple text-in, text-out interactions. By providing LLMs access to functions like web browsing or code execution, they gain agency and decision-making power.

The presentation details how the Copilot SDK allows developers to embed this agent functionality quickly into their applications. It moves from foundational concepts—like what the SDK is and how it works—to advanced use cases such as automatically suggesting fixes for production errors or generating daily summaries by aggregating data from services like Jira and Teams. The speaker emphasizes that while prompt chaining can be easy to prototype, using the SDK provides a more reliable path for repeatable tasks because it integrates deterministic code to handle non-creative steps, leading to higher success rates.

Detailed Walkthrough

The presentation is structured into two main parts: Foundation and Application.

Foundation: The Copilot SDK is described as an open-source SDK from GitHub designed to call GitHub Copilot or other AI services (supporting bring your own key for Anthropic, OpenAI, or local models). It officially supports TypeScript, Python, Go, C, Rust, and Java, with unofficial support in Closure and C++. A basic working example involves creating a `CopilotClient`, calling `startAsync()` to spawn the Copilot process, and then creating a `Session` (which is equivalent to a conversation) [4:00]. Prompts are sent via the session rather than directly on the client.

Defining an Agent: An agent is defined as an LLM enhanced with tools and looping capabilities [5:00]. Since an LLM inherently only takes text in and outputs text, tools allow it to interact with the outside world. A simple example involves giving the LLM access to a `web fetch JSON` tool. The LLM uses its English prompt description of the tool to decide when to use it; if the user asks about a website, the agent delegates the task to the underlying code (the harness), which fetches the content and feeds it back to the LLM for synthesis [7:00].

Agent Capabilities and Use Cases: The core value proposition is giving the LLM agency. Examples of potential applications include:

  • Production Error Fixing: When a production error occurs, an agent could collect logs, clone the source code, suggest a fix, and open a pull request [10:00].
  • Ticket Routing: A support team agent could read incoming tickets, judge severity and team assignment, and route them appropriately [10:00].
  • Daily Briefing (Current Project): The speaker's current project aggregates information from Jira, Teams messages, GitHub repos, and pull requests to provide a daily summary of what the user needs to care about [10:00].

SDK Implementation Demo (The 10-Minute Bit): The initial demo focuses on getting the SDK running. The process involves starting the `CopilotClient` asynchronously and then creating a session with specific configurations, such as setting the model (e.g., Claude Sonnet 4.6) [15:00]. When sending a message, one can choose between `sendAsync()` (returns when the message is queued) or `sendAndWaitAsync()` (waits for the full response) [17:30].

Streaming and Custom Tools: The SDK supports reacting to granular events on the session using event handlers. For user-facing applications, streaming responses are crucial; this allows the LLM's output to be delivered in chunks as it is generated, rather than waiting for the entire response [19:30].

Custom tools can be created by decorating standard functions (e.g., a C# function) with specific attributes. The SDK uses an `AI Function Factory` to handle the conversion of this decorated code into a format Copilot understands as a tool, including its description and parameters [24:15]. When prompted correctly, Copilot can autonomously decide to call this custom tool (e.g., asking for weather in San Antonio triggers the fake weather tool) [30:00].

SDK vs. Prompt Chaining: The speaker contrasts using sequential prompts versus using the SDK with deterministic code. While prompt chaining is easy to prototype, its success rate compounds negatively; if each step has a 10% failure chance, the overall success rate drops significantly (e.g., to 59%) [30:00]. In contrast, when using the SDK, non-creative steps are handled by deterministic code (100% reliability), meaning the overall success rate is limited only by the AI component's performance [30:00].

Advanced Concepts and Production Considerations: The speaker showcases more complex applications, such as a "Showdan" bot that processes images and queries GitHub repos/notes via Telegram integration [35:00]. Another advanced feature demonstrated is session forking, where child sessions are cloned from a parent session, allowing users to explore different strategies while maintaining shared history (e.g., in vacation planning) [48:00].

For production deployment, several considerations must be addressed:

  • Sandboxing: Running the agent outside of "YOLO mode" by using containers like Docker or separate VMs to limit its access [52:30].
  • User Experience: Implementing streaming responses for perceived speed and logging all session events via OpenTelemetry, which is supported out of the box [54:00].
  • Authentication & Architecture: Deciding between using a logged-in CLI user, bringing your own API key (e.g., Anthropic), or using GitHub OAuth/environment variables. The SDK can be configured to use a bundled process or an external one on a remote server [51:00].

Key Takeaways

  • An AI agent is fundamentally an LLM augmented with tools and looping capabilities, not just the LLM itself.
  • The GitHub Copilot SDK allows developers to embed this agent functionality into applications quickly by interfacing with the Copilot CLI.
  • Agents gain agency by being given access to external functions (tools), which are defined via descriptions and parameters that the LLM uses for decision-making.
  • Using the SDK is superior to pure prompt chaining for repeatable tasks because it allows deterministic code to handle non-creative steps, ensuring higher reliability.
  • The SDK supports granular event handling on sessions, enabling features like streaming responses crucial for user experience in AI applications.
  • Custom tools can be created by decorating standard functions with specific attributes, which the SDK translates into callable interfaces for the LLM.
  • For production use, sandboxing (e.g., Docker) is necessary to limit an agent's access and prevent misuse of its capabilities.
  • The SDK supports various authentication methods, including using personal API keys directly without relying on a logged-in GitHub user session.

Practical Notes

Implementation Steps:

  • Create a `CopilotClient`.
  • Call `startAsync()` to initialize the Copilot process/CLI child process.
  • Create a `Session` object, configuring it with desired properties like the model (e.g., Claude Sonnet 4.6) and permission handlers.
  • Send prompts using either `sendAsync()` or `sendAndWaitAsync()`.
  • To implement custom tools, decorate your language-specific functions (e.g., C# methods) with appropriate attributes and register them in the session's `tools` property via the `AI Function Factory.create` method.

Architecture Choices:

  • The SDK communicates with a Copilot CLI process, which can be bundled within the application or run on a remote server accessible via URL.
  • For user-facing apps, consider implementing streaming responses to manage perceived latency.
  • OpenTelemetry is supported out of the box for logging session events and spans.

Risks/Caveats:

  • AI is non-deterministic; code integration provides guardrails around AI outputs by passing them into deterministic execution paths.
  • Governance regarding API access (e.g., allowing Copilot to interact with Jira or make PRs) is a significant hurdle in enterprise deployments.
