What This Video Is About

This video provides a rapid demonstration of how to construct a functional Windows kernel driver and a corresponding user-mode client application within a short timeframe. The central problem addressed is the inability of standard user-mode applications, even those running with administrator privileges, to inspect sensitive data, specifically listing loaded Dynamic Link Libraries (DLL) within protected processes like CSRSS (Client-Server Runtime Subsystem). Kernel drivers operate in system space with elevated privileges, granting them the necessary power to bypass these security restrictions. However, this high level of privilege comes with significant risk; improper implementation can lead to a Blue Screen of Death (BSOD), making careful development crucial.

The goal of the demonstration is to build a driver that grants access to DLS within CSRSS processes. This is achieved by creating a device object exposed via a symbolic link, allowing user-mode applications to interact with the kernel component using standard I/O mechanisms like Device Control. The presentation covers the fundamental concepts of kernel programming, the necessary API calls for inter-process communication between user and kernel mode, and the practical steps required to bring this custom driver online and test its functionality against protected system processes.

Detailed Walkthrough

The process begins by establishing a use case: creating an application named "list DLS" designed to list DLL modules in a specific process ID (PID). Initially, attempts are made using standard user-mode functions like OpenProcess on processes such as CSRSS [2:00]. These attempts fail with error number five, which signifies "Access Denied," indicating insufficient privilege for the required operations.

To overcome this limitation, a kernel driver is necessary to obtain a powerful handle to the target process. The communication between user mode and the driver relies on creating a device object within the kernel that the driver exposes [10:00]. From the user-mode perspective, this involves calling CreateFile, which acts as an abstraction to connect to the driver's exposed device. This connection is established by referencing a symbolic link in the `\\?\` namespace, such as `\\.\ProcessPower` [20:00].

The core interaction mechanism chosen for bidirectional communication is Device Control, as Read File and Write File are unidirectional [20:00]. The user-mode application calls DeviceControl, passing an input buffer containing the PID and expecting a handle to be returned in the output buffer. This requires defining a specific control code, such as `IOCTL_OPEN_PROCESS` [25:00].

On the kernel side, the driver's entry point is the Driver Entry function, which initializes the driver [15:00]. The driver must then create a device object using ICreateDevice and subsequently create a symbolic link pointing to that device object in the `\\?\` namespace using IOCreateSymbolicLink [35:00]. Furthermore, the driver must register support for specific operations by filling out an array of function pointers (dispatch routines), specifically supporting Create, Close, and Device Control [15:00].

The implementation of the Create and Close handlers involves completing the I/O Request Packet (IRP) with a success status to allow handle creation [30:00]. The critical functionality resides in the OnDeviceControl dispatch routine. Within this function, the driver extracts the control code from the IRP's IO Stack Location [45:00]. If the code matches `IOCTL_OPEN_PROCESS`, the kernel-mode equivalent of OpenProcess, ZwOpenProcess, is called using elevated privileges to obtain a powerful handle to the specified PID [50:00]. This handle is then copied back into the user-mode buffer for the client application to use in its subsequent DLS listing operation.

Key Takeaways

  • Kernel drivers operate in system space with full kernel privileges, making them necessary for accessing protected processes like CSRSS that are inaccessible from user mode [2:00].
  • Communication between user mode and a driver is facilitated by creating a device object within the kernel and exposing it via a symbolic link (e.g., `\\.\ProcessPower`) in the `\\?\` namespace [10:00, 20:00].
  • Device Control is the preferred method for bidirectional communication between user mode and a driver, as Read File and Write File are unidirectional [20:00].
  • The kernel API is strictly designed and uniform, contrasting with the often messy nature of the Windows User Mode API; developers should use NTDK.h for kernel declarations [15:00].
  • A driver must implement a Driver Entry function to initialize itself and register supported operations (e.g., Create, Device Control) in its dispatch routine array [15:00].
  • To enable user-mode access, the driver must manually create a device object using ICreateDevice and then create a matching symbolic link using IOCreateSymbolicLink [35:00, 40:00].
  • The OnDeviceControl routine is where the core logic resides; it inspects the control code from the IRP's IO Stack Location to determine the requested operation [45:00].
  • Kernel-mode functions like ZwOpenProcess can be used within the driver to obtain powerful handles to processes, bypassing user-mode security restrictions [50:00].

Practical Notes

  • Development Environment: To begin development, install Windows 10/11 SDK and WDK (Windows Driver Kit). For modern projects, use Visual Studio with appropriate templates [20:00].
  • Kernel API Header: The primary header file for kernel programming is NTDK.h, which serves a function similar to windows.h in user mode [15:00].
  • String Handling: Kernel APIs utilize counted strings (UTF-16), consisting of a pointer to the wide string and an explicit length in bytes, rather than relying solely on null termination [15:00]. Functions like RtlInitUnicodeString are used for initialization [35:00].
  • Driver Structure: A driver requires an unload routine for cleanup, even if it is initially empty. The Driver Entry function must return a success status to indicate proper initialization [15:00].
  • Device Exposure: For user-mode interaction, the device object created by the driver must be exposed via a symbolic link in the `\\?\` namespace (e.g., `\\.\ProcessPower`). The name used for this link *must* match what is referenced in the user-mode application's CreateFile call [40:00].
  • Installation: For simple, software-only drivers, installation can be performed manually via registry manipulation (setting service type and binary path) rather than requiring an INF file [55:00].
